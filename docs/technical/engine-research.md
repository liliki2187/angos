## 二、引擎对比分析

### 2.1 Godot 4

#### 优势

| 方面 | 说明 |
|------|------|
| **完全免费开源** | MIT协议，无版权费用，无收入分成 |
| **轻量高效** | 引擎仅100MB+，启动快，低配置要求 |
| **2D优先设计** | 原生2D引擎，非"2D in 3D"方案 |
| **GDScript** | 类Python语法，学习曲线平缓，适合快速原型 |
| **节点系统** | 直观的场景/节点架构，适合UI密集型游戏 |
| **跨平台** | 一键导出PC/Mac/Linux/Mobile/Web |
| **活跃社区** | 开源社区活跃，文档丰富，教程多 |

#### 劣势

| 方面 | 说明 |
|------|------|
| **生态系统较小** | Asset Store资源较少，很多需要自己实现 |
| **团队协作** | 版本控制友好，但缺乏Unity级别的团队工具 |
| **性能优化** | 对于极高复杂度的UI，可能需要额外优化 |
| **商业支持** | 无官方技术支持，依赖社区 |

#### 适用场景

- ✅ 2D独立游戏开发
- ✅ UI密集型游戏
- ✅ 预算有限的团队
- ✅ 快速原型开发
- ⚠️ 大型商业项目（需评估）

#### 参考案例

- 《Cassette Beasts》- 卡牌收集RPG
- 《Brotato》- 生存类游戏
- 《Discharge》- 卡牌策略游戏

---

### 2.2 Unity

#### 优势

| 方面 | 说明 |
|------|------|
| **行业标准** | 最广泛使用的游戏引擎，人才储备丰富 |
| **Asset Store** | 海量插件和资源，可加速开发 |
| **2D/3D兼顾** | URP渲染管线，2D工具链完善 |
| **团队协作** | Unity Teams、Plastic SCM等企业级工具 |
| **跨平台成熟** | 支持几乎所有平台，导出稳定 |
| **社区资源** | 教程、文档、问答资源最丰富 |
| **商业支持** | 官方技术支持、企业服务 |

#### 劣势

| 方面 | 说明 |
|------|------|
| **授权费用** | 收入超过$200k需付费（Pro版$2200/年/席） |
| **学习曲线** | C# + 引擎概念，学习成本较高 |
| **引擎臃肿** | 安装体积大，启动慢，资源占用高 |
| **UI系统** | UGUI较老，UI Toolkit尚在完善 |
| **2D并非原生** | 2D是基于3D的模拟，有额外开销 |
| **版本迭代** | 版本更新频繁，可能有兼容性问题 |

#### 适用场景

- ✅ 商业游戏开发
- ✅ 需要大量现成资源
- ✅ 团队规模较大
- ✅ 目标多平台发行
- ⚠️ 预算有限的独立团队

#### 参考案例

- 《Inscryption》- 卡牌恐怖游戏（参考苏丹的游戏风格）
- 《Slay the Spire》- 卡牌Roguelike
- 《Reigns》- 卡牌决策游戏

### 2.3 Cocos Creator

#### 优势

| 方面 | 说明 |
|------|------|
| **专注2D** | 原生2D引擎，2D渲染性能优秀 |
| **免费开源** | MIT协议，无授权费用 |
| **Web优先** | H5游戏首选，浏览器兼容性最佳 |
| **中文社区** | 国内开发者多，中文文档丰富 |
| **轻量级** | 引擎轻量，适合中小型项目 |
| **TypeScript** | 现代语言支持，适合前端开发者 |

#### 劣势

| 方面 | 说明 |
|------|------|
| **国际影响力弱** | 海外社区小，国际资源少 |
| **3D能力弱** | 3D功能有限（本项目不涉及） |
| **生态规模** | Asset Store规模小于Unity |
| **PC端支持** | PC原生应用支持较弱 |
| **版本迭代** | 大版本间API变化较大 |

#### 适用场景

- ✅ H5/微信小游戏
- ✅ 移动端优先的2D游戏
- ✅ 国内市场为主
- ⚠️ PC端原生应用（需评估）

#### 参考案例

- 《崩坏3》- 3D动作（证明技术实力）
- 大量微信小游戏
- 各类移动端卡牌游戏

## 四、最终决定

### ✅ 已确认：Godot 4.3

**决策日期：** 2026-03-16（二次确认，从 Unity 回切 Godot）

**选择理由：**

1. **完全免费开源** — MIT 协议，无授权费用，无收入分成，独立游戏零成本
2. **2D 原生设计** — UI 密集型游戏的最佳选择，不是"3D 里做 2D"
3. **学习曲线友好** — GDScript 类 Python 语法，小团队快速上手
4. **轻量高效** — 引擎 100MB+，启动快，迭代速度快
5. **节点系统** — 场景/节点架构天然适合卡牌 UI 和复杂界面布局
6. **开源可控** — 无商业风险，可深度定制

**风险与应对：**
- 生态资源少 → 团队自行实现核心系统，积累技术资产
- 社区资源 → 活跃开源社区，文档丰富，持续增长中
- Steam 发布 → 有成熟的 Steam 导出和集成方案

### 备选方案：Unity（不采用）

### 不推荐：Cocos Creator

## 五、实施建议

### 5.1 Godot开发路线图

| 阶段 | 时间 | 任务 |
|------|------|------|
| 第1周 | 学习期 | 团队Godot基础培训，完成官方教程 |
| 第2-3周 | 原型期 | 核心卡牌系统原型，UI框架搭建 |
| 第4-6周 | 框架期 | 数据管理系统，存档系统，属性系统 |
| 第7-12周 | Alpha | 核心玩法实现，美术资源整合 |
| 第13-24周 | Beta | 内容填充，系统完善，优化调试 |

### 5.2 关键技术点

1. **卡牌系统** - 使用Godot的Control节点树实现
2. **数据管理** - JSON/Resource文件管理卡牌数据
3. **UI系统** - 利用Container节点实现自适应布局
4. **存档系统** - Godot的ConfigFile或自定义序列化
5. **文本系统** - 使用TR文件支持本地化

### 5.3 资源参考

- Godot官方文档：https://docs.godotengine.org
- Godot Asset Library：https://godotengine.org/asset-library
- HeartBeast教程（推荐）
- GDQuest教程（推荐）
