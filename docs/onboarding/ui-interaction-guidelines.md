# UI 与交互执行规范

> **用途**：给后续 AI / 新对话在处理 Angus 的 UI、交互、原型呈现、视觉实验页时使用。
> **定位**：这是执行规则手册，不是错题本，也不是玩法真源。
> **最后更新**：2026-05-21
> **重构说明**：本文按“母原则 -> 审计清单 -> 通用规则 -> 场景规则 -> 验收与更新”重新组织；既有规则不应因重排而省略。

---

## 0. 文档使用方式

### 0.1 与 `ai-collaboration-guidance.md` 的分工

- `ai-collaboration-guidance.md` 是大的协作错题本，记录用户偏好、AI 误判原因、宏观工作方式问题。
- 本文是 UI / 交互执行规范，记录做界面时必须遵守的具体规则、槽位、检查项。
- 看似 UI 的反馈，如果本质是范围失控、需求理解偏差、验证方式错误，应继续写入 `ai-collaboration-guidance.md`。
- 可执行的 UI 规则应沉淀到本文。

### 0.2 触发阅读

凡是任务涉及以下任一内容，必须先读本文：

- HTML / Godot 原型界面；
- 任务卡、地图点、派遣配置、角色详情、骰子表现；
- UI 信息层级、交互动效、视觉符号、截图验收；
- 实验页、生图对照、视觉组件词典。

### 0.3 使用顺序

处理 UI 任务时，先按以下顺序使用本文：

1. 先读 §1，确认这次界面是否符合用户的 UI 设计母原则。
2. 再用 §2 做改动前审计，明确页面职责、主操作、状态语义和不做范围。
3. 然后查 §3-§4 的通用规则，确认信息层级、控件语义、状态距离和位置稳定。
4. 最后查对应场景规则：任务卡看 §5，地图 / 区域看 §6，派遣看 §7，骰子判定看 §9，实验页看 §10。
5. 交付前必须按 §11 做截图、入口和版本验收；若用户纠偏，按 §0.4 检查是否需要沉淀。

### 0.4 更新规则

后续用户给出 UI / 交互反馈时：

- 如果是“为什么 AI 会这样误判”，写入 `ai-collaboration-guidance.md`。
- 如果是“以后界面具体怎么做”，写入本文。
- 如果两者都有，两边都写，但不要复制同一段：协作指南写偏差原因，本文写执行规则。
- 如果用户要求修改、纠偏、返工，或指出某个 UI 判断不合理，不能只改实现和截图；最终回复前必须检查是否需要同步本文、`ai-collaboration-guidance.md`、`docs/设计采纳记录.md`、相关 GDD 或 `docs/会话交接_当前状态.md`。
- 如果判断本次 UI 反馈不需要新增规则，也必须在最终回复里简短说明“不写入规则文档”的原因，避免把沉淀遗漏伪装成默认处理。

## 1. UI 设计母原则

这些原则是后续做界面前的最高优先级判断。具体组件规则若看起来互相冲突，先回到本节判断“玩家此刻到底在做什么决定”。

### 1.1 界面先服务玩家当前决策

- 每个页面先明确唯一主要职责：选地区、选任务、组骰池、查看结果、推进日程，不能混成同一个层级。
- 信息、控件和容器必须围绕当前决策组织，而不是把系统里存在的信息都摆出来。
- 玩家界面只显示会改变当前决策的信息；系统知识和默认规则不主动占用界面。

### 1.2 主操作必须是视线终点

- 当前界面的核心功能按钮必须足够大，并放在玩家视线流程的终点。
- 桌面布局里，单步确认 / 进入下一层 / 开始执行这类最关键动作，默认放在右下角；不要藏在面板右上角、标题旁或小尺寸附属按钮里。
- 核心功能按钮的图标 / 箭头必须明显大于普通标签和辅助图标，能一眼指向“下一步”。
- 次级动作如 `下一天`、`返回`、筛选、预览，只能在视觉重量上低于主按钮。

### 1.3 信息按作用域分层

- 全局资源、任务常规属性、特殊约束、操作可行性和解释性文字属于不同层级，不能偷懒做成并列大模块。
- 探索周剩余天数是全局资源，不应伪装成当前任务属性。
- 耗时是行动成本，不是倒计时；红色截稿是特殊约束，不是常规属性。

### 1.4 常规信息和特殊机制分槽

- 耗时、需求、对手骰是常规 meta。
- 红色截稿、深度调查、黑骰、已派遣、日程推进是特殊机制，必须有专属位置。
- 特殊机制如果是当前卡片或页面的视觉主语，就不能降级成普通 chip 或说明句。

### 1.5 可操作性必须一眼可见

- 可点击、不可点击、只读、执行中、已完成不能只靠小字或轻微色差区分。
- 控件长什么样，就应该表达它能不能被操作；可用对象要有主动邀请感，不可用对象要有明确阻断感。
- 只读预览不能复用可操作任务卡语法。

### 1.6 玩家空间记忆不能被破坏

- 玩家会用空间位置记住角色、任务、地区、卡牌。
- 点击、选中、取消、hover 或展开详情，不应让原对象在同一列表里自动换位置。
- 选择状态靠高亮、槽位、详情和队列表达，不靠重排候选池表达。

### 1.7 UI 不能粉饰玩法问题

- 风险、成功率、推荐和达标率必须和真实结算一致。
- 如果真实达标率暴露出适配队伍仍几乎不可能成功，不能只改风险标签或文案；要检查任务需求、目标公式、骰面密度和支援资源。
- 普通任务不能以满编为默认难度；高风险任务也应保持“可搏”，而不是不可玩。

### 1.8 新增强表达时必须清理旧表达

- 加了高权重倒计时块，就不要再保留重复红标签或重复警戒条。
- 整卡已经可点击，就不要再塞重复的小动作按钮。
- 特殊状态进入专属槽后，旧 chip、旧说明、旧按钮要同步下线或降级。
- 删除信息后必须回收空间，不只是让文字消失。

## 2. UI 改动前审计

### 2.1 工作合约

每次动手前，先在心里或回复中明确三件事：

- **保留**：用户已经认可的结构、位置、交互、视觉重点是什么；
- **修改**：这次只解决哪个具体问题；
- **不做**：哪些相邻问题暂不扩展，避免把局部修正变成整体重构。

默认原则：

- 用户指出“某个元素不明确”时，第一轮只改该元素的文本、图标、提示或状态表达。
- 不要因为一个标签不清楚，就重排整张卡片。
- 不要把用户认可的亮点丢掉，例如右侧大倒计时块、规整深度链叠层、熟悉的 HTML 原型壳层。

### 2.2 改动前 10 问

任何 UI 改动前，先快速过一遍：

1. 这个页面的唯一主要任务是什么？
2. 玩家完成判断后，下一步应该点哪里？
3. 这个主按钮是否足够大、足够靠近流程终点？
4. 页面里哪些东西能点，哪些只是看？
5. 有没有只读信息长得像按钮？
6. 有没有系统动作混进任务 / 选择容器？
7. 有没有特殊机制被塞进普通标签流？
8. 有没有关键数值被低权重文案淹没？
9. 点击、选中、取消后对象位置是否稳定？
10. 新增 UI 后，有没有旧按钮、旧标签、旧说明变成冗余？

## 3. 信息层级与视觉主语

### 3.1 玩家信息过滤规则

玩家界面只显示会改变当前决策的信息。

应显示：

- 耗时、成本、人数限制；
- 需求属性 / 需求骰面；
- 对手骰、黑骰、危险后果；
- 倒计时、截止、即将消失；
- 当前可用操作；
- 当前任务链阶段和下一步可见信息；
- 角色与当前任务的适配结论、相关骰面、明显缺口。

不应显示：

- 内部任务类型说明，例如“白色调查”“红色截稿”这类开发分类；
- 默认规则说明，例如普通任务“可反复追踪”“失败保留”；
- 已由另一个 UI 元素表达过的重复说明；
- 含义不明确的标签，例如没有明确玩法后果的“高压”；
- 为了解释系统而写的说明句，除非玩家必须凭它做选择。

### 3.2 一语义一位置

同一语义只保留一处表达。

处理任何 UI 前先做“信息归属”判断：当前决策事实是什么、哪个位置是主承载位、其它位置是否真的改变职责。合法的二次出现只能承担摘要、引用、对比、解释、阻断或反馈之一；如果只是用相近文案、相近权重、相近粒度再说一遍，就必须删除或降级。不要为了保留重复信息再新增解释。

示例：

- 右侧倒计时已经写“剩余 2天”，就不要再写“过期消失”。
- 线索卡已有“调查 -> 新任务”示意，就不要再写“可能发现新任务”。
- 危险已经有“危险”标签，就不要再放孤立感叹号。

### 3.3 信息层级不能用并列模块偷懒

不要把不同性质、不同作用域、不同触发条件的信息做成同一排同等重量的卡片或按钮。并列布局会暗示这些信息同级、同类、同等重要；如果事实不是这样，就会误导玩家。

设计任何信息组前先判断每项信息属于哪一层：

- **全局资源**：例如探索周剩余天数、资金、宏观指标。应放在全局状态区，不伪装成当前任务属性。
- **任务常规属性**：例如耗时、基础需求、主要奖励 / 风险摘要。用轻量 meta 或任务头部表达，不为低频字段单独开大模块。
- **特殊约束**：例如红色截稿截止、突发任务关闭条件、黑骰介入、危险后果。只在条件成立时用警戒条、状态徽章或局部提示表达，不能和普通属性并列成常规字段。
- **操作可行性**：例如能否开始、还缺什么、谁被占用。应靠近主操作和需求覆盖面板。
- **解释性文字**：只在玩家必须凭它做选择时出现；能由结构、标签、按钮或反馈表达的，不另写说明。

典型反例：派遣页把 `耗时 2天`、`剩余 7天`、`截止 3天`、`需求 探索5 / 生存3` 做成四个并列大框。这里“剩余”是全局资源，“截止”是红色任务特殊约束，“耗时”只是执行成本，“需求”才是本页配骰决策核心。正确做法是把全局剩余移回全局状态，耗时压成轻量 meta 或按钮文案，截止做成红色截稿警戒提示，需求与需求覆盖面板形成主线。

删掉重复信息后，必须重新指定“视觉主语”。不要只把重复标签合并成一句低权重说明；若该信息是玩家当前最该看的数值，例如截稿剩余天数，就要给它独立且明确的视觉承载。信息去重的目标是让关键值更突出，不是让关键值变成说明句的一部分。

同形控件不等于同语义。把多个数值做成同一行、同一组卡片或同一种进度条前，必须先能说清它们共同回答的是同一个玩家问题，且拥有同一对象、同一口径和同一判断结果。若一个数值是总结果 / 达标目标，另一个数值是组成来源 / 属性拆解，两者应做成父子结构，而不是同排同权重。典型反例：在角色骰判定中把 `有效点` 和 `理性 / 洞察` 做成三个同级进度条；`有效点` 回答“本次判定是否过线”，属性条回答“哪些属性贡献了有效点”，它们不是同一类信息。

### 3.4 常规标识和特殊机制必须分槽

常规信息放在常规槽位，特殊机制放在特殊槽位。不要把“耗时 / 需求 / 对手骰”这类每张任务卡都可能出现的普通 meta，和“红色截稿倒计时 / 深度调查当前环 / 黑骰终局 / 已派遣占用”这类会改变玩家理解的例外状态混在同一排标签里。

硬规则：

- 常规槽位只承载常规字段：标题、耗时、需求、对手骰、危险等压力信息。
- 特殊槽位承载例外机制：右侧倒计时块、右侧当前环标识、标题旁状态标记、主操作附近的可执行性提示。
- 特殊机制如果是当前卡片的视觉主语，就必须有高权重专属承载；不要把它降级成普通 chip，也不要和多个普通标签并列。
- 同一种特殊机制在同一类界面中必须固定位置。例如红色截稿用右侧倒计时，深度调查用右侧“深度调查 / 当前任务 N”，已派遣靠近任务名或头像组。
- 选择特殊槽位前先问：这个信息是默认规则，还是少数任务才有的例外？默认规则不显示；例外机制用专属语法显示。

### 3.5 视觉重量

- 高频普通信息用轻量标签。
- 强时效、强危险、当前操作状态可以使用更高权重组件。
- 不要把普通信息做成像按钮的大块。
- 不要让辅助信息抢过标题、倒计时和操作按钮。

## 4. 控件语义与状态距离

### 4.1 标识必须绑定本界面的真实对象

图标、标签、徽章、图例、颜色摘要、数字角标都属于“标识”。标识不是装饰，必须回答：它在这个界面里标记了哪个真实可见对象，表达了哪个当前状态，玩家能否凭它做本界面的决策。

新增任何标识前必须检查：

- **对象是否在本界面可见**：标识必须贴近或解释当前界面里的对象，例如地图点、任务卡、角色、按钮、区域详情；不能借用另一个面板或下一层页面里的对象。
- **状态是否在本界面成立**：当前选中、锁定、已派遣、剩余时间、危险、黑骰、深度链等状态，只能出现在真正承载该状态的界面层。
- **编码是否有对应视觉**：图例只能解释同一层里真实出现的颜色、图形、点形、线型或角标；没有对应编码时不放图例。
- **是否影响当前决策**：如果标识只是在说明系统知识，不能帮助玩家在当前界面选择、进入、派遣、排序或判断风险，就不要显示。

常见分层：

- 地图层：只显示地图点、区域锁定 / 当前选中 / 地图点已有的事件数或状态。
- 详情层：显示选中对象的摘要，例如可见目标、限时机会、线索、深度链。
- 任务卡层：显示该任务自己的耗时、倒计时、需求、风险、链条下一步与操作。
- 派遣层：显示当前队伍、骰池、缺口、风险和可执行性。

同一个词在不同界面出现时，必须语义一致；如果只是聚合摘要，应明确放在摘要面板里，不能伪装成地图或任务卡图例。

状态标识必须贴近被状态化的对象。`已派遣`、`已完成`、`待结算`、`占用中`不是任务类型，也不是危险 / 对手骰这类任务压力，不应混进任务类型或机制标签行。若状态表达的是“谁正在占用这个任务”，地图点优先用角色小头像组，任务列表优先在任务名称旁放状态标记；微型文字角标只能作为辅助，不应承担主要识别。

### 4.2 只读预览不能复用可操作卡片语法

同一套视觉语法会让玩家推断同一套操作。若当前页面的真实任务是“先选择地区 / 对象，再进入下一层”，右侧预览区不能复用下一层的任务卡、主操作按钮、hover 高亮或 `interactive` 类；否则玩家会自然先去点预览条目。

硬规则：

- 可点击对象必须比只读预览更像按钮：有明确选中态、hover 反馈和主操作承接。
- 只读预览最多展示少量摘要行，默认 2 到 3 条；不要纵向铺成完整任务列表。
- 只读预览不放任务操作按钮，不使用任务卡的强边框、倒计时块、可派遣按钮或展开交互。
- 若预览内容来自下一层页面，必须用摘要语法，例如“地区情报概况 / 进入地区后选择具体任务”，而不是直接复刻下一层任务卡。
- 验收时检查预览容器内不应存在 `button`、`a`、`role=button` 或项目内表示可交互的类名，除非该预览本身就是当前页面的真实操作对象。

### 4.3 父级选择器与从属详情不能同级化

带有“选择对象 -> 查看对象信息 -> 确认进入 / 执行”的界面，必须先区分控制器和输出结果。选择器是玩家正在操作的父级控件；详情、数值、预览和说明是当前选中对象的从属信息。两者不能做成同一重量、同一边框、同一 hover 语言的并列卡片组。

硬规则：

- 父级选择器可以像按钮：允许明确选中态、hover 反馈、焦点边框和点击手势。
- 从属详情必须像属性面板或只读内容：降低边框和背景权重，不使用按钮式 hover，不出现 `cursor: pointer`，不与选择器共享强交互类。
- 如果某块信息会随选中对象改变，它应通过标题、缩进、分组标题、连接关系或内嵌位置表达“这是当前选中对象的详情”，而不是另一个可选项集合。
- 摘要数值、只读情报、风险说明和进度信息不得拥有和父级选择卡相同的卡片视觉重量；需要时可改为表格、属性行、轻量列表、分隔线内信息或折叠在选中对象下方。
- 不能只把从属详情“稍微降权”后仍保留卡片语法。可点击选择区和从属详情应使用不同控件语言：选择区像按钮组，有箭头、选中态、hover 和明确指向；详情区像注释 / 属性表 / 普通列表，用缩进、细线、轻量文字和透明背景表达附属关系。
- 设计验收时必须标出当前界面的四类角色：控制器、当前选中对象、从属详情、主操作。四者若看起来同级或同样可点，就必须重排层级。

### 4.4 可用与不可用状态必须拉开行动距离

可点击 / 可进入 / 可执行不是普通状态色差，而是玩家下一步行动的许可信号。不可用对象如果仍保留同等按钮边框、箭头、发光、饱和颜色或 hover 手感，玩家会把它理解为“也能点，只是状态不同”。可用对象必须有主动邀请感，不可用对象必须有明确阻断感和原因。

硬规则：

- 可用对象使用主动按钮语法：更高对比度、更饱和的主色、明确箭头 / 进入图标、hover 抬升、可聚焦边框和可点击光泽。
- 不可用对象使用锁定 / 幽影语法：降低不透明度与饱和度、去掉主箭头或换成锁 / 缺口图标、禁用 hover 抬升、弱化边框，并显示明确原因，例如 `未解锁 · 82%`、`缺口 18%`。
- 同一组里，不能只靠一行小字区分可用与不可用；形状、图标、对比度、hover、光效和文案至少三项同时拉开。
- 地图点、地区按钮、任务卡、道具、角色和主操作按钮都适用。只要某项当前不能执行，就不能保留和可执行项相同的按钮邀请感。

### 4.5 执行中对象应从主操作流下沉为队列

当一个对象已经被派遣、提交、锁定、等待结算或正在执行时，它不再属于当前主操作候选。此时不能只在原卡片上加 `已派遣` / `执行中` / `待结算` 小标识后继续留在可点击列表里；玩家会按原列表语法误以为它仍可作为下一步行动。

硬规则：

- 主列表只放当前可执行 / 可派遣 / 可选择的对象，保持最强按钮感、hover 和整卡点击。
- 已执行对象统一下沉到次级区块，例如 `已派遣 · 执行中`、`执行中队列`、`待结算`；这个区块应有弱框体或收纳语法，和主列表拉开层级。
- 不要默认使用 `等待回收` 这类会暗示玩家应该撤回外派的文案；正常派遣状态优先表达“谁在执行、几天后结算”。
- 已执行对象卡片应降低饱和度、去掉主按钮感和 hover 抬升，默认只读；如果需要查看详情，入口必须弱于主操作，不能和派遣 / 进入等主动作同权重。
- 次级队列里优先显示“谁在执行、还剩几天、何时回收 / 结算”，而不是继续强调任务需求、可派遣压力或主操作文案。
- 地图点可以保留头像组等占用状态，但右侧任务列表必须通过布局分区承担主要区分；不能只靠地图或标题旁小标识解决误点问题。

### 4.6 可识别对象的空间位置必须稳定

玩家会用空间位置记住对象。候选角色、道具、任务、地区、卡牌等列表 / 网格，一旦玩家开始比较和选择，点击、选中、取消、hover、展开详情、容量替换等状态变化都不应让原对象在同一列表里自动换位置。

硬规则：

- 选择状态只改变高亮、勾选、槽位摘要、按钮文案或详情预览，不参与候选列表排序。
- “按贡献 / 相关度 / 推荐度排序”应在进入页面、切换任务或玩家显式切换排序模式时确定；点击某个候选项后不得因为它已选中而移到前面。
- 若需要展示已选对象顺序，应使用单独的已选槽 / 本次骰池 / 队列区，而不是重排候选池本身。
- 排序比较必须有稳定兜底，例如角色基础顺序、创建时间或固定 id，避免同分对象在重渲染后漂移。
- 容量上限不能触发隐式替换。若已选人数 / 道具 / 卡槽达到上限，继续点击新候选时必须提示“已达上限，请先手动撤下一个已选对象”，并保持当前选择不变。不要用 `shift()`、自动撤下最早选择、替换最低贡献对象等方式替玩家决策。
- 悬停预览也要尊重容量上限：已满时只提示无法加入和解除方式，不模拟“自动替换后”的覆盖率、风险或收益。
- 悬停预览不能改变候选列表、卡片网格或鼠标命中区的位置。若预览内容会从一句话扩展成详情面板，预览容器必须按详情态预留高度；鼠标移出单张卡时不要立刻缩回导致布局跳动；连续划过卡片时应有短延迟或保留上一次预览，避免 `mouseenter` / `mouseleave` 循环触发闪动。

### 4.7 操作按钮

- 同一列表中的主要操作按钮应对齐。
- 固定宽度时必须保证文字不换行。
- 按钮文字应是动作：`派遣`、`抢派`、`继续调查`、`进入任务`。
- 行动按钮文案必须描述玩家要执行的中性动作，而不是系统预测的负面结果。不要把 `失败`、`惩罚`、`损伤`、`未达标` 这类结果词直接写进主按钮，例如不要写 `提交失败判定`；按钮应写 `提交判定`，当前会成功、未达标或产生代价放在状态区、结果预览或确认弹窗里说明。例外只限真正危险、不可逆且需要玩家确认的操作，例如 `删除存档`、`放弃任务`。
- 层级返回按钮不是普通标签。`返回区域`、`返回世界探索层`、`返回回合简报` 这类会把玩家带回上一层工作台的按钮，应有稳定的最小点击面积和清晰边框 / hover，不要缩成小型次级胶囊；它们可以低于当前页主动作，但不能低到难以发现。
- 放大返回按钮时优先调整按钮自身的 `min-width`、`min-height`、padding、字体和边框，不为了它重排页面容器、候选列表或主流程按钮位置。交付前要顺手检查同类返回按钮是否一致，避免只修截图里被指出的一处。
- 已派遣、执行中、待结算等状态不应继续显示派遣类主动作按钮，也不应继续保留和可派遣卡片同等的整卡点击强度；应按 §4.5 下沉为队列或只读状态。
- 如果一张卡片只有一个主动作，且整张卡片已经承担点击进入 / 打开 / 派遣配置的交互，就不要再在卡片内部放重复的小按钮。整卡点击和内嵌按钮二选一，避免双入口抢空间、误导权重或挤压倒计时 / 当前任务等特殊状态槽位。
- 内嵌按钮只在以下情况保留：同一卡片有多个动作需要区分；动作不可逆或需要确认；玩家需要在卡片内锁定 / 重投 / 提交；或者整卡点击语义不成立。

### 4.8 角标与徽章必须有安全槽位

数字角标、进度徽章、状态徽章、头像叠层等小元素，只要贴在按钮、地图点、卡片标题或操作区上，就必须有自己的布局槽位或明确预留安全区。不要用 `position:absolute` 直接压在右上角后只靠肉眼调偏移；如果元素可能变成两位数、百分比、长状态词或多头像，默认会遮挡文字。

硬规则：

- 角标表达的信息必须绑定当前对象，遵守 §4.1；不能为了装饰而贴角标。
- 角标和主文字不能共享同一片未预留空间；优先使用 flex / grid 的独立槽位，其次才是带足 `padding` 的绝对定位。
- 角标、徽章、头像组与主标题 / 按钮文字 / 倒计时 / 操作按钮之间必须有可读安全区，不只是不相交。
- 截图验收时，必须检查 `getBoundingClientRect()`：主文字矩形、角标矩形、操作按钮矩形不得重叠；地图点和任务卡这类高密度界面尤其要做自动断言。
- 状态数量可能变化时，必须测试边界文本，例如 `9+`、`80%`、`100%`、`剩余 10天`、多人头像组。

### 4.9 即时反馈必须贴近触发对象

玩家点击某个对象后，系统反馈必须出现在该对象附近，或出现在同一组件的固定反馈槽里。不能把局部对象的失败原因、未解锁原因、不可用原因丢到页面右上角、日志区或远离视线焦点的全局 toast 里；那只证明系统有消息，不证明玩家能看见。

硬规则：

- 对象级反馈应锚定触发对象：地图点、任务环、卡片、角色、道具、按钮的失败 / 未解锁 / 不可用提示，应出现在对象旁边或对象内部。
- 全局 toast 只适合全局结果：保存成功、报刊替换、阶段切换、后台结算等不依赖具体局部对象的位置反馈。
- 如果触发对象和提示之间跨越大半个屏幕，视为未通过验收；截图时必须按玩家真实视口检查反馈是否在第一眼可见范围内。
- 自动验收不能只查文案存在，还要检查提示框与触发对象的距离、是否在同一滚动容器内、是否被裁切或遮挡。

## 5. 任务卡槽位规则

任务卡是当前最容易出错的 UI，必须固定功能槽位。

### 5.1 标准槽位

- 左侧：任务类型图标。
- 标题行：任务名。
- 标题右侧：只放状态、强时效倒计时或特殊入口标识，不放普通耗时。例如“已派遣”应在任务名右侧，而不是混入耗时 / 危险 / 对手骰标签行。
- 标题下方第一标签：耗时，格式为 `耗时 2天`。
- 同一标签行：对手骰、危险、黑骰等压力信息。
- 需求行：探索 / 生存 / 理性等任务需求。
- 右下角：主要操作按钮。

同一类功能标识必须固定在同一信息槽，不得随任务类型漂移。

### 5.2 耗时

- 统一写成 `耗时 N天`，不要只写 `2天`。
- 同一界面内必须固定位置，当前任务卡规范为“标题下方第一枚标签”。
- 耗时是派遣成本，不是倒计时，不要和剩余时间共用槽位。

### 5.3 截稿 / 突发任务

- 截稿任务的重点是“剩余时间”，应使用更高视觉重量。
- 当前任务卡可使用右侧大块倒计时：`剩余 2天`。
- 地图点可写 `剩2天`，避免被理解成耗时。
- 不额外写“临时”“过期消失”，倒计时本身已经说明会消失。
- 在派遣配置页，红色截稿只需要一个主视觉承载：高权重倒计时块，例如 `截稿关闭 / 剩余 3天`。不要同时再放“红色截稿”标签和红色警戒条；后果说明应降级为一行文字，例如“成功或失败后都会结束；失败可能造成队员状态损伤”。

### 5.4 普通任务

- 不显示“普通 / 白色调查 / 可反复追踪 / 失败保留”等默认规则。
- 只显示当前决策需要的耗时、需求、对手骰、操作按钮。

### 5.5 线索

- 线索不是任务，应通过“调查 -> 新任务”这类图形关系表达。
- 不重复写“可能发现新任务”。
- 如果线索需要派人调查，应显示耗时、需求和行动按钮。

### 5.6 深度链任务

- 不在一开始显示 `1/4` 这类总阶段数，除非玩家已经知道总长度。
- 默认只显示当前可做的一环。
- 不使用 `下一页` 这类 UI 翻页语言，也不把后续提示混进普通标签行。
- 区域任务卡中，深度调查应使用右侧专属块显示 `深度调查 / 当前任务 N`；这代表“当前可做的一环”，不透露总长度。
- 已完成的深度链旧环不能继续留在区域任务列表主流里。完成一环后，旧环应退出当前可派遣列表，只保留最新可做的一环；如果需要回顾旧环，另做日志 / 已完成记录，不用和当前任务同样的紫色任务卡语法。
- 若要暗示后续存在，只用不剧透长度的调查语义，例如 `线索未尽`、`仍有下文`、`前序线索`；不要显示 `1/4`、`下一页：?` 或完整后续关卡数。
- 地图上用规整叠层表达“当前入口 + 潜在链”，不要一次摊开整条链。

### 5.7 危险、黑骰、对手骰

- 危险是难度 / 后果标识，不是任务类型。
- 高危任务可以合并进普通任务，用危险标识表达例外。
- 黑骰应有独立图形语言，和危险分开。
- 对手骰必须带文字读法，例如 `对手骰 3`，不要只给一个孤立数字。
- 不使用含义不明的“高压”标签；若要保留，必须改成明确规则词。

## 6. 地图与区域界面规则

### 6.1 地图语义

- 地图类界面必须有可识别空间语义：世界地图、区域轮廓、经纬网、航线、地形或城市结构。
- 不要用抽象圆斑、彩色光区替代地图。
- 若只是临时占位，也要让玩家看出“这是地图”。
- 地图上的图例 / 标识必须遵守 §4.1：只能解释地图层真实可见的状态。不要把右侧详情面板里的任务分类摘要搬到地图上；如果限时、深度链只在右侧卡片里表达，地图图例就不能出现对应标识。
- 全球 / 上层地图页若承担的是“选择地区 / 进入地区”，就不放 `下一天`、结算、结束探索这类时间推进动作。时间推进应放在玩家正在安排任务的当前地区 / 派遣上下文里；推进后默认回到当前地区继续派遣。
- 世界探索 / 全球地图页应优先采用“阶段导航 + 地图主语 + 选中地区详情 + 唯一主操作”的结构：顶部显示当前周流程阶段，地图承担首屏视觉主语，右侧只显示当前选中地区的从属情报预览，右下角主按钮进入地区 / 进入探索。右侧预览不能变成可点击任务列表；具体任务选择只在进入地区后发生。
- 全球地图里的地区情报标签按稳定语义分组：成本、收益、条件、挑战、信息、状态。不要让“可进入”这类入口文案重复占据标签位；地区选择卡应显示难度、推荐或解锁缺口这类能辅助地区决策的信息。行动日志默认折叠，避免压过地图和选中地区详情。
- 全球地图和地区情报预览里的 `条件` 只表示玩家需要满足 / 覆盖的需求，文案必须写成 `条件：探索2`、`条件：生存1` 这类“字段名 + 冒号 + 内容”。`对手骰`、危险、黑骰等是关卡挑战 / 压力，不属于条件，应写成 `挑战：对手骰2` 或进入专门压力槽位。
- 即使时间推进位于当前地区，也不能和任务列表 / 任务详情共用同一个选择容器或确认按钮槽。`日程推进`、`推进到第 N 天`、`结束探索周` 属于日循环系统动作，应作为区域面板底部或独立日程区的特殊动作呈现，并附带“已派遣 / 到期结算 / 倒计时变化”摘要；任务列表只承担“选择具体任务 / 进入派遣配置”。
- 区域页顶部不提供 `结束探索周 / 进入编辑部` 这类跨阶段跳转按钮；区域页的上层导航应返回世界探索层 / 世界取材地图，并按 §4.7 的层级返回按钮标准保持可见。真正结束探索周只能通过日程推进、周循环或明确的周末结算流程触发。

### 6.2 标签避让

地图层只要同时存在多种任务点、线索点或可展开入口，就必须有标签避让机制。

硬规则：

- 不靠手工坐标解决长期堆叠。
- 深度链、黑骰、截稿等高优先级入口优先保位。
- 普通任务和线索标签让位。
- 验收必须覆盖默认态、hover 态、展开态，以及宽 / 中 / 窄视口。
- 元素带 `transform` 过渡时，避让计算不能直接依赖动画中的 `getBoundingClientRect()`；应使用基础矩形 + 偏移量，或临时禁用过渡测量。
- 避让标准是“可读安全区”，不是“矩形刚好不相交”。叠层、阴影、描边光晕、hover 扇出范围都要计入保留区，标签之间应留出清晰空隙。
- 已派遣地图点应在任务点右上角显示派遣角色头像组。头像组属于任务点状态的一部分，需要参与视觉检查：不能太小到看不清，也不能遮住倒计时、任务名或邻近地图点。

### 6.3 深度链地图入口

- 地图只显示当前最新任务。
- 未完成第一环时，地图标识显示第一环。
- 完成第一环后，地图标识替换为第二环。
- 不在地图上同时显示整条链。
- 入口默认是规整叠层；hover 轻微错开；点击后可在同一对象中展开成线性链。
- 深度链地图入口的“展开 / 进入 / 收起”必须分离：第一次点击只展开链；展开后点击当前环任务直接进入任务界面；再次点击同一入口不负责收起。
- 收起属于退出上下文动作，应由点击任务之外的地图空白、切换对象或离开当前区域完成；不要把当前任务入口做成打开/关闭同一个 toggle。
- 展开态里只有当前环承担可执行入口，后续 `?` 环只表达潜在线索，不应有同等按钮感。
- 如果玩家点击未解锁的后续环，必须在该环附近弹出明确提示，例如 `该任务未解锁`；不能沉默，也不能把这次点击解释成收起、进入当前任务或远离对象的全局 toast。

## 7. 派遣配置与角色骰 UI

### 7.1 派遣配置页

派遣配置页要让玩家感到“我正在组本次骰池”，不是只在读属性表。

必须优先表达：

- 已选人数，例如 `已选 2/3`；
- 本次骰池预览；
- 角色骰与道具骰进入同一个骰池；
- 与当前任务相关的骰面；
- 需求覆盖、缺口、空面；
- 临时线人 / 一次性道具是否占人数或占骰池。

追加资源必须保持同组语义一致，并避免把“获取入口”和“阵容管理”混在同一个按钮里：一次性道具仍可在追加区点击选择 / 撤回；临时线人这类会变成完整角色的资源，一旦雇佣成功，就应从追加区移除购买入口，作为候选角色出现在下方列表。后续加入或撤回本次骰池，都通过候选角色卡或已选槽完成，而不是让右侧购买按钮继续承担撤回逻辑。

任务头部必须遵守信息层级：

- 任务名、任务性质、关键需求和耗时可以作为紧凑标题 / meta 出现。
- 耗时是执行成本，不单独占一个 UI 模块；可写入 `耗时 2天` 标签，或体现在 `开始调查（2天后判定）`。
- 探索周剩余天数是全局资源，不放进任务摘要数值组。
- 截止日期是红色截稿 / 突发任务的特殊约束，只在对应任务上以警戒条、倒计时或状态提示出现，不和常规耗时、需求并列。
- 若截止日期是本任务的最关键决策值，优先用右侧倒计时块承载；不要把剩余天数藏在长句里。
- 需求可以在任务头部轻量提示一次；具体覆盖、缺口和风险判断以需求覆盖面板为准。

配置页主操作只能围绕本次派遣：`开始调查`、`开始线索调查`、`返回区域`、`撤销派遣`。其中 `返回区域` 是重要层级导航，不能做得像普通小标签；但视觉重量仍应低于开始调查。全局 `下一天` / `结束探索` 不应作为浮动按钮出现在配置页，否则会压过本页决策，并可能遮挡候选角色。

候选角色区可以提供明确的排序标签，例如 `任务贡献`、`达标提升`、`未派遣优先`，让玩家知道当前候选池为什么这样排列。新增排序标签时只改变候选区排序语义，不应顺手重做任务头、本次骰池、需求覆盖面板或整页壳层；这类大结构必须单独确认。Hover 只能临时提示，不改变选择、不触发隐式替换、不重排候选池。

派遣配置默认应是正式游戏界面，而不是实验页读数面板。宏观属性、公式解释、排序原因、贡献拆解和调试说明只有在影响当前组骰池决策时才显示；同一信息不要同时出现在骰面、文字摘要和右侧预览里。候选角色卡已有 dice net 时，不要再在骰面上方逐字重复“人脉+2 / 洞察+2”等骰面内容；卡片只保留姓名、简短适配等级和骰面本体，详细贡献默认进入详情页。右侧面板默认只做最终复核，例如“是否达标、还差什么、能否开始”；不要把 hover 候选时的变化做成常驻四格调试面板。若确需 hover 提示，也只能是短句，并且不能新增页面高度变化、重复骰面内容或开发口径标签。

派遣配置里的任务需求由“需求覆盖 / 判定目标”区域主承载；任务头只负责身份锚点、耗时和特殊后果，例如截稿倒计时或失败伤害。若右侧已经逐项展示 `人脉 / 洞察` 的覆盖和缺口，任务头不要再写 `需求 人脉2 / 洞察2`，否则同一需求事实会拥有两个视觉主语。

### 7.2 风险提示

- 整体风险应优先看任务需求覆盖度和缺口。
- 不要用“相关骰面 / 总骰面”比例直接标整体风险。
- 加入低贡献角色不应让整体风险反向升高，除非规则明确规定会带来额外代价。
- 低贡献角色应提示“贡献低 / 无相关面 / 占用人数槽”，而不是让任务本身变危险。
- 如果界面显示“低 / 中 / 高风险”或“达标率”，必须使用与真实结算一致的公式。不要用整套骰面池的潜在覆盖、静态面值总和或候选排序分数替代实际掷骰概率；0% 或几乎不可达的组合必须显示为高风险 / 不可达。
- 风险标签不能只给粗粒度等级。若 `低 / 中 / 高` 会让明显不同的概率落在同一档，必须在高权重位置同时显示实际达标率，例如 `风险：中 · 54%`；多人队伍相对单人最佳的提升应在说明中显式写出，例如 `较单人最佳 +21%`。
- 角色骰当前口径：任务需求属性决定哪些骰面算相关；普通任务达标看相关骰面有效点是否达到任务目标，而不是每个属性在单次掷骰中逐项硬达标。UI 应同时显示有效点目标和实际达标率，属性条用于解释相关贡献来源。
- 角色骰判定 UI 中，`有效点 / 目标值 / 达标率` 是判定结果口径；`理性 / 洞察 / 探索` 等是属性贡献拆解。两者不能做成同一行同等样式的进度条。正确结构是先给玩家一个判定目标 / 当前结果摘要，再用次级属性条或展开明细解释哪些属性贡献了这些有效点。
- 如果真实达标率暴露出适配队伍仍几乎不可能成功，不能只改风险标签或文案。UI 规则要反向提醒设计：检查任务需求、目标公式、骰面密度和支援资源，让高风险任务保持“可搏”，而不是把不可玩状态包装成准确提示。

### 7.3 角色骰表现

- 角色骰 UI 必须让玩家联想到骰子。
- 可参考骰面展开、骰子网格、六面展示等方式，而不是普通资料面板。
- 派遣配置页里的候选角色卡、已选角色槽和角色详情，只要展示完整 6 面角色骰，就应优先使用骰子立方体摊开的“长尾十字架 / dice net”结构；不要退回普通横向 chip 行。横向 chip 行只适合极小摘要、日志或一次性道具简表。
- 角色卡默认只显示专精可以，但与当前任务相关的骰面必须更亮、更大。
- 角色详情应先给当前任务适配结论，例如“相关面 3/6，空面 1”。
- 不相关面降级显示。
- 升级分支默认折叠，避免干扰派遣决策。

## 8. 布局与间距规则

### 8.0 避免随意拉伸

除特殊说明外，界面元素不应因为选项变少、内容变短或某个槽位被消耗，就自动拉伸填满剩余空间。固定格式的按钮组、卡槽、资源格、工具条、骰池、候选区等，应优先保持稳定槽位、固定列宽或明确空位，让玩家感知“这里原本有一个位置，现在被使用 / 移除 / 空着”。随意拉伸会削弱品质感，也会破坏玩家对数量、消耗和空间关系的判断。

允许拉伸的特殊情况必须有明确目的，例如正文阅读区域、弹性表格主列、响应式单列移动布局，或内容本身就是需要填满容器的画布 / 地图 / 时间轴。做这类拉伸前要先判断：玩家是否需要感知固定槽位；如果需要，宁可留空也不要让剩余元素变宽。

### 8.1 删减后必须回收空间

删除信息后，必须同步检查：

- `gap`；
- `padding`；
- `min-height`；
- grid 行列占位；
- 绝对定位元素是否还在撑开正文；
- 操作按钮和需求行是否仍对齐。

不能只确认文字消失。内容少了，空间也必须变紧凑。

### 8.2 功能标识位置一致

- 同一类功能信息必须在同一位置出现。
- 如果某任务没有该信息，槽位可以缺省，但已有信息不能换位置。
- 不要因为某张卡有倒计时，就把普通耗时挤到另一侧。

## 9. 骰子判定与玩家控制

- 任何强调单颗骰子、锁定、干预、污染的玩法，应控制骰子总数，默认单次判定不超过 10 颗。
- 骰子判定页和骰子实验页必须显示当前任务需要哪些属性与几点；有效点、达标状态和结果结算必须按真实任务需求计算，不能把所有骰面数值简单相加。
- 骰子判定进入“停骰选择 / 计入选择”阶段后，骰面本身必须成为清晰可点击入口，且选择后固定计入槽、有效点、属性贡献拆解和提交前结果预览要同步变化。玩家不能先提交，才知道本次选择会成功还是失败。
- 骰子判定的提交按钮保持中性动作文案，例如 `提交判定`。不要因为当前未达标就把按钮改成 `提交失败判定` 或染成“不能点”的惩罚语义；未达标、成功、代价和后果应由判定目标、提交预览或二次确认承担。
- 骰子判定页必须为每类信息指定唯一位置：达标 / 差值只在判定摘要表达；属性需求与当前贡献只在目标拆解表达；重投次数只在控制区表达；重投对象只在重投范围附近表达；提交区只表达提交后的后果。不要在标题、顶部徽章、按钮、预览和进度条里反复显示同一个点数或同一句阻断原因。
- 计入槽位应保持固定数量和固定尺寸，空槽继续占位；不要因为已选骰面变少或变多就拉伸组件。空位本身用于表达“这里还可以计入 / 这里已经空着”，也是玩家判断容量的一部分。
- 骰子判定交互实验的首轮默认只验证骰面卡片、需求实时联动和提交前预览；一键推荐、黑骰扩展、完整投骰动效和结果演出要作为后续单独确认项，不要和基础选择入口混成一版整页重做。
- 玩家需要决策时，不自动推进。
- 锁定、重投、确认类动作必须有明确按钮。
- 不用 2 秒自动倒计时替玩家继续，除非用户明确要求快节奏。
- 新机制不能只在结算文案里出现，必须在玩家决策现场被看到、操作到或感知到。
- 骰子判定区是本页主舞台。有效点目标、属性贡献、黑骰说明、计入槽和按钮都必须服务骰子决策，不能把骰子盘挤到页面下沿或做成附属列表。
- 骰子判定弹窗适配 3 到 5 名参判角色时，讨论对象是“参判角色 / 可计入骰面”，不是派遣配置人数。优先使用最多 5 个固定参判槽：3 人判定显示 3 个可计入槽 + 2 个未参判占位，5 人判定打开全部槽位；骰面卡片也按同一 5 槽骨架展示，未参判角色降级为不可点击占位。人数变化不能拉伸卡片、挤压右侧判定目标，也不能自动取消已计入骰面。
- 摇骰表现不能只是数字 / 文案原地闪烁。至少要让玩家看到“实体骰子在盘面中运动”的线索：独立骰盘、骰子本体、位移、旋转、落地反弹、阴影或轨迹。停骰后再进入选择 / 锁定 / 重投阶段，避免像翻牌或刷新标签。
- 如果视觉目标是“投骰”，骰子本体不能做成只有一个面的薄片翻转。必须表达厚度和多个可见面：真实几何体模型、六面结构、侧面明暗、接触阴影、抛物线和多轴旋转至少覆盖其中几项；否则玩家会把它读成卡牌 / 令牌，而不是骰子。若 CSS 3D 仍被用户读成薄片，应停止继续微调 CSS，转向 Three.js / Godot / Blender 等真实模型路线。
- 当方案声称是“立方体 / 实体骰 / 建模骰”时，验收门槛是截图里能读出闭合体积，而不是“画了多个面片”。至少应看到互相连接的顶面与侧面、边线 / 角点能闭合、面与面之间没有漂浮缝隙、遮挡关系与投影一致；如果任一面仍像独立纸片、卡牌或散开的贴片，就必须判定为未通过，不能称为立方体模型。
- 实体骰的面内容必须贴合骰面。骰点、属性文字、符号、贴图、划痕和发光都应服从所在面的透视、旋转、遮挡和边界；不能像 2D 字幕一样浮在屏幕上，不能越过面边缘，不能和面角度相反，也不能为了闭合外轮廓牺牲已成立的贴面效果。若旧版或参考图中某一项已经表现良好，例如文字贴面、投影、落地阴影或材质厚度，返工前必须列入保留项并在新截图中回归检查。
- 实体骰的文字、骰点和符号应尽量在建模 / 材质 / 贴图阶段成为骰面的一部分，而不是投影完成后临时画在屏幕坐标上。Canvas 原型若不用真正 3D 引擎，也应使用离屏面贴图再映射到投影四边形；不能用独立 2D 标签模拟贴面，否则容易出现对位不准、文字角度与面不一致、被裁切或偶发不显示。
- 实体骰的面材质应优先保证可读性。默认骰面不能像透明线框盒；背面、内部结构线、调试线或被遮挡边不能穿过正面文字 / 骰点。除非用户明确要求“透明骰”风格，否则应使用不透明或高遮蔽度面材、暗色面底、清晰外轮廓和少量必要转角线，参考旧版骰子的“实心面 + 高亮边 + 文字在面内”效果。判断面是否不透明时只看截图读法：若背景网格、轨迹环、背面边或另一面的结构线在骰面内部仍清楚可见，或面主要靠线框而不是色块成立，就判定为透明 / 线框感未通过；不能用代码里 `fill()` 过多边形来替代视觉结论。
- 立方体骰子的视觉检查必须做反向提问：玩家第一眼会不会把它读成“薄片 / 几张面 / 翻牌”？如果答案有明显可能是“会”，即使动效、投影、发光、轨迹都存在，也不能继续交付给用户看作实体方案。
- 骰子视觉验收至少包含五类条件：盒体闭合、骰面不透明且可读、面内容贴合、光影投影一致、动作落地可信。只通过其中一类不算通过；尤其不能用“几何结构正确”替代“截图里读起来正确”。
- 如果用户要求“不改变原判定界面布局，只改骰子表现”，动效实验必须嵌在原判定弹窗的信息结构中验证：保留标题、判定摘要、计入槽、骰子判定区、侧栏状态和底部按钮的相对关系，只替换骰子盘内部表现。不要为了展示动效另起一个全屏大舞台，除非用户明确要求脱离原界面比较。
- 当用户只要求应用某个视觉对象、模型或动效时，默认改动边界是“替换该对象本身”，不是重排承载界面。若落地时发现模型尺寸、镜头或动效需要更大空间，必须先问用户是否允许调整布局；不能为了展示效果擅自把列表、卡片网格、进度面板、按钮区或弹窗信息顺序改掉。验收时要把“布局是否仍是原布局”作为独立可否决项。

## 10. 实验页规则

实验页不是低标准页面。

实验页也不是正式制作页。它的价值是用很低成本回答一个清晰问题，而不是提前做一套完整 UI。

### 10.1 快速实验页的时间盒

创建实验页前先写清楚三件事：本页只回答什么问题、只比较哪些局部状态、什么时候停止。若这三件事说不清，优先回到本体做最小切片，或先用文字 / 草图讨论。

快速实验页的默认约束：

- 默认只做局部组件，不复刻完整页面、完整流程、完整背景和完整数据。
- 默认复用现有 CSS 变量、组件类、真实截图或少量静态 DOM；不要为了实验页新写一整套视觉系统。
- 一次只验证 1 个核心判断，最多并排 2 到 3 个状态或方案。
- 交互只保留能回答问题的最小动作，例如 hover、选中、禁用反馈；不补齐正式流程、教程、日志、入口和全链状态。
- 截图默认 1 张关键图；除非用户要求，不做移动端、长链路、汇总页和多状态全覆盖。
- 若实验页实现、验证和文档成本已经接近直接改本体，必须停止扩写实验页，改为在本体承载页做最小落地。

实验页交付时要明确标注“这是局部实验，不是本体已落地”。用户确认方向后，才把选中的局部语言迁移到主界面，并在主界面做真实路径验证。

### 10.2 是否需要独立实验页

默认不为已有游戏本体 UI 另起“假实验页”。如果方向已经明确，目标是修改现有试玩流程，应优先在本体承载页做最小切片，并用真实数据、真实入口、真实状态验证。

只有满足以下情况之一，才先做独立实验页：

- 方向尚未明确，需要并排比较 2-3 个视觉 / 交互方案；
- 新系统没有现成承载页面，必须先搭一个低成本操作面；
- 直接改本体会污染正式流程，且无法用参数、临时入口或本体 lab 模式隔离；
- 用户明确要求先做静态实验页或视觉草稿。

若只是担心风险，优先在本体增加临时参数、隐藏入口或 lab 模式，不默认另写一套假数据、假交互、假 CSS。独立实验页只能验证构图和局部手感，不能替代本体路径验收。

### 10.3 实验页质量底线

只要实验页复用正式 UI 语义，就必须继承正式原型的基础保护：

- 地图标签避让；
- 碰撞审计；
- 响应式检查；
- 状态截图；
- 操作按钮对齐；
- 文案去冗余；
- 信息槽位一致。

实验页可以粗糙，但不能把已经解决过的问题重新带回来。

### 10.4 动效演示页

当用户要求查看动效、动效语言、动画手感或多个动作方案时，应优先单独制作“系列演示页”，不要只把动效散落在正式流程里让用户反复触发。

动效演示页必须：

- 按动作类型分组展示，例如摇骰、停骰、计入飞入、黑骰介入、锁定 / 重投、结果收束；
- 每组都有可手动重播按钮，必要时提供“重播全部 / 暂停循环”；
- 明确标注这是视觉实验页，尚未套用到完整全链或正式流程；
- 保留窄屏可用性与截图验收，不能因为是动效实验就忽略按钮可见、文字溢出或布局挤压；
- 用户确认某个动效进入主干后，再把选中的动作最小化移植到真实承载页面，不默认把实验页全部动效塞进正式流程。
- 实验页方案落地时，必须迁移“被用户认可的视觉主体和实现类别”，不能只迁移动作名、文案或参数名。若实验页通过的是 Canvas / WebGL / Godot / Blender 等实体模型，本体落地不能降级成 CSS 2D 方块、贴片、线框或屏幕标签来冒充同一方案；除非用户明确同意“只先做低保真占位”，否则这类替身实现默认未通过。
- 实验页落地验收必须做基准对照：至少同时查看“已通过实验页截图”和“本体落地截图”的同类局部，逐项比对视觉主体、材质 / 不透明度、内容贴面、动作轴向、节奏、停稳状态和原界面布局。只要本体截图第一眼不像实验页中被确认的对象，就应先返工，不得用“代码也叫横轴急转”“动画属性已变更”或“流程可运行”当作完成依据。
- 如果某个动效直接落入本体后被反馈“表现奇怪”，应优先撤回或隔离本体改动，回到独立动效实验页比较动作语言；不要在正式试玩里继续堆叠临时动画修补。
- 投骰动效实验页应优先用低成本程序化实体方案验证手感，例如 Three.js 程序化模型或 Godot 内置网格；CSS 3D 只能作为极早期草图，不能作为已被质疑薄片感后的主方向。只有当实体方向被确认且需要长期材质、破损、污渍、异化等细节时，才进入正式 Blender / GLB 资产制作。
- 动效实验页必须按用户实际打开方式验收。若交付的是可双击打开的 HTML，不要依赖会被 `file://` 拦截的 ES module、远程 CDN 或本地 HTTP 服务；除非最终同时提供并验证可用的预览 URL。截图脚本通过本地服务不等于用户直接打开也能显示。

### 10.5 实验落地与主线保护

完整全链试玩、Godot 主场景和其它用户日常打开的默认入口，都视为稳定承载页，不直接承接未经确认的实验成果。

- 视觉、动效、布局、模型、地图重绘、入口路由等实验，默认先放在 `labs/`、独立实验页、隐藏入口或明确 query flag 中；用户确认后，才允许迁移到默认试玩入口。
- 实验迁移到完整全链前，必须先留基准截图。HTML 全链至少覆盖：根入口、全球地图、地区页、目标配置 / 判定页；Godot 或其它原型按同等关键路径截图。改完后必须用同一路径、同一视口再截图对照。
- 单次落地只改用户确认的局部。若需求是“只替换骰子表现”，不能顺手改地图、派遣配置、主流程布局、入口版本号或全局视觉系统；确需扩大范围时，先向用户说明并等待确认。
- 高风险 UI 改动必须有隔离开关或回退路径。包括 3D / 实体模型、地图背景重绘、派遣布局重构、默认入口切换、内联 HTML 打包体改写、跨文件同步改动等。默认入口保持稳定，实验入口明确标注实验性质。
- 多文件承载页必须说明真源并保持同步。像 `world-mysteries-full-chain.html`、`world-mysteries-full-chain.js`、`world-mysteries-full-chain-head.htm` 这类拆分 / 内联并存的页面，不能只改其中一个文件造成主线与生成体分叉；若某个文件是产物，应通过既定脚本生成或在交付中明确同步结果。
- 若用户指出“主线界面乱了”，优先查入口是否被实验版本号、旧版 query、缓存或内联产物污染；在用户确认正确基准截图前，不直接回退或继续叠加修补。

## 11. 截图、入口与版本验收

### 11.1 截图文件

可见 UI / 原型改动必须交付真实截图。

- 不复用旧截图文件名，避免聊天端缓存旧图。
- 文件名应带日期和改动主题，例如 `2026-05-12-unified-duration-slot.png`。
- 1-3 张截图直接贴对话；超过 3 张整理汇总页。
- 直接贴图必须确认使用能被当前对话渲染的 Markdown 图片语法。Windows 本地截图用正斜杠绝对路径，例如 `![截图](D:/angos/docs/screenshots/.../01.png)`；不要只给 `D:\...` 路径文本，也不要把不可渲染的路径误当成已展示截图。
- 返工后的截图必须使用新文件名，并在最终回复前实际打开查看。自动化审计只能证明 DOM 条件，不能替代肉眼确认截图里玩家看到的文字、位置和视觉层级。

复杂 UI / 分阶段 UI 重构交付时，优先提供“无标注真实截图 + 程序化标注截图”组合。标注截图必须基于真实页面 DOM 或游戏 UI 元素位置绘制框线、编号与短说明；不要把 AI 生图标注当作验收依据。

### 11.2 截图前检查

交付前逐项检查：

- 是否有标签堆叠；
- 同类功能标识位置是否一致；
- 是否仍有冗余说明；
- 删除内容后空间是否回收；
- 按钮是否对齐且不换行；
- 文字是否溢出、重叠或遮挡；
- 角标、徽章、头像组是否有独立槽位或安全区，是否遮挡标题、按钮文字、倒计时或地图点文字；
- 地图 / 卡片 / 弹窗是否在当前视口正常；
- 若涉及 hover / 展开，是否截图或审计对应状态；
- 是否有缓存旧图风险。

视觉对象还要做一次“目标读法”检查。这不是立方体专用规则，而是所有 UI、动效、模型、图标、状态、布局和视觉符号的通用验收法：

- 先把用户需求里的视觉名词翻译成可观察条件。用户说“立方体”，AI 必须先列出闭合体积、六面或等价可见结构、共享边角、厚度、遮挡、投影；用户说“可点击明显”，AI 必须先列出按钮形状、对比度、hover、指针、文案和 disabled 差异；用户说“不要像任务卡”，AI 必须先列出任务卡语法并确认新方案避开它。
- 返工前必须列出旧版或参考图里已经成立、不能丢的优点。视觉修改不是只消灭一个缺陷；如果新方案修好了 A 却破坏了原本成立的 B，也视为未通过。旧版优点可以是文字贴面、尺寸比例、按钮位置、动效节奏、投影、颜色层级或玩家已经认可的布局关系。
- 同时写出禁止误读项。每次视觉验收都要有一句“它不能被读成什么”，例如“不能像薄片 / 卡牌 / 散开的面片”“不能像可点击任务”“不能像同级模块”“不能像鼓励回收”。后续截图先检查禁止误读是否还存在。
- 实现完成后必须逐条对照这些条件，而不是只问“有没有渲染 / 有没有动画 / 有没有改代码”。自动断言、非空像素、动画播放和整体氛围只能证明页面有东西，不能证明视觉语义成立。
- 查看整页截图后，必须再裁切关键对象区域，以 100% 和放大视图肉眼检查；不能只用远景整图判断。复杂页面还要同时检查默认态、hover / 展开态、选中态、禁用态和用户截图指出的局部。
- 如果截图第一眼读法和方案命名冲突，以截图读法为准，先返工，不要用实现意图、代码结构或“我本来想表达”解释视觉结果。
- 对被用户连续纠偏过的问题，下一次同类工作必须先做回归检查：旧错误是否又出现、旧词汇是否还残留、旧视觉语法是否换了名字继续存在。
- 当用户要求“把实验页方案落地”时，必须检查是否出现“同名不同物”的降级：本体中的对象是否仍是实验页确认的同一种视觉对象，还是被换成了更低保真的按钮、卡片、标签、图标或 CSS 效果。若需求是实体骰，截图必须像实体骰；若需求是地图符号，截图必须像同一套地图符号；若需求是父子详情，截图必须保留同一层级关系。不能只保留方案名称、颜色、文案或某个 transform 参数。
- 模型 / 图标本体不要替状态框背锅。选中态、相关态、风险态应优先落在所属卡片、槽位或提交区；不要把黄色框、白色外环、轨迹环等装饰直接套在实体骰、道具、地图符号上，除非该装饰本身是明确的世界内效果。实体骰要先读成有厚度、有遮挡、有暗边和落地阴影的骰子，再考虑动效。
- 如果 AI 无法自信判断视觉对象是否成立，应先降级为“待确认视觉草图”，或给出局部截图请用户判断；不能把未通过自检的方案包装成已完成。
- 连续返工后的总结必须能举一反三：把问题拆成“对象主体、内容归属、材质 / 遮挡、状态动作、旧版优点、禁止误读”几个维度。不要只写“这个骰子要这样做”；应写成“任何声称某内容属于某个视觉对象时，都要验证内容是否随对象一起变形、遮挡、缩放和状态切换”。地图标签、按钮角标、卡片徽章、模型贴图、图标动效都适用同一原则。

### 11.3 截图数量与对话负载

- 截图较多时，优先生成截图汇总页，并在对话里只贴最关键的 1-3 张。
- 如果当前对话已经因为大量截图、长链路 UI 讨论或历史上下文变得明显繁重，AI 应主动提醒用户：建议新建对话，并给出当前任务状态、关键文件和截图目录，避免继续堆上下文导致未响应或卡死。

### 11.4 网页入口

- 根目录入口必须能清楚区分当前试玩原型、旧参考页、实验页。
- 截至 2026-05-27，根 `index.html` 在本仓库承担开发导航职责，不等同于正式玩家入口。做冗余信息审计时必须先判定页面角色：开发导航可以保留实验目录和旧参考入口，但必须标注清楚；完整全链试玩等主流程承载页则不能把演示、调试、视图切换控件暴露给玩家。
- 按钮文字要说明实际含义，例如“试玩当前原型”“打开符号实验页”，不要让用户误以为某个实验页是正式游戏。
- 如果存在缓存风险，入口链接应带版本参数或说明当前版本。
- 新增实验页后，应考虑是否需要在根入口放临时入口，避免用户翻目录。
- 如果用户要求“在 index 体验到某个刚确认的 UI 改动”，根入口的主按钮必须指向该改动所在页面；不能只把它藏在次级卡片里。
- 如果完整全链原型还没有套用某个实验页 UI，入口必须明确标注“旧壳层 / 次级入口 / 尚未套用”，避免用户误以为改动回退。
- 当用户确认某个实验页 UI 应进入完整版或主干时，必须同步所有实际承载页，不得只改 index 文案。验收至少打开一次 `index -> 完整全链 -> 目标界面` 的真实浏览器路径，并截图确认与当前 UI 规则一致。
- 交付 HTML 实验页、动效演示页或可交互原型时，必须同时提供“浏览器可直接播放 / 操作”的入口，例如 `http://127.0.0.1:端口/...` 或明确可双击打开的 HTML 路径；不能只给仓库源码文件链接。
- 默认试玩入口不得指向未经完整路径验收的实验版本号。若为了破缓存调整 `?v=...`，必须确认该版本已经通过根入口、全球地图、地区页、目标配置 / 判定页截图；否则保持默认入口指向最后确认的稳定版本。
- 若对话中的文件链接会打开编辑器源码，应明确标注“源码文件”，并另给“预览网页”链接。用户要体验动效时，默认应打开预览网页而不是源码文件。
- 行动日志不默认等同于开发日志或噪音。若它承载玩家行动回顾、现实剧情文字、因果反馈或世界状态变化，可以保留为叙事组件；只有纯调试输出、公式流水或重复主信息的日志才应删除、折叠或降级。

### 11.5 截图与网页版本一致

- 给用户的截图必须来自当前页面的真实浏览器截图。
- 不复用旧截图文件名。
- 如果用户指出网页和截图不一致，先检查缓存、截图裁切范围、是否打开了旧入口，不要直接解释成用户看错。

### 11.6 美术风格稿与 UI 原型

- 玩法 UI 原型先验证功能呈现和交互，不等同于最终 GUI 包装。
- 正式美术风格稿确定后，可用“玩法原型截图 + 风格稿”生成目标效果图。
- 效果图不能替代可操作原型；必须再拆解回组件、状态、槽位和实现规则。
- 临时风格尝试不要自动写入正式设计文档，除非用户明确要求沉淀。
