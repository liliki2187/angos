> **用途**：给后续 AI / 新对话在处理 Angus 的 UI、交互、原型呈现、视觉实验页时使用。
> **定位**：这是执行规则手册，不是错题本，也不是玩法真源。
> **最后更新**：2026-05-13

## 2. UI 改动前的工作合约

每次动手前，先在心里或回复中明确三件事：

- **保留**：用户已经认可的结构、位置、交互、视觉重点是什么；
- **修改**：这次只解决哪个具体问题；
- **不做**：哪些相邻问题暂不扩展，避免把局部修正变成整体重构。

默认原则：

- 用户指出“某个元素不明确”时，第一轮只改该元素的文本、图标、提示或状态表达。
- 不要因为一个标签不清楚，就重排整张卡片。
- 不要把用户认可的亮点丢掉，例如右侧大倒计时块、规整深度链叠层、熟悉的 HTML 原型壳层。

## 3. 玩家信息过滤规则

玩家界面只显示会改变当前决策的信息。

### 3.1 应显示

- 耗时、成本、人数限制；
- 需求属性 / 需求骰面；
- 对手骰、黑骰、危险后果；
- 倒计时、截止、即将消失；
- 当前可用操作；
- 当前任务链阶段和下一步可见信息；
- 角色与当前任务的适配结论、相关骰面、明显缺口。

### 3.2 不应显示

- 内部任务类型说明，例如“白色调查”“红色截稿”这类开发分类；
- 默认规则说明，例如普通任务“可反复追踪”“失败保留”；
- 已由另一个 UI 元素表达过的重复说明；
- 含义不明确的标签，例如没有明确玩法后果的“高压”；
- 为了解释系统而写的说明句，除非玩家必须凭它做选择。

### 3.3 一语义一位置

同一语义只保留一处表达。

示例：

- 右侧倒计时已经写“剩余 2天”，就不要再写“过期消失”。
- 线索卡已有“调查 -> 新任务”示意，就不要再写“可能发现新任务”。
- 危险已经有“危险”标签，就不要再放孤立感叹号。

### 3.4 标识必须绑定本界面的真实对象

图标、标签、徽章、图例、颜色摘要、数字角标都属于“标识”。标识不是装饰，必须回答：它在这个界面里标记了哪个真实可见对象，表达了哪个当前状态，玩家能否凭它做本界面的决策。

新增任何标识前必须检查：

- **对象是否在本界面可见**：标识必须贴近或解释当前界面里的对象，例如地图点、任务卡、角色、按钮、区域详情；不能借用另一个面板或下一层页面里的对象。
- **状态是否在本界面成立**：当前选中、锁定、已派遣、剩余时间、危险、黑骰、深度链等状态，只能出现在真正承载该状态的界面层。
- **编码是否有对应视觉**：图例只能解释同一层里真实出现的颜色、图形、点形、线型或角标；没有对应编码时不放图例。
- **是否影响当前决策**：如果标识只是在说明系统知识，不能帮助玩家在当前界面选择、进入、派遣、排序或判断风险，就不要显示。

常见分层：

- 地图层：只显示地图点、区域锁定 / 当前选中 / 地图点已有的事件数或状态。
- 详情层：显示选中对象的摘要，例如可见目标、限时机会、线索、深度链。
- 任务卡层：显示该任务自己的耗时、倒计时、需求、风险、链条下一步与操作。
- 派遣层：显示当前队伍、骰池、缺口、风险和可执行性。

同一个词在不同界面出现时，必须语义一致；如果只是聚合摘要，应明确放在摘要面板里，不能伪装成地图或任务卡图例。

状态标识必须贴近被状态化的对象。`已派遣`、`已完成`、`待结算`、`占用中`不是任务类型，也不是危险 / 对手骰这类任务压力，不应混进任务类型或机制标签行。若状态表达的是“谁正在占用这个任务”，地图点优先用角色小头像组，任务列表优先在任务名称旁放状态标记；微型文字角标只能作为辅助，不应承担主要识别。

### 3.4.1 角标与徽章必须有安全槽位

数字角标、进度徽章、状态徽章、头像叠层等小元素，只要贴在按钮、地图点、卡片标题或操作区上，就必须有自己的布局槽位或明确预留安全区。不要用 `position:absolute` 直接压在右上角后只靠肉眼调偏移；如果元素可能变成两位数、百分比、长状态词或多头像，默认会遮挡文字。

硬规则：

- 角标表达的信息必须绑定当前对象，遵守 §3.4；不能为了装饰而贴角标。
- 角标和主文字不能共享同一片未预留空间；优先使用 flex / grid 的独立槽位，其次才是带足 `padding` 的绝对定位。
- 角标、徽章、头像组与主标题 / 按钮文字 / 倒计时 / 操作按钮之间必须有可读安全区，不只是不相交。
- 截图验收时，必须检查 `getBoundingClientRect()`：主文字矩形、角标矩形、操作按钮矩形不得重叠；地图点和任务卡这类高密度界面尤其要做自动断言。
- 状态数量可能变化时，必须测试边界文本，例如 `9+`、`80%`、`100%`、`剩余 10天`、多人头像组。

### 3.4.2 只读预览不能复用可操作卡片语法

同一套视觉语法会让玩家推断同一套操作。若当前页面的真实任务是“先选择地区 / 对象，再进入下一层”，右侧预览区不能复用下一层的任务卡、主操作按钮、hover 高亮或 `interactive` 类；否则玩家会自然先去点预览条目。

硬规则：

- 可点击对象必须比只读预览更像按钮：有明确选中态、hover 反馈和主操作承接。
- 只读预览最多展示少量摘要行，默认 2 到 3 条；不要纵向铺成完整任务列表。
- 只读预览不放任务操作按钮，不使用任务卡的强边框、倒计时块、可派遣按钮或展开交互。
- 若预览内容来自下一层页面，必须用摘要语法，例如“地区情报概况 / 进入地区后选择具体任务”，而不是直接复刻下一层任务卡。
- 验收时检查预览容器内不应存在 `button`、`a`、`role=button` 或项目内表示可交互的类名，除非该预览本身就是当前页面的真实操作对象。

### 3.5 信息层级不能用并列模块偷懒

不要把不同性质、不同作用域、不同触发条件的信息做成同一排同等重量的卡片或按钮。并列布局会暗示这些信息同级、同类、同等重要；如果事实不是这样，就会误导玩家。

设计任何信息组前先判断每项信息属于哪一层：

- **全局资源**：例如探索周剩余天数、资金、宏观指标。应放在全局状态区，不伪装成当前任务属性。
- **任务常规属性**：例如耗时、基础需求、主要奖励 / 风险摘要。用轻量 meta 或任务头部表达，不为低频字段单独开大模块。
- **特殊约束**：例如红色截稿截止、突发任务关闭条件、黑骰介入、危险后果。只在条件成立时用警戒条、状态徽章或局部提示表达，不能和普通属性并列成常规字段。
- **操作可行性**：例如能否开始、还缺什么、谁被占用。应靠近主操作和需求覆盖面板。
- **解释性文字**：只在玩家必须凭它做选择时出现；能由结构、标签、按钮或反馈表达的，不另写说明。

典型反例：派遣页把 `耗时 2天`、`剩余 7天`、`截止 3天`、`需求 探索5 / 生存3` 做成四个并列大框。这里“剩余”是全局资源，“截止”是红色任务特殊约束，“耗时”只是执行成本，“需求”才是本页配骰决策核心。正确做法是把全局剩余移回全局状态，耗时压成轻量 meta 或按钮文案，截止做成红色截稿警戒提示，需求与需求覆盖面板形成主线。

删掉重复信息后，必须重新指定“视觉主语”。不要只把重复标签合并成一句低权重说明；若该信息是玩家当前最该看的数值，例如截稿剩余天数，就要给它独立且明确的视觉承载。信息去重的目标是让关键值更突出，不是让关键值变成说明句的一部分。

## 4. 任务卡槽位规则

任务卡是当前最容易出错的 UI，必须固定功能槽位。

### 4.1 标准槽位

- 左侧：任务类型图标。
- 标题行：任务名。
- 标题右侧：只放状态、强时效倒计时或特殊入口标识，不放普通耗时。例如“已派遣”应在任务名右侧，而不是混入耗时 / 危险 / 对手骰标签行。
- 标题下方第一标签：耗时，格式为 `耗时 2天`。
- 同一标签行：对手骰、危险、黑骰等压力信息。
- 需求行：探索 / 生存 / 理性等任务需求。
- 右下角：主要操作按钮。

同一类功能标识必须固定在同一信息槽，不得随任务类型漂移。

### 4.2 耗时

- 统一写成 `耗时 N天`，不要只写 `2天`。
- 同一界面内必须固定位置，当前任务卡规范为“标题下方第一枚标签”。
- 耗时是派遣成本，不是倒计时，不要和剩余时间共用槽位。

### 4.3 截稿 / 突发任务

- 截稿任务的重点是“剩余时间”，应使用更高视觉重量。
- 当前任务卡可使用右侧大块倒计时：`剩余 2天`。
- 地图点可写 `剩2天`，避免被理解成耗时。
- 不额外写“临时”“过期消失”，倒计时本身已经说明会消失。
- 在派遣配置页，红色截稿只需要一个主视觉承载：高权重倒计时块，例如 `截稿关闭 / 剩余 3天`。不要同时再放“红色截稿”标签和红色警戒条；后果说明应降级为一行文字，例如“成功或失败后都会结束；失败可能造成队员状态损伤”。

### 4.4 普通任务

- 不显示“普通 / 白色调查 / 可反复追踪 / 失败保留”等默认规则。
- 只显示当前决策需要的耗时、需求、对手骰、操作按钮。

### 4.5 线索

- 线索不是任务，应通过“调查 -> 新任务”这类图形关系表达。
- 不重复写“可能发现新任务”。
- 如果线索需要派人调查，应显示耗时、需求和行动按钮。

### 4.6 深度链任务

- 不在一开始显示 `1/4` 这类总阶段数，除非玩家已经知道总长度。
- 默认只显示当前可做的一环。
- 下一环最多以 `下一页：?` 预告。
- 地图上用规整叠层表达“当前入口 + 潜在链”，不要一次摊开整条链。

### 4.7 危险、黑骰、对手骰

- 危险是难度 / 后果标识，不是任务类型。
- 高危任务可以合并进普通任务，用危险标识表达例外。
- 黑骰应有独立图形语言，和危险分开。
- 对手骰必须带文字读法，例如 `对手骰 3`，不要只给一个孤立数字。
- 不使用含义不明的“高压”标签；若要保留，必须改成明确规则词。

### 4.8 操作按钮

- 同一列表中的主要操作按钮应对齐。
- 固定宽度时必须保证文字不换行。
- 按钮文字应是动作：`派遣`、`抢派`、`继续调查`、`进入任务`。
- 当前界面的核心功能按钮必须足够大，并放在玩家视线流程的终点。桌面布局里，单步确认 / 进入下一层 / 开始执行这类最关键动作，默认放在右下角；不要藏在面板右上角、标题旁或小尺寸附属按钮里。
- 核心功能按钮的图标 / 箭头必须明显大于普通标签和辅助图标，能一眼指向“下一步”。次级动作如 `下一天`、`返回`、筛选、预览，只能在视觉重量上低于主按钮。

## 5. 布局与间距规则

### 5.1 删减后必须回收空间

删除信息后，必须同步检查：

- `gap`；
- `padding`；
- `min-height`；
- grid 行列占位；
- 绝对定位元素是否还在撑开正文；
- 操作按钮和需求行是否仍对齐。

不能只确认文字消失。内容少了，空间也必须变紧凑。

### 5.2 功能标识位置一致

- 同一类功能信息必须在同一位置出现。
- 如果某任务没有该信息，槽位可以缺省，但已有信息不能换位置。
- 不要因为某张卡有倒计时，就把普通耗时挤到另一侧。

### 5.3 视觉重量

- 高频普通信息用轻量标签。
- 强时效、强危险、当前操作状态可以使用更高权重组件。
- 不要把普通信息做成像按钮的大块。
- 不要让辅助信息抢过标题、倒计时和操作按钮。

## 6. 地图与区域界面规则

### 6.1 地图语义

- 地图类界面必须有可识别空间语义：世界地图、区域轮廓、经纬网、航线、地形或城市结构。
- 不要用抽象圆斑、彩色光区替代地图。
- 若只是临时占位，也要让玩家看出“这是地图”。
- 地图上的图例 / 标识必须遵守 §3.4：只能解释地图层真实可见的状态。不要把右侧详情面板里的任务分类摘要搬到地图上；如果限时、深度链只在右侧卡片里表达，地图图例就不能出现对应标识。
- 全球 / 上层地图页若承担的是“选择地区 / 进入地区”，就不放 `下一天`、结算、结束探索这类时间推进动作。时间推进应放在玩家正在安排任务的当前地区 / 派遣上下文里；推进后默认回到当前地区继续派遣，而不是回到全球选择层。
- 当前版本若不支持 7 天内换地区，区域页不要提供会让玩家误以为能换区的返回全球 / 换区主入口；后续若支持，必须把换区做成明确系统并说明成本或限制。

### 6.2 标签避让

地图层只要同时存在多种任务点、线索点或可展开入口，就必须有标签避让机制。

硬规则：

- 不靠手工坐标解决长期堆叠。
- 深度链、黑骰、截稿等高优先级入口优先保位。
- 普通任务和线索标签让位。
- 验收必须覆盖默认态、hover 态、展开态，以及宽 / 中 / 窄视口。
- 元素带 `transform` 过渡时，避让计算不能直接依赖动画中的 `getBoundingClientRect()`；应使用基础矩形 + 偏移量，或临时禁用过渡测量。
- 避让标准是“可读安全区”，不是“矩形刚好不相交”。叠层、阴影、描边光晕、hover 扇出范围都要计入保留区，标签之间应留出清晰空隙。
- 已派遣地图点应在任务点右上角显示派遣角色头像组。头像组属于任务点状态的一部分，需要参与视觉检查：不能太小到看不清，也不能遮住倒计时、任务名或邻近地图点。

### 6.3 深度链地图入口

- 地图只显示当前最新任务。
- 未完成第一环时，地图标识显示第一环。
- 完成第一环后，地图标识替换为第二环。
- 不在地图上同时显示整条链。
- 入口默认是规整叠层；hover 轻微错开；点击后可在同一对象中展开成线性链。

## 7. 派遣配置与角色骰 UI

### 7.1 派遣配置页

派遣配置页要让玩家感到“我正在组本次骰池”，不是只在读属性表。

必须优先表达：

- 已选人数，例如 `已选 2/3`；
- 本次骰池预览；
- 角色骰与道具骰进入同一个骰池；
- 与当前任务相关的骰面；
- 需求覆盖、缺口、空面；
- 临时线人 / 一次性道具是否占人数或占骰池。

任务头部必须遵守信息层级：

- 任务名、任务性质、关键需求和耗时可以作为紧凑标题 / meta 出现。
- 耗时是执行成本，不单独占一个 UI 模块；可写入 `耗时 2天` 标签，或体现在 `开始调查（2天后判定）`。
- 探索周剩余天数是全局资源，不放进任务摘要数值组。
- 截止日期是红色截稿 / 突发任务的特殊约束，只在对应任务上以警戒条、倒计时或状态提示出现，不和常规耗时、需求并列。
- 若截止日期是本任务的最关键决策值，优先用右侧倒计时块承载；不要把剩余天数藏在长句里。
- 需求可以在任务头部轻量提示一次；具体覆盖、缺口和风险判断以需求覆盖面板为准。

配置页主操作只能围绕本次派遣：`开始调查`、`开始线索调查`、`返回区域`、`撤销派遣`。全局 `下一天` / `结束探索` 不应作为浮动按钮出现在配置页，否则会压过本页决策，并可能遮挡候选角色。

### 7.2 风险提示

- 整体风险应优先看任务需求覆盖度和缺口。
- 不要用“相关骰面 / 总骰面”比例直接标整体风险。
- 加入低贡献角色不应让整体风险反向升高，除非规则明确规定会带来额外代价。
- 低贡献角色应提示“贡献低 / 无相关面 / 占用人数槽”，而不是让任务本身变危险。
- 如果界面显示“低 / 中 / 高风险”或“达标率”，必须使用与真实结算一致的公式。不要用整套骰面池的潜在覆盖、静态面值总和或候选排序分数替代实际掷骰概率；0% 或几乎不可达的组合必须显示为高风险 / 不可达。
- 角色骰当前口径：任务需求属性决定哪些骰面算相关；普通任务达标看相关骰面有效点是否达到任务目标，而不是每个属性在单次掷骰中逐项硬达标。UI 应同时显示有效点目标和实际达标率，属性条用于解释相关贡献来源。
- 如果真实达标率暴露出适配队伍仍几乎不可能成功，不能只改风险标签或文案。UI 规则要反向提醒设计：检查任务需求、目标公式、骰面密度和支援资源，让高风险任务保持“可搏”，而不是把不可玩状态包装成准确提示。

### 7.3 角色骰表现

- 角色骰 UI 必须让玩家联想到骰子。
- 可参考骰面展开、骰子网格、六面展示等方式，而不是普通资料面板。
- 派遣配置页里的候选角色卡、已选角色槽和角色详情，只要展示完整 6 面角色骰，就应优先使用骰子立方体摊开的“长尾十字架 / dice net”结构；不要退回普通横向 chip 行。横向 chip 行只适合极小摘要、日志或一次性道具简表。
- 角色卡默认只显示专精可以，但与当前任务相关的骰面必须更亮、更大。
- 角色详情应先给当前任务适配结论，例如“相关面 3/6，空面 1”。
- 不相关面降级显示。
- 升级分支默认折叠，避免干扰派遣决策。

## 8. 骰子判定与玩家控制

- 任何强调单颗骰子、锁定、干预、污染的玩法，应控制骰子总数，默认单次判定不超过 10 颗。
- 玩家需要决策时，不自动推进。
- 锁定、重投、确认类动作必须有明确按钮。
- 不用 2 秒自动倒计时替玩家继续，除非用户明确要求快节奏。
- 新机制不能只在结算文案里出现，必须在玩家决策现场被看到、操作到或感知到。

## 9. 实验页规则

实验页不是低标准页面。

### 9.1 是否需要独立实验页

默认不为已有游戏本体 UI 另起“假实验页”。如果方向已经明确，目标是修改现有试玩流程，应优先在本体承载页做最小切片，并用真实数据、真实入口、真实状态验证。

只有满足以下情况之一，才先做独立实验页：

- 方向尚未明确，需要并排比较 2-3 个视觉 / 交互方案；
- 新系统没有现成承载页面，必须先搭一个低成本操作面；
- 直接改本体会污染正式流程，且无法用参数、临时入口或本体 lab 模式隔离；
- 用户明确要求先做静态实验页或视觉草稿。

若只是担心风险，优先在本体增加临时参数、隐藏入口或 lab 模式，不默认另写一套假数据、假交互、假 CSS。独立实验页只能验证构图和局部手感，不能替代本体路径验收。

### 9.2 实验页质量底线

只要实验页复用正式 UI 语义，就必须继承正式原型的基础保护：

- 地图标签避让；
- 碰撞审计；
- 响应式检查；
- 状态截图；
- 操作按钮对齐；
- 文案去冗余；
- 信息槽位一致。

实验页可以粗糙，但不能把已经解决过的问题重新带回来。

### 9.3 动效演示页

当用户要求查看动效、动效语言、动画手感或多个动作方案时，应优先单独制作“系列演示页”，不要只把动效散落在正式流程里让用户反复触发。

动效演示页必须：

- 按动作类型分组展示，例如摇骰、停骰、计入飞入、黑骰介入、锁定 / 重投、结果收束；
- 每组都有可手动重播按钮，必要时提供“重播全部 / 暂停循环”；
- 明确标注这是视觉实验页，尚未套用到完整全链或正式流程；
- 保留窄屏可用性与截图验收，不能因为是动效实验就忽略按钮可见、文字溢出或布局挤压；
- 用户确认某个动效进入主干后，再把选中的动作最小化移植到真实承载页面，不默认把实验页全部动效塞进正式流程。

## 10. 截图与验收规则

可见 UI / 原型改动必须交付真实截图。

### 10.1 截图文件

- 不复用旧截图文件名，避免聊天端缓存旧图。
- 文件名应带日期和改动主题，例如 `2026-05-12-unified-duration-slot.png`。
- 1-3 张截图直接贴对话；超过 3 张整理汇总页。

复杂 UI / 分阶段 UI 重构交付时，优先提供“无标注真实截图 + 程序化标注截图”组合。标注截图必须基于真实页面 DOM 或游戏 UI 元素位置绘制框线、编号与短说明；不要把 AI 生图标注当作验收依据。

### 10.2 截图前检查

交付前逐项检查：

- 是否有标签堆叠；
- 同类功能标识位置是否一致；
- 是否仍有冗余说明；
- 删除内容后空间是否回收；
- 按钮是否对齐且不换行；
- 文字是否溢出、重叠或遮挡；
- 角标、徽章、头像组是否有独立槽位或安全区，是否遮挡标题、按钮文字、倒计时或地图点文字；
- 地图 / 卡片 / 弹窗是否在当前视口正常；
- 若涉及 hover / 展开，是否截图或审计对应状态；
- 是否有缓存旧图风险。

### 10.3 截图数量与对话负载

- 截图较多时，优先生成截图汇总页，并在对话里只贴最关键的 1-3 张。
- 如果当前对话已经因为大量截图、长链路 UI 讨论或历史上下文变得明显繁重，AI 应主动提醒用户：建议新建对话，并给出当前任务状态、关键文件和截图目录，避免继续堆上下文导致未响应或卡死。

## 11. 入口、版本与视觉风格稿

### 11.1 网页入口

- 根目录入口必须能清楚区分当前试玩原型、旧参考页、实验页。
- 按钮文字要说明实际含义，例如“试玩当前原型”“打开符号实验页”，不要让用户误以为某个实验页是正式游戏。
- 如果存在缓存风险，入口链接应带版本参数或说明当前版本。
- 新增实验页后，应考虑是否需要在根入口放临时入口，避免用户翻目录。
- 如果用户要求“在 index 体验到某个刚确认的 UI 改动”，根入口的主按钮必须指向该改动所在页面；不能只把它藏在次级卡片里。
- 如果完整全链原型还没有套用某个实验页 UI，入口必须明确标注“旧壳层 / 次级入口 / 尚未套用”，避免用户误以为改动回退。
- 当用户确认某个实验页 UI 应进入完整版或主干时，必须同步所有实际承载页，不得只改 index 文案。验收至少打开一次 `index -> 完整全链 -> 目标界面` 的真实浏览器路径，并截图确认与当前 UI 规则一致。
- 交付 HTML 实验页、动效演示页或可交互原型时，必须同时提供“浏览器可直接播放 / 操作”的入口，例如 `http://127.0.0.1:端口/...` 或明确可双击打开的 HTML 路径；不能只给仓库源码文件链接。
- 若对话中的文件链接会打开编辑器源码，应明确标注“源码文件”，并另给“预览网页”链接。用户要体验动效时，默认应打开预览网页而不是源码文件。

### 11.2 截图与网页版本一致

- 给用户的截图必须来自当前页面的真实浏览器截图。
- 不复用旧截图文件名。
- 如果用户指出网页和截图不一致，先检查缓存、截图裁切范围、是否打开了旧入口，不要直接解释成用户看错。

### 11.3 美术风格稿与 UI 原型

- 玩法 UI 原型先验证功能呈现和交互，不等同于最终 GUI 包装。
- 正式美术风格稿确定后，可用“玩法原型截图 + 风格稿”生成目标效果图。
- 效果图不能替代可操作原型；必须再拆解回组件、状态、槽位和实现规则。
- 临时风格尝试不要自动写入正式设计文档，除非用户明确要求沉淀。

## 12. 更新规则

后续用户给出 UI / 交互反馈时：

- 如果是“为什么 AI 会这样误判”，写入 `ai-collaboration-guidance.md`。
- 如果是“以后界面具体怎么做”，写入本文。
- 如果两者都有，两边都写，但不要复制同一段：协作指南写偏差原因，本文写执行规则。
- 如果用户要求修改、纠偏、返工，或指出某个 UI 判断不合理，不能只改实现和截图；最终回复前必须检查是否需要同步本文、`ai-collaboration-guidance.md`、`docs/设计采纳记录.md`、相关 GDD 或 `docs/会话交接_当前状态.md`。
- 如果判断本次 UI 反馈不需要新增规则，也必须在最终回复里简短说明“不写入规则文档”的原因，避免把沉淀遗漏伪装成默认处理。
