# UI 与交互执行规范

> **用途**：给后续 AI / 新对话在处理 Angus 的 UI、交互、原型呈现、视觉实验页时使用。
> **定位**：这是执行规则手册，不是错题本，也不是玩法真源。
> **最后更新**：2026-05-12

---

## 1. 文档关系

### 1.1 与 `ai-collaboration-guidance.md` 的分工

- `ai-collaboration-guidance.md` 是大的协作错题本，记录用户偏好、AI 误判原因、宏观工作方式问题。
- 本文是 UI / 交互执行规范，记录做界面时必须遵守的具体规则、槽位、检查项。
- 看似 UI 的反馈，如果本质是范围失控、需求理解偏差、验证方式错误，应继续写入 `ai-collaboration-guidance.md`。
- 可执行的 UI 规则应沉淀到本文。

### 1.2 触发阅读

凡是任务涉及以下任一内容，必须先读本文：

- HTML / Godot 原型界面；
- 任务卡、地图点、派遣配置、角色详情、骰子表现；
- UI 信息层级、交互动效、视觉符号、截图验收；
- 实验页、生图对照、视觉组件词典。

## 2. UI 改动前的工作合约

每次动手前，先在心里或回复中明确三件事：

- **保留**：用户已经认可的结构、位置、交互、视觉重点是什么；
- **修改**：这次只解决哪个具体问题；
- **不做**：哪些相邻问题暂不扩展，避免把局部修正变成整体重构。

默认原则：

- 用户指出“某个元素不明确”时，第一轮只改该元素的文本、图标、提示或状态表达。
- 不要因为一个标签不清楚，就重排整张卡片。
- 不要把用户认可的亮点丢掉，例如右侧大倒计时块、规整深度链叠层、熟悉的 HTML 原型壳层。

## 3. 玩家信息过滤规则

玩家界面只显示会改变当前决策的信息。

### 3.1 应显示

- 耗时、成本、人数限制；
- 需求属性 / 需求骰面；
- 对手骰、黑骰、危险后果；
- 倒计时、截止、即将消失；
- 当前可用操作；
- 当前任务链阶段和下一步可见信息；
- 角色与当前任务的适配结论、相关骰面、明显缺口。

### 3.2 不应显示

- 内部任务类型说明，例如“白色调查”“红色截稿”这类开发分类；
- 默认规则说明，例如普通任务“可反复追踪”“失败保留”；
- 已由另一个 UI 元素表达过的重复说明；
- 含义不明确的标签，例如没有明确玩法后果的“高压”；
- 为了解释系统而写的说明句，除非玩家必须凭它做选择。

### 3.3 一语义一位置

同一语义只保留一处表达。

示例：

- 右侧倒计时已经写“剩余 2天”，就不要再写“过期消失”。
- 线索卡已有“调查 -> 新任务”示意，就不要再写“可能发现新任务”。
- 危险已经有“危险”标签，就不要再放孤立感叹号。

## 4. 任务卡槽位规则

任务卡是当前最容易出错的 UI，必须固定功能槽位。

### 4.1 标准槽位

- 左侧：任务类型图标。
- 标题行：任务名。
- 标题右侧：只放状态、强时效倒计时或特殊入口标识，不放普通耗时。
- 标题下方第一标签：耗时，格式为 `耗时 2天`。
- 同一标签行：对手骰、危险、黑骰等压力信息。
- 需求行：探索 / 生存 / 理性等任务需求。
- 右下角：主要操作按钮。

同一类功能标识必须固定在同一信息槽，不得随任务类型漂移。

### 4.2 耗时

- 统一写成 `耗时 N天`，不要只写 `2天`。
- 同一界面内必须固定位置，当前任务卡规范为“标题下方第一枚标签”。
- 耗时是派遣成本，不是倒计时，不要和剩余时间共用槽位。

### 4.3 截稿 / 突发任务

- 截稿任务的重点是“剩余时间”，应使用更高视觉重量。
- 当前任务卡可使用右侧大块倒计时：`剩余 2天`。
- 地图点可写 `剩2天`，避免被理解成耗时。
- 不额外写“临时”“过期消失”，倒计时本身已经说明会消失。

### 4.4 普通任务

- 不显示“普通 / 白色调查 / 可反复追踪 / 失败保留”等默认规则。
- 只显示当前决策需要的耗时、需求、对手骰、操作按钮。

### 4.5 线索

- 线索不是任务，应通过“调查 -> 新任务”这类图形关系表达。
- 不重复写“可能发现新任务”。
- 如果线索需要派人调查，应显示耗时、需求和行动按钮。

### 4.6 深度链任务

- 不在一开始显示 `1/4` 这类总阶段数，除非玩家已经知道总长度。
- 默认只显示当前可做的一环。
- 下一环最多以 `下一页：?` 预告。
- 地图上用规整叠层表达“当前入口 + 潜在链”，不要一次摊开整条链。

### 4.7 危险、黑骰、对手骰

- 危险是难度 / 后果标识，不是任务类型。
- 高危任务可以合并进普通任务，用危险标识表达例外。
- 黑骰应有独立图形语言，和危险分开。
- 对手骰必须带文字读法，例如 `对手骰 3`，不要只给一个孤立数字。
- 不使用含义不明的“高压”标签；若要保留，必须改成明确规则词。

### 4.8 操作按钮

- 同一列表中的主要操作按钮应对齐。
- 固定宽度时必须保证文字不换行。
- 按钮文字应是动作：`派遣`、`抢派`、`继续调查`、`进入任务`。

## 5. 布局与间距规则

### 5.1 删减后必须回收空间

删除信息后，必须同步检查：

- `gap`；
- `padding`；
- `min-height`；
- grid 行列占位；
- 绝对定位元素是否还在撑开正文；
- 操作按钮和需求行是否仍对齐。

不能只确认文字消失。内容少了，空间也必须变紧凑。

### 5.2 功能标识位置一致

- 同一类功能信息必须在同一位置出现。
- 如果某任务没有该信息，槽位可以缺省，但已有信息不能换位置。
- 不要因为某张卡有倒计时，就把普通耗时挤到另一侧。

### 5.3 视觉重量

- 高频普通信息用轻量标签。
- 强时效、强危险、当前操作状态可以使用更高权重组件。
- 不要把普通信息做成像按钮的大块。
- 不要让辅助信息抢过标题、倒计时和操作按钮。

## 6. 地图与区域界面规则

### 6.1 地图语义

- 地图类界面必须有可识别空间语义：世界地图、区域轮廓、经纬网、航线、地形或城市结构。
- 不要用抽象圆斑、彩色光区替代地图。
- 若只是临时占位，也要让玩家看出“这是地图”。

### 6.2 标签避让

地图层只要同时存在多种任务点、线索点或可展开入口，就必须有标签避让机制。

硬规则：

- 不靠手工坐标解决长期堆叠。
- 深度链、黑骰、截稿等高优先级入口优先保位。
- 普通任务和线索标签让位。
- 验收必须覆盖默认态、hover 态、展开态，以及宽 / 中 / 窄视口。
- 元素带 `transform` 过渡时，避让计算不能直接依赖动画中的 `getBoundingClientRect()`；应使用基础矩形 + 偏移量，或临时禁用过渡测量。

### 6.3 深度链地图入口

- 地图只显示当前最新任务。
- 未完成第一环时，地图标识显示第一环。
- 完成第一环后，地图标识替换为第二环。
- 不在地图上同时显示整条链。
- 入口默认是规整叠层；hover 轻微错开；点击后可在同一对象中展开成线性链。

## 7. 派遣配置与角色骰 UI

### 7.1 派遣配置页

派遣配置页要让玩家感到“我正在组本次骰池”，不是只在读属性表。

必须优先表达：

- 已选人数，例如 `已选 2/3`；
- 本次骰池预览；
- 角色骰与道具骰进入同一个骰池；
- 与当前任务相关的骰面；
- 需求覆盖、缺口、空面；
- 临时线人 / 一次性道具是否占人数或占骰池。

### 7.2 风险提示

- 整体风险应优先看任务需求覆盖度和缺口。
- 不要用“相关骰面 / 总骰面”比例直接标整体风险。
- 加入低贡献角色不应让整体风险反向升高，除非规则明确规定会带来额外代价。
- 低贡献角色应提示“贡献低 / 无相关面 / 占用人数槽”，而不是让任务本身变危险。

### 7.3 角色骰表现

- 角色骰 UI 必须让玩家联想到骰子。
- 可参考骰面展开、骰子网格、六面展示等方式，而不是普通资料面板。
- 角色卡默认只显示专精可以，但与当前任务相关的骰面必须更亮、更大。
- 角色详情应先给当前任务适配结论，例如“相关面 3/6，空面 1”。
- 不相关面降级显示。
- 升级分支默认折叠，避免干扰派遣决策。

---

## 8. 骰子判定与玩家控制

- 任何强调单颗骰子、锁定、干预、污染的玩法，应控制骰子总数，默认单次判定不超过 10 颗。
- 玩家需要决策时，不自动推进。
- 锁定、重投、确认类动作必须有明确按钮。
- 不用 2 秒自动倒计时替玩家继续，除非用户明确要求快节奏。
- 新机制不能只在结算文案里出现，必须在玩家决策现场被看到、操作到或感知到。

## 9. 实验页规则

实验页不是低标准页面。

只要实验页复用正式 UI 语义，就必须继承正式原型的基础保护：

- 地图标签避让；
- 碰撞审计；
- 响应式检查；
- 状态截图；
- 操作按钮对齐；
- 文案去冗余；
- 信息槽位一致。

实验页可以粗糙，但不能把已经解决过的问题重新带回来。

---

## 10. 截图与验收规则

可见 UI / 原型改动必须交付真实截图。

### 10.1 截图文件

- 不复用旧截图文件名，避免聊天端缓存旧图。
- 文件名应带日期和改动主题，例如 `2026-05-12-unified-duration-slot.png`。
- 1-3 张截图直接贴对话；超过 3 张整理汇总页。

### 10.2 截图前检查

交付前逐项检查：

- 是否有标签堆叠；
- 同类功能标识位置是否一致；
- 是否仍有冗余说明；
- 删除内容后空间是否回收；
- 按钮是否对齐且不换行；
- 文字是否溢出、重叠或遮挡；
- 地图 / 卡片 / 弹窗是否在当前视口正常；
- 若涉及 hover / 展开，是否截图或审计对应状态；
- 是否有缓存旧图风险。

## 11. 入口、版本与视觉风格稿

### 11.1 网页入口

- 根目录入口必须能清楚区分当前试玩原型、旧参考页、实验页。
- 按钮文字要说明实际含义，例如“试玩当前原型”“打开符号实验页”，不要让用户误以为某个实验页是正式游戏。
- 如果存在缓存风险，入口链接应带版本参数或说明当前版本。
- 新增实验页后，应考虑是否需要在根入口放临时入口，避免用户翻目录。

### 11.2 截图与网页版本一致

- 给用户的截图必须来自当前页面的真实浏览器截图。
- 不复用旧截图文件名。
- 如果用户指出网页和截图不一致，先检查缓存、截图裁切范围、是否打开了旧入口，不要直接解释成用户看错。

### 11.3 美术风格稿与 UI 原型

- 玩法 UI 原型先验证功能呈现和交互，不等同于最终 GUI 包装。
- 正式美术风格稿确定后，可用“玩法原型截图 + 风格稿”生成目标效果图。
- 效果图不能替代可操作原型；必须再拆解回组件、状态、槽位和实现规则。
- 临时风格尝试不要自动写入正式设计文档，除非用户明确要求沉淀。

## 12. 更新规则

后续用户给出 UI / 交互反馈时：

- 如果是“为什么 AI 会这样误判”，写入 `ai-collaboration-guidance.md`。

- 如果两者都有，两边都写，但不要复制同一段：协作指南写偏差原因，本文写执行规则。
