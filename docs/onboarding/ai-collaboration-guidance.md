# AI 协作指引：如何更精准理解当前用户

> **用途**：给后续接手本仓库的 AI / 新对话提供一份可执行的协作约束与偏差复盘。  
> **对象**：不是给用户看的总结稿，而是给 AI 自己看的工作指引。  
> **维护方式**：每当用户给出新的负反馈、纠偏指令或暴露出新的误判模式，都应补充到本文对应章节。  
> **最后更新**：2026-05-13

---

## 1. 文档定位

这份文档记录的不是“项目功能需求”，而是**当前用户的沟通方式、判断标准、常见雷区，以及 AI 在本项目中最容易犯的偏差**。

目标只有两个：

1. 让后续 AI 在接手时更快抓住用户真正要的东西。  
2. 把已经暴露过的误判模式写成硬规则，避免重复犯错。

如果本文与临时想法冲突，**优先遵守本文**，除非用户在新对话里明确推翻。

涉及 UI、交互、原型呈现、视觉实验页时，还必须同时读取并执行 [`ui-interaction-guidelines.md`](./ui-interaction-guidelines.md)。本文保留“为什么犯错”的复盘，UI 文档提供“具体怎么做”的执行规则。

## 2. 用户的核心偏好画像

### 2.1 用户最看重“玩家是否能明显感知到变化”

用户判断“做出来了没有”，不是看规则是否已经在后台生效，而是看：

- 玩家是否在界面上看得到；
- 玩家是否在操作中感知得到；
- 玩家是否能明确分辨“这次和之前不一样”。

因此：

- **规则存在** 不等于 **玩法成立**；
- **结算文案提到** 不等于 **机制被感知到**；
- **内部状态变化** 不等于 **玩家体验发生变化**。

### 2.2 用户偏好“最小改动验证”，不喜欢顺手改大范围外观或载体

当用户说“完善一下某个玩法”时，默认理解为：

- 优先在**当前熟悉的载体**上验证；
- 优先做**最小行为改动**；
- 不要顺手把 UI、节奏、入口、运行方式一起重做。

尤其是：

- 如果用户原本在看 HTML 原型，**不要擅自切到 Godot 运行时做演示**；
- 如果用户要的是“补骰子玩法”，**不要顺手重做阶段切页、整体布局或产品壳层**。

### 2.3 用户希望“首个可玩区域就能体验到核心机制”

用户非常重视“上手即得”。

如果某套新机制需要：

- 解锁后续区域；
- 玩很久才能遇到；
- 或藏在不容易触发的节点里，

用户通常会认为“实际上没做出来”。

因此：

- 优先把新机制集中到**第一个可玩区域**；
- 优先把 showcase 节点放到**最早可触达的位置**；
- 不要把主要机制分散在后续地图里等玩家慢慢打。

### 2.4 用户要“可思考、可控制”的操作权

用户对“自动替玩家推进”的系统反应会比较强，尤其是：

- 自动重投；
- 自动跳过思考阶段；
- 很短的时间窗口；
- 玩家还没理解就已经进入下一段。

因此：

- 玩家需要决策时，必须有**明确的停顿**；
- 最好有**明确按钮**而不是自动继续；
- 锁定 / 重投 / 确认这类动作，优先给玩家手控权。

### 2.5 用户对界面偏差很敏感

如果用户要求的是玩法补强，而结果把界面视觉体验改得面目全非，用户会认为：

- 做偏了；
- 没有抓住重点；
- 甚至怀疑“你是不是在做另一件事”。

因此：

- 改玩法时默认**不改大 UI 风格**；
- 若必须改展示层，优先局部嵌入，不重做整体布局；
- 如果要换载体或大改交互壳层，必须先明确征得用户同意。

## 3. 本次对话里用户提出过的关键问题与指令

以下不是完整逐字转录，而是对后续最重要的需求轨迹摘要。

### 3.1 关于骰子玩法本身

用户反复要求：

- 先发散，再收束；
- 把可操作维度和机制按层级梳理；
- 新机制不能只停留在“系统算了”，必须让玩家明显体会到；
- 没有直接对抗时，也要做出“像被对抗”的感觉；
- 关卡类型（考古 / 科学调查 / 异常接触等）应有不同机制偏好。

### 3.2 关于当前验证载体

用户一开始接受把玩法落进当前 demo，但在实际看到效果后明确表达：

- 不希望因为做骰子玩法，导致界面观感和熟悉的网页版本差太远；
- 重点是**保留熟悉体验的前提下补玩法**；
- 如果 Godot 包体 UI 太开发态，应优先回到 HTML 原型继续做展示。

### 3.3 关于“玩家能感受到”这件事

用户提出的关键判断标准包括：

- 干预、污染等不能只存在于最后结算文字里；
- 每个小机制应该有明显触发时机；
- 机制之间要有间隔，不要一闪而过；
- 单颗骰子要足够清晰，骰子太多时会让所有机制失效；
- 应有手动锁定某个骰子的操作；
- 锁定后要有明确的“开始重投”按钮，不能自动替玩家继续。

### 3.4 关于内容分布

用户明确要求：

- 把主要新机制优先放在北美禁区；
- 因为那是第一个能玩到的区域；
- 不要把 showcase 主要放在后续区域。

### 3.5 关于后续协作

用户希望形成一份长期文档，记录：

- 本次对话里所有关键问题和指令；
- AI 为什么没做好、没考虑到；
- 用户思考方式和偏好；
- 后续新反馈继续往里追加；
- 并且要让别的对话或 AI 接手时也能受到约束。

## 4. 本次对话里 AI 已暴露的主要偏差

### 偏差 1：把“补玩法”误做成“换载体验证”

#### 表现

用户原本更熟悉 HTML 原型，但 AI 为了“让用户能玩到”，优先把新玩法塞进了 Godot `weekly_run`。

#### 为什么这是问题

这改变了用户熟悉的体验壳层。即使规则本身没错，用户看到的也会先是：

- 界面不一样；
- 节奏不一样；
- 操作感觉不一样；

从而掩盖掉用户本来真正想看的“骰子玩法变化”。

#### 后续硬规则

- 如果用户最初讨论的是 HTML 原型或网页体验，默认先在 **HTML** 验证。  
- 不得以“更容易运行”为理由，擅自把玩法迁到另一种载体演示。  
- 如果必须换载体，必须先明确征得用户同意。

### 偏差 2：把“规则被算进去了”误当成“玩家已经体会到了”

#### 表现

AI 先把：

- 类型包、
- 副轴、
- 干预、
- 余波、
- 链式节点

都写进了逻辑层，但玩家看到时仍然觉得“还是原来的简单作废骰”。

#### 为什么这是问题

因为这些变化主要在：

- 计算层；
- 结果文案；
- 状态写回；

而不是在玩家视野里的阶段演出中。

#### 后续硬规则

- 任何新机制如果只在结算文字或状态里可见，默认视为**未做完**。  
- 规则层完成后，必须立刻追问：  
  - 玩家在哪个时刻第一次能感知到它？  
  - 玩家会误以为它还是老机制吗？  
  - 没看说明能不能区分出来？  

### 偏差 3：低估了“骰子数量过多”对可读性的破坏

#### 表现

早期实现仍然使用大量骰子，导致：

- 单颗骰子存在感过低；
- 被干预、被污染、被保住的具体骰子看不清；
- 锁定决策没有重量。

#### 为什么这是问题

当骰子过多时：

- 任何单颗机制都会被淹没；
- 反馈再多也只是噪声；
- 玩家不会觉得自己在“做选择”，只会觉得系统在“播结果”。

#### 后续硬规则

- 只要玩法想强调单颗骰子、锁定、干预或污染，优先先压缩总骰数。  
- 在没有强理由前，把单次判定总骰数控制在 **10 个以内**。  
- 若强调“角色骰”，优先按角色出骰，而不是按属性点数直接堆骰。

### 偏差 4：过早自动推进，剥夺了用户思考窗口

#### 表现

锁定阶段一开始只有 2 秒自动重投。

#### 为什么这是问题

用户明确重视：

- 可思考；
- 可慢慢看；
- 可手动确认；

自动推进会直接破坏这类体验。

#### 后续硬规则

- 玩家需要决策时，默认不要自动继续。  
- 锁定 / 重投 / 确认类交互，优先提供明确按钮。  
- 如果有时间限制，必须先得到用户明确认可。

### 偏差 5：没有足够早地把“核心机制集中到首个区域”当作硬要求

#### 表现

早期把一些 showcase 放到了东亚或更后面区域。

#### 为什么这是问题

用户想要的是：

- 马上打开就能试；
- 第一张图就能覆盖主要机制；

否则就会被判断为“实际上玩不到”。

#### 后续硬规则

- 新机制的第一轮验证，默认优先集中到**首个可玩区域**。  
- 若需要 showcase 节点，也优先放在首区。  
- 后续区域可以留第二版或补充版机制，但不能承载第一轮验证的主要内容。

### 偏差 6：补玩法时顺手改了太多展示层

#### 表现

在 Godot 阶段，AI 一度同时改了：

- 规则层；
- 任务面板文案；
- 阶段显隐；
- 禁用态样式；
- 整体流程切页表现。

#### 为什么这是问题

这会让用户无法分辨：

- 是玩法变了；
- 还是整个界面被重做了。

#### 后续硬规则

- 如果用户要求是“补玩法”，默认展示层改动只做**局部嵌入式增强**。  
- 不主动做全局 UI 重组、阶段切页或视觉风格迁移。  
- 一旦发现实现开始往“另一个任务”扩张，必须停下来收束。

## 5. 当前用户的思考方式与决策风格

### 5.1 用户习惯先发散，再收束

当用户谈新系统时，通常会先：

- 大量发散机制方向；
- 看可能性空间；
- 再要求压成分层、路线、范围。

因此 AI 在前半段可以发散，但一旦用户开始要求：

- 总结、
- 梳理、
- 正式文档、
- 组合建议，

就必须从脑暴切到收束模式。

### 5.2 用户偏好“先做出可玩的最小体验，再慢慢加层”

用户不反对复杂机制，但很明确地希望：

- 先有一个可玩的第一层；
- 再看是否需要更复杂的层；
- 不要一上来把所有层都做进同一套演示里。

### 5.3 用户非常看重“感知到”

用户并不满足于：

- 文档写明了；
- 代码里算到了；
- 结果页写出来了。

用户真正要的是：

- 玩的时候能看出来；
- 能操作到；
- 能记住发生了什么；
- 能感觉到“这次和以前不同”。

### 5.4 用户会持续纠偏，并希望 AI 能把纠偏转成永久规则

用户不是只想要一次性修正，而是希望 AI：

- 把反馈抽象成规律；
- 写进文档；
- 让以后别的 AI 也遵守；
- 避免每次都重新教一遍。

因此，收到用户强反馈后，AI 应主动做：

1. 纠正当前实现；  
2. 把问题上升为规则；  
3. 把复盘写成未来可执行的判断流程，而不是只写“这次错了”；
4. 更新长期协作文档。

复盘和总结必须能被未来 AI 迁移使用。写入文档时不要只记录具体对象，例如“骰子文字歪了”；应按以下结构抽象：

- **用户看到的症状**：截图第一眼被读成什么、哪里违背用户目标；
- **AI 错误判断口径**：是把实现意图当感知结果、把局部进步当整体通过、只看新问题不查旧优点，还是用远景替代局部检查；
- **可观察门槛**：下次肉眼要检查哪些具体信号，哪些信号一出现就判定失败；
- **可迁移规则**：同类问题在按钮、地图点、模型、贴图、动效、信息层级中对应什么检查；
- **交付约束**：最终回复只能陈述截图已经支持的结论，不能把“比上一轮好”说成“已通过”。

典型反例：不要写“3D 骰子以后要做得更好”。正确写法是“声称 3D / 实体 / 贴面时，必须分别检查闭合体积、不透明面、内容是否属于材质、遮挡噪音和旧版优点回归；任何一项截图未通过，都不能用技术实现解释为通过”。

## 6. 后续协作硬规则（必须执行）

以下规则是从本次对话里提炼出来的**长期约束**。

### 6.1 关于载体选择

- 默认优先在用户当前讨论和熟悉的载体上验证。  
- 不擅自从 HTML 切到 Godot，或从 Godot 切到 HTML。  
- 如果要换载体，先明确说明原因并征得用户同意。

### 6.2 关于展示与感知

- 新机制必须明确回答：  
  - 玩家在哪个时刻第一次感知到它？  
  - 玩家在哪个界面理解到它？  
  - 它是否只存在于结算文案？  
- 若答案是“只在结算文案里”，则视为未完成。

### 6.3 关于范围控制

- “补玩法”不等于“顺手重做 UI”。  
- 默认只做与用户明确目标直接相关的最小改动。  
- 一旦实现开始扩张到：
  - 全局 UI、
  - 阶段切页、
  - 入口重构、
  - 运行时迁移，
  就必须停下来重新确认。

### 6.4 关于首区试玩

- 第一轮机制验证优先集中在**首个可玩区域**。  
- showcase 节点优先也放在首区。  
- 如果用户还没玩到，就不要把关键体验放到后面区域。

### 6.5 关于玩家决策权

- 只要玩家有“锁定 / 重投 / 确认”决策，默认给明确按钮。  
- 不自动 2 秒倒计时，不替玩家自动推进。  
- 如果用户没明确要求快节奏，就优先让玩家“慢慢看、慢慢选”。

### 6.6 关于骰子数量

- 需要强调单颗骰子的玩法时，先压缩总骰数。  
- 默认目标：单次判定总骰数不超过 10。  
- 角色基础骰优先于属性点直接堆骰。

### 6.7 关于反馈循环

- 用户一旦指出“体会不到”“还是原来的”“太快了”“界面变了”，必须优先判断这是：
  - 感知层问题，
  - 节奏问题，
  - 载体问题，
  - 还是范围失控问题。  
- 不要立刻再加新规则，先处理感知与节奏。

## 7. 后续追加规则

从现在开始，每次用户给出新的强反馈，都应按以下模板补一条：

### 新增反馈模板

- **时间**：YYYY-MM-DD  
- **用户反馈**：用户原话的简要概括  
- **暴露问题类型**：载体 / 感知 / 范围 / 节奏 / 可读性 / 其他  
- **AI 当时偏差**：一句话说明  
- **新增硬规则**：以后必须怎么做  
- **影响范围**：HTML / Godot / 文档 / 所有会话

### 2026-04-28：从采纳记录转方案前必须先复述用户评注意图

- **用户反馈**：用户指出「没有按照我对每条的评注想法来推进具体方案」，要求先理解评注和想法，再进行方案撰写。
- **暴露问题类型**：文档 / 范围 / 其他。
- **AI 当时偏差**：把 `docs/设计采纳记录.md` 中的条目机械扩写成字段与实现清单，忽略了用户评注里的适用范围、层级区分、体验意图和不泛用约束。
- **新增硬规则**：当用户要求把采纳条目转成方案时，AI 必须先按条目复述「用户已采纳的核心意图 / 限定范围 / 不该做什么 / 玩家应感知到什么」，确认理解后再写落地方案；不得直接跳到规格文档或代码字段。
- **影响范围**：文档 / 所有会话。

### 2026-05-09：地图类界面不能用抽象圆斑替代空间语义

- **用户反馈**：用户指出区域任务图“显示效果太差”，希望背景替换为世界地图背景。
- **暴露问题类型**：感知 / UI / 可读性。
- **AI 当时偏差**：为了快速表达任务点分布，用抽象彩色圆斑充当地图背景，导致玩家无法把任务点和世界空间建立联系。
- **新增硬规则**：地图、区域、节点总览等承载空间关系的界面，必须优先使用可识别的地图语义（世界地图、区域轮廓、航线/经纬网、地形或城市结构），不要只用抽象光斑、圆区或纯装饰背景代替。若只是临时占位，也要明确让玩家看出“这是地图”。
- **影响范围**：HTML / Godot / UI / 所有会话。

### 2026-05-09：深度连续任务地图入口应表达“当前入口 + 潜在链”，不要一次摊开整条链

- **用户反馈**：用户指出深度连续任务在地图上的表现很差，不应在地图界面完整展示所有链路节点；地图应只显示当前最新任务，用叠层卡片等方式暗示“多个任务合一”，点击后再展开任务链，而且后续任务应随完成逐步显现。
- **暴露问题类型**：感知 / UI / 认知门槛 / 节奏。

- **AI 当时偏差**：把“找更明确的剩余时间表达”误扩展成“重新组织任务卡信息架构”，丢掉了用户认可的右侧大倒计时块和原有卡片层级。
- **新增硬规则**：当用户指出某个 UI 元素“语义不明确”时，第一轮修正应只改该元素的表达方式、标签文本、图标或 tooltip；除非用户明确要求重排，否则不要改变已被认可的空间结构。改前要先识别用户真正想保留的优点。
- **影响范围**：HTML / Godot / UI / 所有会话。

### 2026-05-12：玩家界面只显示会改变决策的信息，不展示默认规则说明

- **用户反馈**：用户指出任务卡仍有大量冗余信息，例如“可能发现新任务”和调查转新任务示意重复，“白色调查 / 可反复追踪 / 失败保留”属于设计层内容，不必给玩家看；玩家只需要知道哪些任务是例外，例如突发任务会消失；“高压”等组件含义不明。
- **暴露问题类型**：UI / 信息架构 / 冗余信息 / 玩家视角。
- **AI 当时偏差**：把内部任务类型、默认规则和例外规则都平铺到卡片上，误以为“说明更全”就是“更清楚”，导致玩家需要过滤大量不影响当前选择的信息。
- **新增硬规则**：任务卡面向玩家时，只显示会改变当前决策的信息：耗时、需求、对手骰、可用操作、倒计时、危险/黑骰等明确例外。默认规则不显示，例如“普通任务失败保留、可反复追踪”不写；同一含义只保留一处表达，例如倒计时已经写“剩余 2天”时，不再写“过期消失”。含义不明确或没有直接玩法后果的标签（如“高压”）必须删除或改成明确规则词。
- **影响范围**：HTML / Godot / UI / 任务卡 / 所有会话。

### 2026-05-12：实验页也必须继承正式原型已验证的可读性保护

- **用户反馈**：用户指出视觉实验页里仍然出现地图标签堆叠，质疑之前要求处理的堆叠机制为什么没有生效。
- **暴露问题类型**：UI / 实验页 / 验收遗漏 / 可读性。
- **AI 当时偏差**：把“正式全链原型已修复标签避让”误当成全项目已解决，却新建了一个静态实验页，仍用手写绝对坐标，没有同步接入避让机制；第一次补救又被 CSS transform 过渡影响，候选位置测量一直读到动画起点。
- **新增硬规则**：只要实验页复用地图、任务卡、可展开入口等正式 UI 语义，就必须继承对应的基础保护机制：标签避让、碰撞审计、状态截图、响应式检查。不能因为页面是临时实验而退回手工坐标。做避让时若元素带 transform 过渡，候选位置应使用“基础矩形 + 偏移量”计算，或临时禁用过渡，不能直接读动画中的 `getBoundingClientRect()`。
- **影响范围**：HTML / UI / 实验页 / 地图任务呈现 / 所有会话。

### 2026-05-12：同一类功能标识必须固定在同一信息槽

- **用户反馈**：用户指出“耗时”有的任务显示在右边，有的显示在左边，同类功能性标识的位置必须统一。
- **暴露问题类型**：UI / 信息架构 / 可读性 / 一致性。
- **AI 当时偏差**：为了迁就不同任务卡的局部布局，把同一类信息放在不同槽位：普通/深度任务耗时在标题右侧，线索/突发/黑骰耗时在正文标签区，导致玩家需要逐卡寻找信息。
- **新增硬规则**：任务卡中的同类功能标识必须固定槽位，不得随任务类型漂移。耗时统一放在标题下方第一枚标签；需求标签跟在耗时后；对手骰/危险/黑骰等压力信息放在同一标签行；右侧保留给状态、倒计时或主要操作，不与耗时混用。若某任务没有该信息，槽位可以缺省，但已有信息不能换位置。
- **影响范围**：HTML / Godot / UI / 任务卡 / 所有会话。

### 2026-05-12：删减信息后必须回收空间并重测间距

- **用户反馈**：用户指出删除冗余信息后，红色任务卡仍保留过大的空隙，导致这一项比其他任务松散。
- **暴露问题类型**：UI / 布局 / 可读性 / 后续整理。
- **AI 当时偏差**：只删除了文字和标签，没有同步调整原先为这些内容预留的网格行高、占位和间距；右侧倒计时仍参与正文布局，撑大了卡片内部空白。
- **新增硬规则**：任何 UI 信息删减后，必须同步检查并回收布局空间：行高、gap、padding、min-height、grid/slot 占位、绝对/流式布局关系都要复核。可视组件若只是状态角标或倒计时角标，应避免参与正文流导致空行。交付前截图必须确认“内容少了，空间也变紧凑”，不能只确认文字消失。
- **影响范围**：HTML / Godot / UI / 任务卡 / 所有会话。

### 2026-05-12：index 可体验到某改动，主入口必须指向该改动

- **用户反馈**：用户指出从 index 进入后又看到旧任务探索界面，质疑为什么界面变回老版本。
- **暴露问题类型**：入口语义 / UI 验收 / 需求理解。
- **AI 当时偏差**：把“让我能在 index 网页版体验到”理解成“在 index 下方增加一个实验页链接”，但主按钮仍然指向完整全链原型；完整全链原型还没套用任务卡符号实验页，因此用户按最显眼入口进入时看到的就是旧壳层。
- **新增硬规则**：当用户要求某个已确认 UI 改动能从 index 体验到，index 的主入口必须指向该改动所在页面；若完整全链仍是旧壳层，必须降级为次级入口并明确标注“旧壳层 / 尚未套用当前 UI”。验收截图也要覆盖用户最可能点击的主入口路径。
- **影响范围**：HTML / index 入口 / 原型验收 / 所有会话。

### 2026-05-13：实验页通过不等于完整版已同步

- **用户反馈**：用户指出“完整版打开后，不是最新版内容”，质疑探索任务界面改动是否没有同步。
- **暴露问题类型**：入口语义 / 原型同步 / UI 验收 / 交付范围。
- **AI 当时偏差**：把“任务探索实验页已经通过验收”和“完整全链原型已经套用同一套任务探索 UI”混在一起；虽然 index 文案提示了旧壳层，但用户期望完整版也使用当前任务 UI 时，AI 没有主动完成同步，导致同一入口体系下出现两个版本。
- **新增硬规则**：当一个 UI 实验页被用户确认“没有大问题”并要求并入主干或从 index 体验时，必须明确检查所有用户会点击的承载页：实验页、index 主入口、完整全链入口、相关子流程入口。若完整版仍未同步，只能临时标注；一旦用户要求“完整版也是最新版”，必须同步实现并用真实浏览器截图验证完整路径，不能只改入口文案。
- **影响范围**：HTML / index 入口 / 完整全链原型 / UI 验收 / 所有会话。

### 2026-05-13：已有本体承载时，不默认先做假实验页

- **用户反馈**：用户要求对比“先做实验页再落地本体”的耗时和 token，并采纳“后续不默认先做假的独立实验页”的方案。
- **暴露问题类型**：验证路径 / 成本控制 / UI 真实约束 / 原型同步。
- **AI 当时偏差**：为了快速验证构图，先做了独立静态实验页；但该页面使用假数据和简化交互，不能暴露本体里的真实约束，例如主入口路径、真实状态、面板滚动、hover/click 重渲染稳定性、地图图例与右侧详情的层级关系。后续落地本体时仍需再实现和再验收，形成重复工作。
- **新增硬规则**：当已有游戏本体或完整试玩承载页，且 UI 方向已经明确时，默认直接在本体做最小切片验证；不另起“假实验页”。只有在需要比较多方案、新系统无承载页、必须隔离风险且无法用参数 / lab 模式解决，或用户明确要求时，才创建独立实验页。本体截图与真实路径验收优先级高于静态实验页截图。
- **影响范围**：HTML / Godot / UI 原型 / 实验页 / 验收 / 所有会话。

### 2026-05-13：界面标识必须按所在层级显示

- **用户反馈**：用户指出地图图例问题不只是“地图上多了两个标签”，而是暴露了各个界面如何显示标识的通用规则问题。
- **暴露问题类型**：UI 信息架构 / 标识语义 / 跨界面混用 / 验收遗漏。
- **AI 当时偏差**：把“限时机会 / 深度链”当作全局可展示的摘要标签，却没有检查它们是否在世界地图层有对应可见对象或视觉编码。实际这些信息只在右侧选中区域详情里成立，放到地图层就变成了跨层借语义。
- **新增硬规则**：任何图标、标签、徽章、图例、颜色摘要和数字角标，都必须绑定本界面真实可见的对象、状态和决策用途。不能把详情面板、下一层页面或内部数据里的摘要搬到当前界面当图例；如果标识没有对应可见编码、不能帮助当前界面决策，默认删除或移动到正确层级。具体执行见 `docs/onboarding/ui-interaction-guidelines.md` §4.1。
- **影响范围**：HTML / Godot / UI / 地图 / 任务卡 / 详情面板 / 派遣配置 / 所有会话。

### 2026-05-13：复杂 UI 改动要能直观看出“改了哪里”

- **用户反馈**：用户确认“旧版 / 新版 / 新版程序化标注截图”的试样效果可以，要求加入后续规则。
- **暴露问题类型**：UI 交付 / 截图说明 / 分阶段重构 / 验收可读性。
- **AI 当时偏差**：复杂页面只给真实截图时，用户仍需要自己逐块找差异；尤其是分阶段重构、同一页面多区域同时变动时，单纯截图不足以说明本次改动边界。
- **新增硬规则**：复杂 UI、分阶段 UI 重构或改动点不容易肉眼定位时，交付应优先提供“无标注真实截图 + 程序化标注截图”。标注必须基于真实页面 DOM / UI 元素位置绘制框线、编号与短说明；AI 生图标注可用于汇报海报，但不能替代验收。具体执行见 `docs/onboarding/功能改动截图指引.md` §3.4。
- **影响范围**：HTML / Godot / UI 原型 / 截图交付 / 所有会话。

### 2026-05-13：截图和上下文过重时，主动建议新建对话

- **用户反馈**：用户指出截图过多会导致当前对话经常未响应或卡死，要求如果判断对话已经过度繁冗，可以直接提醒新建对话。
- **暴露问题类型**：协作节奏 / 上下文管理 / 截图交付 / 响应稳定性。
- **AI 当时偏差**：为了保持连续性，容易在同一对话里持续追加大量截图、长说明和多轮验证，导致用户端可读性与响应稳定性变差。
- **新增硬规则**：当当前对话已经积累大量截图、长链路 UI 讨论、反复截图重跑或明显变慢时，AI 应主动提醒用户建议新建对话，并给出最小交接信息：当前目标、已改文件、截图目录、验证命令和下一步。超过 3 张截图优先用汇总页，当前对话只贴关键图。
- **影响范围**：所有会话 / UI 截图 / 长任务交接 / 协作流程。

### 2026-05-13：不默认做竖屏手机截图

- **用户反馈**：用户指出当前阶段暂时不用考虑竖屏手机端使用，后续不应额外制作这类截图，以节省时间和 token。
- **暴露问题类型**：验证范围 / 成本控制 / 截图交付。
- **AI 当时偏差**：把“覆盖窄屏验收”当成默认勤奋项，没有先判断该视口是否属于当前产品目标或用户关心的问题，导致额外截图、额外汇总和额外上下文负担。
- **新增硬规则**：常规 UI / 原型交付默认覆盖主要桌面或实际目标视口；不主动追加竖屏手机 / 极窄长页截图。只有用户明确要求、任务本身涉及移动端 / 窄屏适配，或小视口是问题复现场景时才补。
- **影响范围**：HTML / Godot / UI 截图 / 验收成本 / 所有会话。

### 2026-05-13：信息去重后不能弱化关键数值

- **用户反馈**：用户指出派遣页里“红色截稿”标签和红色提示条重复，但上一步改法把最重要的“剩余 3天”藏进了低权重说明句，反而不如上一层区域任务卡明显。
- **暴露问题类型**：UI 信息架构 / 视觉重量 / 关键数值呈现 / 复盘沉淀。
- **AI 当时偏差**：把“去重”理解为删除重复文案和合并说明，没有重新指定哪个元素承载最关键的信息；修掉“信息层级并列”后，只检查了不再重复，却没有检查玩家第一眼是否更容易看到核心数值。此外，前一轮只把规则写进 UI 执行文档和设计采纳记录，漏写错题本，说明我只记录了“怎么做”，没有记录“为什么会犯错”。
- **新增硬规则**：UI 去重后必须重排视觉主语：关键数值要有唯一且高权重的承载，不得被降级为说明句的一部分。遇到用户明确指出 AI 误判或反复纠偏时，必须同时检查三类落盘：执行规则写入 `ui-interaction-guidelines.md`，偏差原因写入本文，若属于用户明确采纳 / 修订的设计意见则写入 `docs/设计采纳记录.md`。
- **影响范围**：HTML / Godot / UI / 派遣配置 / 任务卡 / 文档沉淀 / 所有会话。

### 2026-05-13：用户要求修改 / 纠偏后必须做沉淀检查

- **用户反馈**：用户明确要求以后避免“让我修改的内容，你没有反思、没有总结加入错题本和对应规则文档”。
- **暴露问题类型**：协作流程 / 返工复盘 / 规则沉淀 / 设计文档同步。
- **AI 当时偏差**：容易把用户的修改要求只理解成“把界面或代码改到位”，把截图验证和实现结果当成收尾；只有在用户追问时才补反思和规则文档。这说明我没有把“用户要求返工本身暴露了 AI 判断失误”当成必须沉淀的信号，也没有在最终回复前固定执行文档同步检查。
- **新增硬规则**：用户要求修改、纠偏、返工，或指出“这里不合理 / 你为什么这样做 / 后续要避免”时，最终回复前必须做四步沉淀检查：一，是否暴露 AI 误判原因，若是写入本文；二，是否能抽象成可执行规则，若是写入对应规则文档，如 `ui-interaction-guidelines.md`、截图指引或相关 GDD；三，是否属于用户明确采纳 / 修订 / 撤回的设计意见，若是写入 `docs/设计采纳记录.md`；四，是否影响接手状态，若是写入 `docs/会话交接_当前状态.md`。若判断某项不需要更新，最终回复必须简短说明原因。
- **影响范围**：所有会话 / UI / 原型 / 玩法文档 / 设计采纳 / 交接文档。

### 2026-05-13：已派遣状态不能混进类型标签或小角标

- **用户反馈**：用户指出区域地图中已派遣小角标太小、不明显，可以直接在地图点右上角显示派遣人的小头像；右侧任务卡的“已派遣”不应放在标识任务类型的标签行，而应放在任务名称右边。
- **暴露问题类型**：UI 状态表达 / 对象占用反馈 / 标签语义混淆 / 视觉优先级。
- **AI 当时偏差**：把“已派遣”当成一个可以随手塞进 chip 列表的普通标识，只表达了布尔状态，却没有表达“谁占用了这个任务”；同时用很小的“派”字角标承担地图上的主要状态反馈，既不明显，也没有利用角色头像这种更具体、更可识别的对象信息。根本问题是没有先区分“任务类型 / 任务压力 / 当前状态 / 占用对象”四类信息。
- **新增硬规则**：任务当前状态不能混入任务类型或机制标签行。`已派遣`、`已完成`、`待结算`、`占用中`等状态应靠近任务标题、主操作或被状态化的对象；若状态涉及人员占用，优先显示人员头像或姓名摘要。地图点上的已派遣状态不得只靠微型文字角标承担，至少要用足够可见的头像组或等价高识别对象来表达。
- **影响范围**：HTML / Godot / 区域地图 / 任务卡 / 派遣队列 / UI 标签体系。

### 2026-05-13：风险预览必须使用真实结算公式

- **用户反馈**：用户试玩后指出“没有一次达标的，即使是风险低的”，怀疑派遣结算有问题。

- **暴露问题类型**：上下文漂移 / 术语复用误判 / UI 对象识别错误 / 纠偏后收回错误落地。
- **AI 当时偏差**：我被“3 到 5 人”和近期派遣页人数上限问题带偏，没有先回到最近一轮明确对象“骰子判定弹窗”，也没有在动手前核对目标文件。根本错误不是少改了一个布局，而是把用户正在讨论的交互对象替换成了另一个相邻系统。
- **新增硬规则**：当用户连续讨论多个相邻 UI，且追问“人数 / 槽位 / 卡片 / 页面布局”这类跨系统词时，必须先确认它属于当前轮最新对象：骰子判定里的“人”是参判角色 / 可计入骰面，派遣里的“人”才是队伍人数。若对象已经由上一条消息明确，就直接按该对象落地；若仍不确定，先用一句话澄清，不能凭近邻记忆把改动落到别的页面。
- **影响范围**：HTML / Godot / 骰子判定 / 派遣配置 / 任务列表 / 所有相邻系统间共用“人数 / 槽位 / 卡片”术语的 UI。

### 2026-05-19：摇骰表现要先像实体骰子

- **用户反馈**：用户指出当前摇骰子的表现效果很一般，没有体现到骰子实体摇的感觉。
- **暴露问题类型**：UI 动效 / 玩法感知 / 骰子主舞台 / 截图验收。
- **AI 当时偏差**：容易把“有动画、有数值变化、有逐颗揭示”误判为“摇骰表现成立”，但玩家期待的是实体骰子在盘面里运动、碰撞、落稳的感受。只让文字或骰面标签原地闪烁，本质上更像抽签 / 翻牌，而不是投骰。
- **新增硬规则**：涉及投骰、锁定、重投、黑骰介入时，骰子区必须是判定弹窗的视觉主舞台；摘要、解释和按钮不能把骰盘挤到下沿。摇骰动效至少包含实体骰盘、骰子本体、位移、旋转、落地反弹、阴影或轨迹中的多项线索。交付截图要覆盖摇骰中、停骰选择、锁定 / 重投后至少一种状态。
- **二次修订（2026-05-19）**：用户进一步指出直接落地到游戏后的表现非常奇怪，要求先不要落地到游戏，而是先做独立实验页单独演示各类骰子动画。后续动效未成熟时不能硬塞进完整全链；应先撤出本体或保持隔离，在动效系列实验页中比较动作语言，用户确认后再最小化迁移。
- **三次修订（2026-05-19）**：用户指出单面薄片翻转仍然很奇怪，并追问是否需要 3D 模型。后续判断投骰动效时，应先用程序化 3D 立方体验证体积、多面、接触阴影、多轴旋转、弹跳和停稳；不要把“薄片翻面 + 标签文字”当作骰子实体感，也不要在动作语言未确认前投入正式建模资产。
- **四次修订（2026-05-21）**：用户指出 CSS 3D 立方体仍然看起来像薄片，要求考虑建模方式。后续若 CSS 3D 被读成薄片，应立即转向真实几何体模型路线，例如 Three.js 程序化模型或 Godot / Blender 资产；不要继续在 CSS 透视和贴片上消耗时间。实验阶段可先用 Three.js 本地 vendor 和程序化 `BoxGeometry` 骰子验证，正式美术资产再等方向确认后制作。
- **五次修订（2026-05-21）**：用户反馈 Three.js 实验页在他的打开方式下显示不出来，并进一步明确“不改变原有判定界面布局，只是改成顶视角看骰子投入”。后续动效实验必须按用户实际打开方式验收，不能只在本地 HTTP / 脚本截图环境通过；如果要交付可直接打开的 HTML，应避免被 `file://` 拦截的 ES module。动效实验也必须嵌回原界面布局验证，只替换骰子盘内部表现，不把局部动效扩成全新大舞台。
- **六次修订（2026-05-21）**：用户指出当前所谓“立方体”仍明显只是几张面片，面模型没有建完。AI 的问题不是实现细节，而是视觉验收口径错误：把“有多个面形状、会动、有投影”误判为“实体骰成立”，没有裁切关键对象、放大检查闭合体积，也没有用“玩家会不会读成薄片 / 散面片”做反向验证。后续凡是声称立方体、实体骰或建模骰，必须先验证截图中有闭合体积：相邻面共享边角、侧面有厚度、面之间无漂浮缝隙、遮挡和投影一致。非空像素、动画播放、脚本断言和实现意图都不能替代这条视觉判断。
- **七次修订（2026-05-21）**：用户强调这不只是立方体问题，而是所有视觉验收都必须举一反三。后续处理任何视觉需求时，AI 不能直接按模糊印象实现；必须先把用户的视觉名词和禁止项拆成可观察条件，再实现并截图逐项核对。通用流程是：提取需求词 -> 写出成立条件 -> 写出禁止误读 -> 整页截图 -> 局部裁切放大 -> 多状态检查 -> 旧错回归。只要截图读法和用户目标不一致，就视为未通过，不得用实现意图或技术细节辩解。
- **八次修订（2026-05-21）**：用户指出我新改的 Canvas 版本虽然比面片更闭合，但骰面文字已经明显错乱；旧版截图的优点是“每个面上的文字贴合立方体”，我却只盯着闭合体积，没有回归检查旧版做得好的面内容贴合。后续视觉返工必须先列出“旧版必须保留的优点”和“本轮只修的问题”，不能为了修一个缺陷破坏另一个已成立的视觉条件。对骰子而言，实体感不只来自外轮廓闭合，还来自骰点、文字、贴图、光照、阴影都服从同一个面透视；如果面内容像浮在屏幕上、旋转方向和骰面不一致、压过边线或脱离四边形平面，即使盒体闭合，也必须判定为未通过。
- **九次修订（2026-05-21）**：用户进一步指出当前最大问题是骰子各面像透明的，立方体结构线本身干扰每个面文字阅读；旧版优点是各面更像不透明骰面，文字位于实心面内。后续不能为了证明闭合结构保留过重线框、背面边或内部调试线。实体骰默认应是不透明或高遮蔽度面材：正面文字 / 骰点前方不能有背面线穿过，边线只服务外轮廓和必要转角，结构验证应在实现里完成，不能暴露成读图噪音。透明判断必须基于截图：如果背景网格、轨迹线、隐藏边或另一面的边线在骰面内部仍可辨认，或骰子主要由轮廓线成立而不是由实心面块成立，即使代码已填充多边形，也算验收失败。
- **十次修订（2026-05-21）**：用户指出当前文字看起来不是建模时存在于面上，而是临时生成的屏幕层，因此会有对位不准、相对骰面歪斜和偶发不显示。后续实体骰的文字、骰点、符号应优先作为面材质 / 贴图生成，再随面一起投影或渲染；Canvas 原型至少要做离屏骰面贴图并映射到投影四边形，不能把屏幕坐标文字当作骰面内容。
- **十一次修订（2026-05-21 · 3D 化复盘口径）**：这轮连续出错的根因不是某个 Canvas 技巧，而是 AI 多次用实现侧进展替代视觉侧成立：把“画了多个面”当成立方体，把“修好了闭合”当成整体验收通过，把“代码填充了面”当成不透明，把“屏幕文字位置接近”当成贴图内容。后续做任何视觉 / 模型 / 动效返工时，必须把对象拆成独立可否决条件逐项验收；每次修 A 都要回归旧版已经成立的 B；截图结论必须来自逐对象、逐子项、局部放大的肉眼检查，而不是来自“比上一版好”的趋势判断。复盘文档也必须写出可迁移规则，不能只记录某个具体 bug。
- **影响范围**：HTML / Godot / 骰子判定 / 黑骰实验 / UI 动效 / 截图交付。

### 2026-05-13：动效演示必须有独立系列页和可直接播放的浏览器入口

- **用户反馈**：用户指出点击动效实验页链接时打开的是 HTML 源码，不是可直接体验的演示网页，并要求写入两条规则：动效演示单独出系列演示页；动效演示页必须给浏览器可以直接播放的版本。
- **暴露问题类型**：UI / 动效 / 实验页 / 入口语义 / 交付方式。
- **AI 当时偏差**：把仓库文件链接当作实验页交付入口，导致用户点击后进入源码视图；同时没有把“动效需要集中分类演示”固化成实验页交付规则。
- **新增硬规则**：当用户要看动效、动画手感或动作语言时，AI 应优先制作独立系列演示页，按动作类型分类并提供重播控制。交付时必须给出浏览器可直接播放 / 操作的预览入口，例如本机 HTTP 链接；源码文件链接只能作为“源码”补充，不能替代演示入口。具体执行见 `docs/onboarding/ui-interaction-guidelines.md` §10.4 与 §11.4。
- **影响范围**：HTML / UI / 动效 / 实验页 / index 入口 / 所有会话。

## 7A. 设计采纳沉淀机制（长期固定机制）

> 自 2026-04-27 起，本机制纳入长期协作约束，所有后续 AI / 新对话都必须遵守。

### 7A.1 沉淀文档

- **文件位置**：[`docs/设计采纳记录.md`](../设计采纳记录.md)
- **文档定位**：用户在讨论中**已采纳 / 待定 / 进阶 / 合并 / 撤回** 的设计意见的长期索引；不是讨论原文，也不是规格文档，而是**决策留痕**。

### 7A.2 何时写入（触发条件）

只要用户在讨论中出现以下任一**明确表态**，AI 在当次回复结束前必须把对应条目追加进该文档：

- 「这个采纳」「加入 …」「作为 …」「可以考虑作为 …」「合并到 …」；
- 「先留着」「待定」「回头再想」「加入待定区」；
- 「作为后期进阶」「作为某类型设计」「作为某角色特性」；
- 「撤回」「不要这条了」「改成 …」（撤回/修订路径，见 7A.5）。

**反例（不写入）**：

- AI 单方面的设想或推理（即使写得再合理）；
- 仅用户在发散讨论里"提到"但未表态的内容；
- 仅 AI 在回复里"建议"而用户未确认的内容。

### 7A.3 分类与编号

文档内分五节，编号即未来引用 ID：

| 节 | 编号前缀 | 含义 |
|----|----------|------|
| 一、已采纳 · 通用机制 | `A` | 影响通用玩法的核心采纳 |
| 二、已采纳 · 特殊任务/特殊角色类型 | `B` | 限定在某类任务或某类角色上的采纳 |
| 三、后期进阶选项 | `C` | 方向上认同，但不在第一阶段实现 |
| 四、待定区 | `D` | 暂存，回头再考虑 |
| 五、机制库（备查，未采纳） | `M` | 仅备查的外部机制参考 |

新条目优先放在已有同类节的**末尾**，编号沿用前缀 + 自增数字（如 `A3`、`B4`）。

### 7A.4 条目格式

每条至少包含以下字段：

- **来源**：会话日期 / 上下文，例如 `2026-04-27 骰子机制讨论`；
- **机制**：一段话说明该机制本身；
- **应用范围**：通用 / 特定任务类型 / 特定角色 / 进阶 / 待定；
- **状态**：已采纳 / 待定 / 进阶 / 合并 / 已修订 / 已撤回；
- 如有合并、撤回、修订，再追加 **合并备注** / **修订备注**。

具体写法示例参见现有条目（如 A1、B1、C2）。

### 7A.5 修订与撤回

- 用户调整或撤回某条意见时，**保留原条目**，在条目末尾追加：
  - `已修订（YYYY-MM-DD）：…新表述…`
  - 或 `已撤回（YYYY-MM-DD）：…原因…`
- **不允许直接覆盖**或删除原条目，确保决策路径可追溯。
- 引用关系（如 B2 与 B1 合并）也应保留并标记。

### 7A.6 与其他文档的关系

- 本机制与 7 节「新增反馈模板」**互不替代**：7 节记的是"AI 偏差与用户偏好"，本机制记的是"具体玩法/机制的采纳决策"。两者并存。
- `docs/设计采纳记录.md` **不是规格文档**，仅做索引；某条目要落地为代码或字段时，应另起规格文档（或直接落到 `design/gdd/`），并在采纳条目里**回引规格文档**。
- 如果用户后续要求把某条采纳"转为正式真源"，AI 应把该条同步到对应 `design/gdd/` 文档，并在原条目里标注 `已升级为真源：路径`。

### 7A.7 编外内容先确认

- 用户明确纠偏（2026-05-07）：明显属于编外内容、外部商业化素材、活动外观命名、临时文案方案，或与 Angus 当前项目风格 / 玩法真源关系不明确的内容，**不要默认写入** `docs/设计采纳记录.md`。
- 这类内容即使用户选定了名称或文案，也应先判断是否属于项目长期设计决策；若不确定，先询问用户“是否需要沉淀到项目记录”，得到明确同意后再追加。
- 若已经开始准备写入但用户说明“这是编外内容”，应立即停止写入；已写入的需按用户要求修订或移除。

### 7A.8 待细化工作流点子：美术风格稿与视觉组件词典

- 用户明确留档（2026-05-07）：先作为点子记录，后续再详细制定正式美术风格稿和“视觉组件词典”的规则。
- 初步方向：先确立一份可复用的美术风格稿；后续每次玩法改动先在玩法 UI 原型中落地，暂不做完整 GUI 视觉包装；再基于新的玩法原型截图和美术风格稿生成目标效果图；最后把效果图拆解为可实现的视觉组件、状态、符号和布局规则。
- 目的：把“机制是否成立”和“视觉包装是否强化玩家感知”分开验证，避免 AI 直接把规则实现成无感 UI，也避免生图效果图脱离可操作原型。
- 后续待制定：美术风格稿真源格式、视觉组件词典字段、效果图 prompt 模板、效果图拆解模板、真实截图对照验收方式。
- 记录边界：当前只记录工作流点子，不把本轮临时生图尝试或具体风格图本身写入 `docs/设计采纳记录.md`。

## 8. 建议后续 AI 的开场动作

当后续 AI 接手本仓库，尤其接到玩法、原型、交互、UI 感知类任务时，建议顺序：

1. 先读本文。
2. 再读 `design/gdd/core-experience.md`，先确认本次任务服务的核心体验和玩家感受。
3. 若涉及玩法、规则、任务、骰子、黑骰、深度链或路线设计，再读 `design/gdd/gameplay-design-principles.md`。
4. 若涉及 UI / 交互 / 原型呈现，再读 `docs/onboarding/ui-interaction-guidelines.md`。
5. 再读 `docs/onboarding/repository-map.md`。
6. 再确认当前任务讨论的是哪种载体：
   - `design/prototypes/html/`  
   - `gd_project/`  
7. 明确回答自己三个问题：
   - 用户要的是规则层，还是体验层？  
   - 用户要的是最小补丁，还是重做？  
   - 用户最早想在哪个区域 / 界面玩到它？

重要：`design/gdd/core-experience.md` 是最高层体验锚点。后续任何新增或修改该文档的内容，都必须先向用户确认；不能仅凭 AI 自己判断“这很核心”就自动写入。其他设计意见仍按 `docs/设计采纳记录.md` 的规则先沉淀。

如果这三个问题没有答清楚，不要急着改。
