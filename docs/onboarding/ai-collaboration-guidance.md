# AI 协作指引：如何更精准理解当前用户

> **用途**：给后续接手本仓库的 AI / 新对话提供一份可执行的协作约束与偏差复盘。  
> **对象**：不是给用户看的总结稿，而是给 AI 自己看的工作指引。  
> **维护方式**：每当用户给出新的负反馈、纠偏指令或暴露出新的误判模式，都应补充到本文对应章节。  
> **最后更新**：2026-05-13

---

## 1. 文档定位

这份文档记录的不是“项目功能需求”，而是**当前用户的沟通方式、判断标准、常见雷区，以及 AI 在本项目中最容易犯的偏差**。

目标只有两个：

1. 让后续 AI 在接手时更快抓住用户真正要的东西。  
2. 把已经暴露过的误判模式写成硬规则，避免重复犯错。

如果本文与临时想法冲突，**优先遵守本文**，除非用户在新对话里明确推翻。

涉及 UI、交互、原型呈现、视觉实验页时，还必须同时读取并执行 [`ui-interaction-guidelines.md`](./ui-interaction-guidelines.md)。本文保留“为什么犯错”的复盘，UI 文档提供“具体怎么做”的执行规则。

## 2. 用户的核心偏好画像

### 2.1 用户最看重“玩家是否能明显感知到变化”

用户判断“做出来了没有”，不是看规则是否已经在后台生效，而是看：

- 玩家是否在界面上看得到；
- 玩家是否在操作中感知得到；
- 玩家是否能明确分辨“这次和之前不一样”。

因此：

- **规则存在** 不等于 **玩法成立**；
- **结算文案提到** 不等于 **机制被感知到**；
- **内部状态变化** 不等于 **玩家体验发生变化**。

### 2.2 用户偏好“最小改动验证”，不喜欢顺手改大范围外观或载体

当用户说“完善一下某个玩法”时，默认理解为：

- 优先在**当前熟悉的载体**上验证；
- 优先做**最小行为改动**；
- 不要顺手把 UI、节奏、入口、运行方式一起重做。

尤其是：

- 如果用户原本在看 HTML 原型，**不要擅自切到 Godot 运行时做演示**；
- 如果用户要的是“补骰子玩法”，**不要顺手重做阶段切页、整体布局或产品壳层**。

### 2.3 用户希望“首个可玩区域就能体验到核心机制”

用户非常重视“上手即得”。

如果某套新机制需要：

- 解锁后续区域；
- 玩很久才能遇到；
- 或藏在不容易触发的节点里，

用户通常会认为“实际上没做出来”。

因此：

- 优先把新机制集中到**第一个可玩区域**；
- 优先把 showcase 节点放到**最早可触达的位置**；
- 不要把主要机制分散在后续地图里等玩家慢慢打。

### 2.4 用户要“可思考、可控制”的操作权

用户对“自动替玩家推进”的系统反应会比较强，尤其是：

- 自动重投；
- 自动跳过思考阶段；
- 很短的时间窗口；
- 玩家还没理解就已经进入下一段。

因此：

- 玩家需要决策时，必须有**明确的停顿**；
- 最好有**明确按钮**而不是自动继续；
- 锁定 / 重投 / 确认这类动作，优先给玩家手控权。

### 2.5 用户对界面偏差很敏感

如果用户要求的是玩法补强，而结果把界面视觉体验改得面目全非，用户会认为：

- 做偏了；
- 没有抓住重点；
- 甚至怀疑“你是不是在做另一件事”。

因此：

- 改玩法时默认**不改大 UI 风格**；
- 若必须改展示层，优先局部嵌入，不重做整体布局；
- 如果要换载体或大改交互壳层，必须先明确征得用户同意。

## 3. 本次对话里用户提出过的关键问题与指令

以下不是完整逐字转录，而是对后续最重要的需求轨迹摘要。

### 3.1 关于骰子玩法本身

用户反复要求：

- 先发散，再收束；
- 把可操作维度和机制按层级梳理；
- 新机制不能只停留在“系统算了”，必须让玩家明显体会到；
- 没有直接对抗时，也要做出“像被对抗”的感觉；
- 关卡类型（考古 / 科学调查 / 异常接触等）应有不同机制偏好。

### 3.2 关于当前验证载体

用户一开始接受把玩法落进当前 demo，但在实际看到效果后明确表达：

- 不希望因为做骰子玩法，导致界面观感和熟悉的网页版本差太远；
- 重点是**保留熟悉体验的前提下补玩法**；
- 如果 Godot 包体 UI 太开发态，应优先回到 HTML 原型继续做展示。

### 3.3 关于“玩家能感受到”这件事

用户提出的关键判断标准包括：

- 干预、污染等不能只存在于最后结算文字里；
- 每个小机制应该有明显触发时机；
- 机制之间要有间隔，不要一闪而过；
- 单颗骰子要足够清晰，骰子太多时会让所有机制失效；
- 应有手动锁定某个骰子的操作；
- 锁定后要有明确的“开始重投”按钮，不能自动替玩家继续。

### 3.4 关于内容分布

用户明确要求：

- 把主要新机制优先放在北美禁区；
- 因为那是第一个能玩到的区域；
- 不要把 showcase 主要放在后续区域。

### 3.5 关于后续协作

用户希望形成一份长期文档，记录：

- 本次对话里所有关键问题和指令；
- AI 为什么没做好、没考虑到；
- 用户思考方式和偏好；
- 后续新反馈继续往里追加；
- 并且要让别的对话或 AI 接手时也能受到约束。

## 4. 本次对话里 AI 已暴露的主要偏差

### 偏差 1：把“补玩法”误做成“换载体验证”

#### 表现

用户原本更熟悉 HTML 原型，但 AI 为了“让用户能玩到”，优先把新玩法塞进了 Godot `weekly_run`。

#### 为什么这是问题

这改变了用户熟悉的体验壳层。即使规则本身没错，用户看到的也会先是：

- 界面不一样；
- 节奏不一样；
- 操作感觉不一样；

从而掩盖掉用户本来真正想看的“骰子玩法变化”。

#### 后续硬规则

- 如果用户最初讨论的是 HTML 原型或网页体验，默认先在 **HTML** 验证。  
- 不得以“更容易运行”为理由，擅自把玩法迁到另一种载体演示。  
- 如果必须换载体，必须先明确征得用户同意。

### 偏差 2：把“规则被算进去了”误当成“玩家已经体会到了”

#### 表现

AI 先把：

- 类型包、
- 副轴、
- 干预、
- 余波、
- 链式节点

都写进了逻辑层，但玩家看到时仍然觉得“还是原来的简单作废骰”。

#### 为什么这是问题

因为这些变化主要在：

- 计算层；
- 结果文案；
- 状态写回；

而不是在玩家视野里的阶段演出中。

#### 后续硬规则

- 任何新机制如果只在结算文字或状态里可见，默认视为**未做完**。  
- 规则层完成后，必须立刻追问：  
  - 玩家在哪个时刻第一次能感知到它？  
  - 玩家会误以为它还是老机制吗？  
  - 没看说明能不能区分出来？  

### 偏差 3：低估了“骰子数量过多”对可读性的破坏

#### 表现

早期实现仍然使用大量骰子，导致：

- 单颗骰子存在感过低；
- 被干预、被污染、被保住的具体骰子看不清；
- 锁定决策没有重量。

#### 为什么这是问题

当骰子过多时：

- 任何单颗机制都会被淹没；
- 反馈再多也只是噪声；
- 玩家不会觉得自己在“做选择”，只会觉得系统在“播结果”。

#### 后续硬规则

- 只要玩法想强调单颗骰子、锁定、干预或污染，优先先压缩总骰数。  
- 在没有强理由前，把单次判定总骰数控制在 **10 个以内**。  
- 若强调“角色骰”，优先按角色出骰，而不是按属性点数直接堆骰。

### 偏差 4：过早自动推进，剥夺了用户思考窗口

#### 表现

锁定阶段一开始只有 2 秒自动重投。

#### 为什么这是问题

用户明确重视：

- 可思考；
- 可慢慢看；
- 可手动确认；

自动推进会直接破坏这类体验。

#### 后续硬规则

- 玩家需要决策时，默认不要自动继续。  
- 锁定 / 重投 / 确认类交互，优先提供明确按钮。  
- 如果有时间限制，必须先得到用户明确认可。

### 偏差 5：没有足够早地把“核心机制集中到首个区域”当作硬要求

#### 表现

早期把一些 showcase 放到了东亚或更后面区域。

#### 为什么这是问题

用户想要的是：

- 马上打开就能试；
- 第一张图就能覆盖主要机制；

否则就会被判断为“实际上玩不到”。

#### 后续硬规则

- 新机制的第一轮验证，默认优先集中到**首个可玩区域**。  
- 若需要 showcase 节点，也优先放在首区。  
- 后续区域可以留第二版或补充版机制，但不能承载第一轮验证的主要内容。

### 偏差 6：补玩法时顺手改了太多展示层

#### 表现

在 Godot 阶段，AI 一度同时改了：

- 规则层；
- 任务面板文案；
- 阶段显隐；
- 禁用态样式；
- 整体流程切页表现。

#### 为什么这是问题

这会让用户无法分辨：

- 是玩法变了；
- 还是整个界面被重做了。

#### 后续硬规则

- 如果用户要求是“补玩法”，默认展示层改动只做**局部嵌入式增强**。  
- 不主动做全局 UI 重组、阶段切页或视觉风格迁移。  
- 一旦发现实现开始往“另一个任务”扩张，必须停下来收束。

## 5. 当前用户的思考方式与决策风格

### 5.1 用户习惯先发散，再收束

当用户谈新系统时，通常会先：

- 大量发散机制方向；
- 看可能性空间；
- 再要求压成分层、路线、范围。

因此 AI 在前半段可以发散，但一旦用户开始要求：

- 总结、
- 梳理、
- 正式文档、
- 组合建议，

就必须从脑暴切到收束模式。

### 5.2 用户偏好“先做出可玩的最小体验，再慢慢加层”

用户不反对复杂机制，但很明确地希望：

- 先有一个可玩的第一层；
- 再看是否需要更复杂的层；
- 不要一上来把所有层都做进同一套演示里。

### 5.3 用户非常看重“感知到”

用户并不满足于：

- 文档写明了；
- 代码里算到了；
- 结果页写出来了。

用户真正要的是：

- 玩的时候能看出来；
- 能操作到；
- 能记住发生了什么；
- 能感觉到“这次和以前不同”。

### 5.4 用户会持续纠偏，并希望 AI 能把纠偏转成永久规则

用户不是只想要一次性修正，而是希望 AI：

- 把反馈抽象成规律；
- 写进文档；
- 让以后别的 AI 也遵守；
- 避免每次都重新教一遍。

因此，收到用户强反馈后，AI 应主动做：

1. 纠正当前实现；  
2. 把问题上升为规则；  
3. 更新长期协作文档。

## 6. 后续协作硬规则（必须执行）

以下规则是从本次对话里提炼出来的**长期约束**。

### 6.1 关于载体选择

- 默认优先在用户当前讨论和熟悉的载体上验证。  
- 不擅自从 HTML 切到 Godot，或从 Godot 切到 HTML。  
- 如果要换载体，先明确说明原因并征得用户同意。

### 6.2 关于展示与感知

- 新机制必须明确回答：  
  - 玩家在哪个时刻第一次感知到它？  
  - 玩家在哪个界面理解到它？  
  - 它是否只存在于结算文案？  
- 若答案是“只在结算文案里”，则视为未完成。

### 6.3 关于范围控制

- “补玩法”不等于“顺手重做 UI”。  
- 默认只做与用户明确目标直接相关的最小改动。  
- 一旦实现开始扩张到：
  - 全局 UI、
  - 阶段切页、
  - 入口重构、
  - 运行时迁移，
  就必须停下来重新确认。

### 6.4 关于首区试玩

- 第一轮机制验证优先集中在**首个可玩区域**。  
- showcase 节点优先也放在首区。  
- 如果用户还没玩到，就不要把关键体验放到后面区域。

### 6.5 关于玩家决策权

- 只要玩家有“锁定 / 重投 / 确认”决策，默认给明确按钮。  
- 不自动 2 秒倒计时，不替玩家自动推进。  
- 如果用户没明确要求快节奏，就优先让玩家“慢慢看、慢慢选”。

### 6.6 关于骰子数量

- 需要强调单颗骰子的玩法时，先压缩总骰数。  
- 默认目标：单次判定总骰数不超过 10。  
- 角色基础骰优先于属性点直接堆骰。

### 6.7 关于反馈循环

- 用户一旦指出“体会不到”“还是原来的”“太快了”“界面变了”，必须优先判断这是：
  - 感知层问题，
  - 节奏问题，
  - 载体问题，
  - 还是范围失控问题。  
- 不要立刻再加新规则，先处理感知与节奏。

## 7. 后续追加规则

从现在开始，每次用户给出新的强反馈，都应按以下模板补一条：

### 新增反馈模板

- **时间**：YYYY-MM-DD  
- **用户反馈**：用户原话的简要概括  
- **暴露问题类型**：载体 / 感知 / 范围 / 节奏 / 可读性 / 其他  
- **AI 当时偏差**：一句话说明  
- **新增硬规则**：以后必须怎么做  
- **影响范围**：HTML / Godot / 文档 / 所有会话

### 2026-04-28：从采纳记录转方案前必须先复述用户评注意图

- **用户反馈**：用户指出「没有按照我对每条的评注想法来推进具体方案」，要求先理解评注和想法，再进行方案撰写。
- **暴露问题类型**：文档 / 范围 / 其他。
- **AI 当时偏差**：把 `docs/设计采纳记录.md` 中的条目机械扩写成字段与实现清单，忽略了用户评注里的适用范围、层级区分、体验意图和不泛用约束。
- **新增硬规则**：当用户要求把采纳条目转成方案时，AI 必须先按条目复述「用户已采纳的核心意图 / 限定范围 / 不该做什么 / 玩家应感知到什么」，确认理解后再写落地方案；不得直接跳到规格文档或代码字段。
- **影响范围**：文档 / 所有会话。

### 2026-05-09：地图类界面不能用抽象圆斑替代空间语义

- **用户反馈**：用户指出区域任务图“显示效果太差”，希望背景替换为世界地图背景。
- **暴露问题类型**：感知 / UI / 可读性。
- **AI 当时偏差**：为了快速表达任务点分布，用抽象彩色圆斑充当地图背景，导致玩家无法把任务点和世界空间建立联系。
- **新增硬规则**：地图、区域、节点总览等承载空间关系的界面，必须优先使用可识别的地图语义（世界地图、区域轮廓、航线/经纬网、地形或城市结构），不要只用抽象光斑、圆区或纯装饰背景代替。若只是临时占位，也要明确让玩家看出“这是地图”。
- **影响范围**：HTML / Godot / UI / 所有会话。

### 2026-05-09：深度连续任务地图入口应表达“当前入口 + 潜在链”，不要一次摊开整条链

- **用户反馈**：用户指出深度连续任务在地图上的表现很差，不应在地图界面完整展示所有链路节点；地图应只显示当前最新任务，用叠层卡片等方式暗示“多个任务合一”，点击后再展开任务链，而且后续任务应随完成逐步显现。
- **暴露问题类型**：感知 / UI / 认知门槛 / 节奏。
- **AI 当时偏差**：把“深度链”理解成需要在地图上画出完整路径，于是提前暴露多个节点、阶段编号和连线，造成画面杂乱、剧透后续阶段，也让玩家误以为这些都是当前可操作任务。
- **新增硬规则**：地图层只承担入口识别和当前行动选择；连续任务应显示“当前可做的一环”，并用叠层、折角、堆叠阴影、轻微 hover 错位等视觉语言暗示背后有链。点击入口后再展示任务链详情：已完成环可见，当前环可操作，下一环最多以“？”预告，未到达的后续环不得提前完整展示。完成一环后，地图标识应替换为下一环，而不是保留旧环或同时显示整条链。
- **影响范围**：HTML / Godot / UI / 任务链设计 / 所有会话。

### 2026-05-09：深度链叠层交互应从规整堆叠“形变”为线性链，不要另开详情面板

- **用户反馈**：用户指出斜向、扑克牌式的叠层不规整；希望叠层在交互时流畅排列出线性的链，而不是另外出现一个界面展示任务链。
- **暴露问题类型**：感知 / UI / 动效 / 操作连续性。

### 2026-05-14：派遣角色骰面不能退回普通 chip 行

- **用户反馈**：用户指出派遣候选角色卡里的骰面又变成普通分行，而不是之前说过的骰子式长尾十字架，即骰子立方体摊开。
- **暴露问题类型**：UI 感知层回退 / 组件复用错误 / 视觉验收遗漏 / 已采纳规则未执行。
- **AI 当时偏差**：代码里已经存在 `diceNetHtml()` 和 `.dice-net` 样式，角色详情也在使用骰子摊开结构；但我在新版派遣配置和后续压缩候选卡高度时，继续复用了 `diceFacesHtml(..., "staff-pick-faces")` 这种横向 chip 行。根本原因是我把“压缩卡片高度”和“快速展示相关面”放在了“角色骰必须像骰子”的感知原则之前，而且没有把候选角色卡纳入骰子视觉语法验收。
- **新增硬规则**：派遣配置页中，只要展示完整 6 面角色骰，候选角色卡、已选角色槽和角色详情都应优先使用骰子立方体摊开的长尾十字架 / dice net；横向 chip 行只能用于极小摘要、日志或一次性道具简表。任何压缩卡片高度的改动都不得牺牲已采纳的核心视觉语法。
- **影响范围**：HTML / Godot / 派遣配置 / 角色卡 / 角色骰详情 / 截图验收 / 所有会话。

### 2026-05-14：角标和徽章必须有安全区，不能覆盖文字

- **用户反馈**：用户指出全球地图锁定区域按钮上，右上角 `80%` / `82%` 进度角标明显遮挡了区域名称，并要求避免后续任何类似遮挡。
- **暴露问题类型**：UI 可读性 / 角标布局 / 自动验收缺口 / 高密度地图控件。
- **AI 当时偏差**：把数字角标当成装饰性叠层，用 `position:absolute` 贴在按钮右上角，却没有给按钮文字预留安全区；同时只看整体截图，没有检查“主文字矩形”和“角标矩形”是否重叠。根本原因是我把角标视为附加视觉，而不是会占用空间、会随内容长度变化的真实信息组件。
- **新增硬规则**：任何数字角标、进度徽章、状态徽章或头像叠层，只要贴在按钮、地图点、卡片标题或操作区上，都必须拥有独立布局槽位或明确预留安全区。交付前应对高密度界面运行矩形重叠审计，至少检查主文字、角标、倒计时、操作按钮之间不相交且留有可读间距；测试内容必须覆盖 `9+`、百分比、长状态词和多人头像组这类边界文本。
- **影响范围**：HTML / Godot / 全球地图 / 区域地图 / 任务卡 / 派遣队列 / 任何带角标或徽章的按钮与卡片。

### 2026-05-14：只读预览不能长得像可点击任务列表

- **用户反馈**：用户指出全球地图右侧结构和区域任务页很一致，右边铺了很多任务条目，天然会让人想先点任务；但该页唯一功能其实是选择地区并确认进入。
- **暴露问题类型**：UI 交互语法 / 页面任务不清 / 只读预览误导 / 视觉复用过度。
- **AI 当时偏差**：为了让全球地图“信息丰富”，直接复用了区域页的任务卡结构展示预览情报，却没有先判断当前页面的真实操作对象。根本原因是我把数据预览当成内容填充，而不是从玩家会如何读这个界面入手：像任务卡的东西就会被理解为可点任务。
- **新增硬规则**：页面的主要可操作对象必须在视觉语法上唯一。若当前页面只负责选择地区 / 对象，右侧只能显示地区选择器、摘要数值和少量只读预览；不能复用下一层任务卡、hover 高亮、操作按钮或 `interactive` 类。只读预览默认控制在 2 到 3 条，并在自动验收中检查预览容器内没有按钮、链接、`role=button` 或交互类名。
- **影响范围**：HTML / Godot / 全球地图 / 区域选择 / 任务预览 / 任何“上层选择 + 下层详情”的界面。

### 2026-05-14：核心主操作必须大，并放在右下角流程终点

- **用户反馈**：用户指出“进入地图”作为最核心功能，按钮和图标必须大；关键功能按钮一般应放右下角。当前我把进入按钮放在右上角且尺寸偏小，玩家很难找到。
- **暴露问题类型**：主操作权重 / 视觉动线 / 按钮位置 / 可发现性。
- **AI 当时偏差**：在解决“右侧任务列表误导点击”时，我把进入按钮移到面板标题右侧，误以为靠近说明标题就足够清晰；但右上角在这类工具界面里更像辅助操作或状态按钮，且小尺寸无法承担“下一步”的主流程信号。根本原因是我只整理了信息结构，没有同步检查玩家完成当前页面任务时的最后视线落点。
- **新增硬规则**：当前界面的核心功能按钮必须比次级按钮更大、更醒目，并默认放在右下角或玩家完成选择后的流程终点。进入下一层、开始执行、确认提交这类主动作需要明显图标 / 箭头；右上角只适合状态、筛选、次级工具或非主流程操作。交付前必须检查主按钮是否一眼可见，是否比 `下一天`、`返回`、筛选等次级动作更强。
- **影响范围**：HTML / Godot / 全球地图 / 派遣配置 / 判定确认 / 任务详情 / 所有带主操作的界面。

### 2026-05-14：上层选择页不放时间推进，推进后回当前工作上下文

- **用户反馈**：用户指出全球地图页不应该有“下一天”；“下一天”应放在任务派遣界面。第二天到来后默认也应显示当前地区任务派遣界面，当前版本暂不支持 7 天内换地区。
- **暴露问题类型**：流程归属 / 时间推进语义 / 地区锁定 / 页面职责混淆。
- **AI 当时偏差**：我把“下一天”当成全局探索期按钮，觉得放在全球地图也能推进时间；但全球地图当前是地区选择入口，不是玩家日常派遣工作台。把时间推进放在这里会暗示玩家可以在地区间游走并从全球层管理每一天，和当前“选择地区后本周在该地区推进”的流程不一致。
- **新增硬规则**：上层选择页只承担选择与进入，不承担时间推进。`下一天`、结算、结束探索等推进动作必须放在当前工作上下文，比如当前地区任务列表或派遣配置页。推进一天或完成到期结算后，若本周未结束，默认回到当前地区继续派遣；若当前版本不支持周内换地区，就不要提供“返回全球 / 换区”入口。
- **影响范围**：HTML / Godot / 全球地图 / 区域任务页 / 派遣队列 / 周内日循环。

### 2026-05-13：动效演示必须有独立系列页和可直接播放的浏览器入口

- **用户反馈**：用户指出点击动效实验页链接时打开的是 HTML 源码，不是可直接体验的演示网页，并要求写入两条规则：动效演示单独出系列演示页；动效演示页必须给浏览器可以直接播放的版本。
- **暴露问题类型**：UI / 动效 / 实验页 / 入口语义 / 交付方式。
- **AI 当时偏差**：把仓库文件链接当作实验页交付入口，导致用户点击后进入源码视图；同时没有把“动效需要集中分类演示”固化成实验页交付规则。
- **新增硬规则**：当用户要看动效、动画手感或动作语言时，AI 应优先制作独立系列演示页，按动作类型分类并提供重播控制。交付时必须给出浏览器可直接播放 / 操作的预览入口，例如本机 HTTP 链接；源码文件链接只能作为“源码”补充，不能替代演示入口。具体执行见 `docs/onboarding/ui-interaction-guidelines.md` §9.3 与 §11.1。
- **影响范围**：HTML / UI / 动效 / 实验页 / index 入口 / 所有会话。

## 7A. 设计采纳沉淀机制（长期固定机制）

> 自 2026-04-27 起，本机制纳入长期协作约束，所有后续 AI / 新对话都必须遵守。

### 7A.1 沉淀文档

- **文件位置**：[`docs/设计采纳记录.md`](../设计采纳记录.md)
- **文档定位**：用户在讨论中**已采纳 / 待定 / 进阶 / 合并 / 撤回** 的设计意见的长期索引；不是讨论原文，也不是规格文档，而是**决策留痕**。

### 7A.2 何时写入（触发条件）

只要用户在讨论中出现以下任一**明确表态**，AI 在当次回复结束前必须把对应条目追加进该文档：

- 「这个采纳」「加入 …」「作为 …」「可以考虑作为 …」「合并到 …」；
- 「先留着」「待定」「回头再想」「加入待定区」；
- 「作为后期进阶」「作为某类型设计」「作为某角色特性」；
- 「撤回」「不要这条了」「改成 …」（撤回/修订路径，见 7A.5）。

**反例（不写入）**：

- AI 单方面的设想或推理（即使写得再合理）；
- 仅用户在发散讨论里"提到"但未表态的内容；
- 仅 AI 在回复里"建议"而用户未确认的内容。

### 7A.3 分类与编号

文档内分五节，编号即未来引用 ID：

| 节 | 编号前缀 | 含义 |
|----|----------|------|
| 一、已采纳 · 通用机制 | `A` | 影响通用玩法的核心采纳 |
| 二、已采纳 · 特殊任务/特殊角色类型 | `B` | 限定在某类任务或某类角色上的采纳 |
| 三、后期进阶选项 | `C` | 方向上认同，但不在第一阶段实现 |
| 四、待定区 | `D` | 暂存，回头再考虑 |
| 五、机制库（备查，未采纳） | `M` | 仅备查的外部机制参考 |

新条目优先放在已有同类节的**末尾**，编号沿用前缀 + 自增数字（如 `A3`、`B4`）。

### 7A.4 条目格式

每条至少包含以下字段：

- **来源**：会话日期 / 上下文，例如 `2026-04-27 骰子机制讨论`；
- **机制**：一段话说明该机制本身；
- **应用范围**：通用 / 特定任务类型 / 特定角色 / 进阶 / 待定；
- **状态**：已采纳 / 待定 / 进阶 / 合并 / 已修订 / 已撤回；
- 如有合并、撤回、修订，再追加 **合并备注** / **修订备注**。

具体写法示例参见现有条目（如 A1、B1、C2）。

### 7A.5 修订与撤回

- 用户调整或撤回某条意见时，**保留原条目**，在条目末尾追加：
  - `已修订（YYYY-MM-DD）：…新表述…`
  - 或 `已撤回（YYYY-MM-DD）：…原因…`
- **不允许直接覆盖**或删除原条目，确保决策路径可追溯。
- 引用关系（如 B2 与 B1 合并）也应保留并标记。

### 7A.6 与其他文档的关系

- 本机制与 7 节「新增反馈模板」**互不替代**：7 节记的是"AI 偏差与用户偏好"，本机制记的是"具体玩法/机制的采纳决策"。两者并存。
- `docs/设计采纳记录.md` **不是规格文档**，仅做索引；某条目要落地为代码或字段时，应另起规格文档（或直接落到 `design/gdd/`），并在采纳条目里**回引规格文档**。
- 如果用户后续要求把某条采纳"转为正式真源"，AI 应把该条同步到对应 `design/gdd/` 文档，并在原条目里标注 `已升级为真源：路径`。

### 7A.7 编外内容先确认

- 用户明确纠偏（2026-05-07）：明显属于编外内容、外部商业化素材、活动外观命名、临时文案方案，或与 Angus 当前项目风格 / 玩法真源关系不明确的内容，**不要默认写入** `docs/设计采纳记录.md`。
- 这类内容即使用户选定了名称或文案，也应先判断是否属于项目长期设计决策；若不确定，先询问用户“是否需要沉淀到项目记录”，得到明确同意后再追加。
- 若已经开始准备写入但用户说明“这是编外内容”，应立即停止写入；已写入的需按用户要求修订或移除。

### 7A.8 待细化工作流点子：美术风格稿与视觉组件词典

- 用户明确留档（2026-05-07）：先作为点子记录，后续再详细制定正式美术风格稿和“视觉组件词典”的规则。
- 初步方向：先确立一份可复用的美术风格稿；后续每次玩法改动先在玩法 UI 原型中落地，暂不做完整 GUI 视觉包装；再基于新的玩法原型截图和美术风格稿生成目标效果图；最后把效果图拆解为可实现的视觉组件、状态、符号和布局规则。
- 目的：把“机制是否成立”和“视觉包装是否强化玩家感知”分开验证，避免 AI 直接把规则实现成无感 UI，也避免生图效果图脱离可操作原型。
- 后续待制定：美术风格稿真源格式、视觉组件词典字段、效果图 prompt 模板、效果图拆解模板、真实截图对照验收方式。
- 记录边界：当前只记录工作流点子，不把本轮临时生图尝试或具体风格图本身写入 `docs/设计采纳记录.md`。

## 8. 建议后续 AI 的开场动作

当后续 AI 接手本仓库，尤其接到玩法、原型、交互、UI 感知类任务时，建议顺序：

1. 先读本文。
2. 再读 `design/gdd/core-experience.md`，先确认本次任务服务的核心体验和玩家感受。
3. 若涉及玩法、规则、任务、骰子、黑骰、深度链或路线设计，再读 `design/gdd/gameplay-design-principles.md`。
4. 若涉及 UI / 交互 / 原型呈现，再读 `docs/onboarding/ui-interaction-guidelines.md`。
5. 再读 `docs/onboarding/repository-map.md`。
6. 再确认当前任务讨论的是哪种载体：
   - `design/prototypes/html/`  
   - `gd_project/`  
7. 明确回答自己三个问题：
   - 用户要的是规则层，还是体验层？  
   - 用户要的是最小补丁，还是重做？  
   - 用户最早想在哪个区域 / 界面玩到它？

重要：`design/gdd/core-experience.md` 是最高层体验锚点。后续任何新增或修改该文档的内容，都必须先向用户确认；不能仅凭 AI 自己判断“这很核心”就自动写入。其他设计意见仍按 `docs/设计采纳记录.md` 的规则先沉淀。

如果这三个问题没有答清楚，不要急着改。
