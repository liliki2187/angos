# 功能改动截图指引

> **用途**：后续 AI 在完成游戏 / 原型可见改动后，用统一流程生成真实截图与文字说明，避免每次重新摸索，也避免用户逐张翻找。
> **适用范围**：HTML 原型、Godot 原型、可运行 demo、可视化交互改动。

## 1. 交付硬规则

每次游戏 / 原型有可见改动后，最终交付必须包含：

1. **真实运行截图**：从实际页面、游戏窗口或运行时抓取，不能只贴设计稿或口头描述。
2. **文字说明**：说明每张截图“看什么、改了什么、玩家应感知到什么”。
3. **展示方式**：1-3 张截图可直接在对话里展示并配说明；多于 3 张截图时，必须生成一个 `index.html` 或等价说明页，把截图按玩家流程串起来。
4. **验证说明**：最终回复里说明截图位置、验证方式、是否有未覆盖的风险。
5. **复杂 UI 标注**：复杂页面、分阶段 UI 重构或“改动点不容易肉眼定位”的交付，应补一张程序化标注截图；标注基于真实运行页面的 DOM / UI 元素坐标绘制，不使用 AI 生图作为验收依据。

推荐目录：

```text
docs/screenshots/YYYY-MM-DD-简短主题/
  01-xxx.png
  02-xxx.png
  index.html
```

## 2. 截图前检查

截图前先确认：

- 当前任务的验证载体是什么：HTML、Godot，还是其他 demo。
- 用户最关心的改动点在哪些界面首次出现。
- 是否需要覆盖“点击前”和“点击后”的差异。若玩家必须点击才能理解机制，必须截点击后的状态。
- 默认不额外制作竖屏手机 / 极窄移动端截图；当前阶段暂不考虑竖屏手机端使用。只有用户明确要求、任务本身涉及移动端 / 窄屏适配，或改动点明显依赖小视口布局时，才补对应截图。
- 是否有新手引导、弹窗、随机事件会遮挡关键画面；若遮挡，应在截图脚本里关闭或稳定处理。

## 3. HTML 原型截图流程

优先用浏览器真实渲染截图。若项目没有 Playwright / Puppeteer，可用 Chrome / Edge 的 DevTools Protocol 临时脚本，不必新增依赖。

### 3.1 启动浏览器

检查浏览器路径：

```powershell
Test-Path "C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe"
Test-Path "C:\Program Files\Google\Chrome\Application\chrome.exe"
```

可用参数：

```text
--headless=new
--remote-debugging-port=<port>
--user-data-dir=<临时目录>
--allow-file-access-from-files
--window-size=1440,1100
```

### 3.2 通过 CDP 控制页面

Node 24 已有 `WebSocket`，可直接连接：

1. 启动浏览器。
2. 访问 `http://127.0.0.1:<port>/json/list` 获取 `webSocketDebuggerUrl`。
3. 调用：
   - `Page.enable`
   - `Runtime.enable`
   - `Emulation.setDeviceMetricsOverride`
   - `Page.navigate`
   - `Runtime.evaluate`
   - `Page.captureScreenshot`

截图脚本应放在临时文件里，完成后删除，不要把一次性脚本留在仓库根目录。

### 3.3 稳定截图建议

- 截图前写入 localStorage 关闭新手引导：

```js
localStorage.setItem("wm_angos_no_tutorials_v1", "1");
localStorage.setItem("wm_angos_tutorial_intro_offer_done_v1", "1");
```

- 每次点击后等待目标状态，而不是固定 sleep：

```js
await waitFor("!document.getElementById('view-setup').classList.contains('hidden')");
```

- 对随机结算截图，必要时临时固定 `Math.random` 序列，让结果稳定复现。
- 截图前 `window.scrollTo(0, 0)`，除非要展示长页中段。

### 3.4 程序化改动标注截图

复杂 UI 改动建议生成三类截图：

1. **旧版同状态**：用于确认改动前玩家看到什么。
2. **新版同状态**：用于确认真实页面改成了什么。
3. **新版标注版**：用脚本在真实页面上叠加编号、框线和短说明。

标注实现建议：

- 用 Playwright / CDP 获取真实页面渲染后的元素位置，例如 `getBoundingClientRect()`。
- 在页面上临时注入 overlay、label、banner，再截图；截图后删除临时脚本，不把一次性标注逻辑留在正式代码里。
- 标注文字要短，只说明“改了哪里 / 玩家要看什么”，不要写成长说明书。
- 标注图只作为交付说明；真实验收仍以无标注截图、运行路径和自动/人工检查为准。
- 如果一张标注图会遮挡太多内容，拆成多张局部标注图或改用截图汇总页。

## 4. HTML 原型常见坑

### 4.1 外部 JS 与 HTML 内联脚本不同步

本项目部分 HTML 原型历史上存在“HTML 内联脚本”和同名外部 JS 并存的情况。截图前必须确认当前页面实际执行的是改过的脚本。

检查方法：

- 搜索 HTML 是否存在大段 `<script> ... </script>` 内联逻辑。
- 若改动落在外部 `world-mysteries-full-chain.js`，但 HTML 仍执行旧内联脚本，应先改成加载外部脚本，或同步改内联脚本。
- 截第一张图后要肉眼确认新标签 / 新文案确实出现；没有出现就不要继续批量截图。

### 4.2 弹层遮挡

奖励弹窗、日推进弹窗、确认弹窗、新手引导弹窗都可能阻塞截图流程。截图脚本应提供：

- `closeRewards()`
- `closeTutorials()`
- `clickConfirmOk()`
- `waitForVisible(selector)`

### 4.3 文件路径

Windows 下生成页面链接时，截图汇总页内优先使用相对路径：

```html
<img src="01-region-overview.png" alt="..." />
```

不要写绝对盘符路径，避免换机器后失效。

## 5. 截图汇总页结构

`index.html` 至少包含：

- 标题：本次改动主题与日期。
- 摘要：这次改了哪些可见功能。
- 导航：按玩家流程跳转。
- 每张截图：
  - 编号与标题；
  - 对应文件；
  - 改动点；
  - 玩家应感知到什么；
  - 相关机制标签。

推荐一张图一个区块：

```html
<article>
  <h3>02 · 白色调查配置</h3>
  <p>说明玩家在配置页能看到白色调查的行为差异。</p>
  <img src="02-white-investigation-setup.png" alt="白色调查配置页截图" />
</article>
```

## 6. Godot 原型截图流程

若改动落在 Godot：

1. 优先运行现有项目启动命令，不擅自切换验证载体。
2. 若可用自动化截图命令，使用项目已有脚本；没有脚本时，用 Godot 运行窗口截图。
3. 截图必须覆盖：
   - 首次可见入口；
   - 玩家点击后的说明 / 操作状态；
   - 结算或反馈状态。
4. 同样生成 `docs/screenshots/YYYY-MM-DD-主题/index.html`。

## 7. 最终回复模板

最终回复简短说明：

```text
已生成截图说明页：
`docs/screenshots/YYYY-MM-DD-主题/index.html`

覆盖：
- 入口/点击前状态
- 点击后的配置或操作说明
- 结算/反馈状态

验证：
- ReadLints 无报错
- 运行 / 语法检查通过
```

若无法截图，必须说明原因，并给出替代验证方式；不能只说“已改完”。

## 8. 对话负载与新对话提醒

截图越多，对话越容易变慢。执行截图交付时应主动控制对话负载：

- 1-3 张关键截图可直接贴在对话里。
- 超过 3 张时，优先生成截图汇总页，并在最终回复中只贴关键图或给出汇总页路径。
- 为节省时间与对话负载，常规 UI / 原型回归只覆盖玩家主要使用视口；不要默认追加竖屏手机端或极窄长页截图。
- 如果当前对话已经积累了大量截图、长链路 UI 讨论、反复截图重跑或明显开始未响应，应直接提醒用户“建议新建对话继续”，并附上最小交接信息：当前目标、已改文件、截图目录、验证命令和下一步。
- 不要为了维持同一对话而继续塞大量图片；可读性和响应稳定性优先。

## 9. 完成前检查清单

- [ ] 截图来自真实运行界面。
- [ ] 每个可见改动至少有一张对应截图。
- [ ] 多张截图已整理成汇总页。
- [ ] 复杂 UI / 分阶段改动已提供程序化标注截图，或说明为什么不需要。
- [ ] 截图数量已控制；若当前对话过重，已建议用户新建对话。
- [ ] 汇总页内图片链接使用相对路径。
- [ ] 最终回复给出汇总页路径，而不是只列图片文件名。
- [ ] 临时截图脚本、临时浏览器 profile 已清理。
- [ ] 已运行必要的 linter / 语法检查。
