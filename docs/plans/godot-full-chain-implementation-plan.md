# Godot 全链条实现计划

## 背景

截至 2026-03-24，Angus 已经有：

- 可单独运行的事件检定原型
- 主菜单与神秘特刊预览
- 多份网页参考原型
- 新整理出的探索、全链条和报刊结算设计细节

但还没有一个能在 Godot 中跑通“探索 -> 线索 -> 编辑 -> 结算 -> 下一周”的完整周循环。

本计划的目标就是把这条链拆成可以连续推进的工作包。

## 目标

1. 在 Godot 中跑通一条完整周循环。
2. 统一探索、检定、编辑、结算的状态层。
3. 避免继续依赖零散网页原型来表达核心玩法规则。

## 工作拆分

### Phase 1：全局状态与周循环骨架

- 建立统一的周状态管理，承载 `week/day`、五维属性、订阅数、`editorialProfile`、本周线索池。
- 明确哪些状态跨周保留，哪些状态周结算后清空。
- 把主菜单进入点与周循环主场景衔接起来。

交付标准：

- 能进入“第 1 周”
- 能消耗天数
- 能结束本周并进入下一阶段

### Phase 2：探索地图与节点

- 实现全球层、区域层和基础节点列表。
- 支持常驻 / 临时 / 隐藏节点的显示逻辑。
- 支持职员派遣、需求预览和节点禁用状态。

交付标准：

- 至少 2 个区域、若干节点可完整试玩
- 节点能按日数和解锁条件限制进入

### Phase 3：事件检定接入探索

- 把现有事件检定原型接成探索节点的真实执行步骤。
- 确认 split 检定与对手抵消逻辑回写探索结果。
- 把结果档位映射到线索质量与属性变化。

交付标准：

- 节点执行后能真实产出线索
- 检定结果会影响周刊长期属性

### Phase 4：线索转故事与编辑部场景

- 定义线索到故事的转换规则。
- 建立固定 6 版位的编辑部场景。
- 支持拖拽放置、替换、移除、清空和实时预览。

交付标准：

- 本周线索能自动变成故事库
- 玩家能完成一轮组版操作

### Phase 5：经济结算与跨周继承

- 落地销量、广告、订阅、成本、利润公式。
- 更新 `editorialProfile` 与订阅数。
- 结算后进入下一周并保留长期状态。

交付标准：

- 可以完整结算一期
- 下一周会带着上一周结果继续运行

### Phase 6：配置化与内容生产

- 把节点、检定和故事补稿逻辑逐步从硬编码迁到配置。
- 引入事件脚本示例与字段模板。
- 为策划迭代预留稳定的数据入口。

交付标准：

- 至少一部分探索和检定内容可通过配置扩展

## 近期优先级

当前最该先做的是：

1. 全局状态层
2. 探索节点 -> 事件检定 -> 线索产出
3. 编辑部 6 槽结算

不要先做：

- 大规模美术 polish
- 报社等级扩展版面
- 复杂势力分支
- 多套并行规则实验

## 风险

- 如果没有统一状态层，探索和编辑会继续变成独立原型。
- 如果不先打通线索到故事的接口，编辑部只能靠手工造数据。
- 如果不尽快把结算公式落进 Godot，网页原型和 Godot 原型会持续分叉。
