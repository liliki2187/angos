# ADR-0002：Godot UI 场景优先与绑定式脚本规范

## 状态
已接受

## 日期
2026-04-08

## 背景

### 问题陈述

Angus 后续会继续以 Godot 作为唯一正式运行时推进，而项目当前已有的若干原型 UI 主要通过代码运行时动态创建 `Button`、`Label`、`Container` 等控件来拼装界面。这种方式适合快速验证，但一旦进入持续开发，会带来几个问题：

- UI 层级、布局和视觉结构难以在编辑器中直接查看与调整。
- 程序、美术、GUI 设计和策划难以围绕同一份 `.tscn` 场景协作。
- AI 代理容易把原型写法误当成正式写法，继续扩散运行时手搓控件树。
- 数据绑定、交互逻辑和控件构造混在一起，导致脚本职责失衡。

项目已经通过 `ADR-0001` 确认要分离内容数据、周状态 / 规则层和场景表现层；因此 UI 层也需要进一步明确实现约束。

### 约束

- 当前团队需要保留 Godot 原型中的一部分可运行行为，不能为 UI 规范一次性全面重写。
- 项目后续会持续引入 PSD、美术稿和高保真布局，需要有稳定的场景挂接位置。
- 某些数量可变的列表、卡片、日志项仍需要运行时生成实例。
- 少数开发工具、调试界面和一次性演示界面仍可能需要更高自由度。

### 要求

- 正式 UI 必须优先基于 `.tscn` 场景和可复用组件场景进行搭建。
- 运行时代码应主要负责节点引用、信号连接、数据绑定和状态切换。
- 需要允许“实例化预制 item scene”这类合理的动态行为，但不允许继续手写固定布局的原始控件树。
- 必须让 AI 和团队成员都能从仓库文档中清楚判断什么是正式 UI 写法，什么只是原型例外。

## 决策

Angus 将采用“场景优先、绑定驱动”的 Godot UI 规范。

### 规范摘要

- 正式 UI 的布局、层级、容器关系和基础控件结构，应优先在 `.tscn` 中完成。
- 正式 UI 脚本默认只负责：
  - 缓存节点引用
  - 连接信号
  - 绑定显示数据
  - 更新显隐、禁用、文案、样式状态
  - 实例化预先制作好的组件场景
- 对于数量可变的 UI 内容，应优先实例化独立的组件场景，例如卡片项、列表项、日志项、槽位项，而不是直接在代码里 `Button.new()` / `Label.new()` / `HBoxContainer.new()` 现场拼树。
- UI 不拥有玩法真状态。UI 只读取状态、展示状态，并通过信号或明确接口发出动作请求。
- `dev`、`debug`、临时原型、纯表现特效或一次性工具界面，可以保留受控例外；但需要在对应 spec、注释或 ADR 中说明其性质，不得被当作正式模板扩散。

### 架构示意

```text
设计稿 / PSD / 组件规范
        |
        v
Godot UI 场景（.tscn）
  - Screen 场景
  - 复用组件场景
  - item 场景
        |
        v
UI 绑定脚本
  - onready 引用
  - 信号连接
  - 数据绑定
  - 状态切换
  - 组件实例化
        |
        v
周状态 / 规则层
  - 只读展示数据
  - 动作请求接口
```

### 关键接口

- **设计资源 -> UI 场景**
  - 高保真稿、PSD、切图和组件说明进入 `design/ui/`、`gd_project/Assets/ui/` 与 `gd_project/scenes/ui/`
- **UI 场景 -> UI 脚本**
  - 通过稳定节点路径、导出属性或命名约定完成绑定
- **UI 脚本 -> 状态 / 规则层**
  - 读取状态快照
  - 发出动作请求
  - 不直接拥有长期玩法状态

## 备选方案

### 方案 1：继续允许正式 UI 以代码动态拼装控件树为默认模式

- **描述**：保留当前原型习惯，继续在脚本里动态创建多数固定布局控件。
- **优点**：短期写起来快，适合单人快速试验。
- **缺点**：难以维护、难以协作、难以让 AI 稳定复用，也不利于美术和 GUI 人员直接接手。
- **否决原因**：这会继续把原型债务扩散到正式运行时。

### 方案 2：完全禁止任何运行时 UI 实例化

- **描述**：所有 UI 都必须完全静态写死在场景里。
- **优点**：规则最简单，最容易审查。
- **缺点**：不适合卡片、日志、列表、槽位等数量动态变化的界面，也会让复用能力变差。
- **否决原因**：过于僵硬，不符合项目实际需要。

### 方案 3：仅在口头约定中说明，不写入仓库规范

- **描述**：团队内部知道这条偏好，但不形成正式文档。
- **优点**：零文档成本。
- **缺点**：AI 无法稳定遵守，新成员也无法判断哪些写法是允许的。
- **否决原因**：该决策正是为了降低口头规则失效带来的漂移。

## 影响

### 正面

- UI 布局和层级可以在 Godot 编辑器中直接检查和调整。
- 美术、GUI、程序能够围绕同一份场景资源协作。
- AI 在读取 `technical-preferences`、ADR 和 spec 时，会更倾向于生成“场景 + 绑定脚本”的实现。
- 更利于后续将 PSD 导入链路、组件复用和正式 UI 资产接入运行时。

### 负面

- 短期内会暴露出现有原型脚本中大量动态拼控件的历史债务。
- 某些原型脚本重构时，需要先补组件场景，再迁移逻辑，初期速度会略慢。

### 风险

- **风险**：团队把“允许运行时实例化组件场景”误解成“仍可随意 `new` 原始控件”。
  - **缓解**：在 `technical-preferences` 和功能 spec 中明确区分“实例化预制场景”与“现场拼原始控件树”。
- **风险**：旧原型文件继续被 AI 当作正例模仿。
  - **缓解**：在后续正式功能中尽快建立新的 UI 正例目录与样板场景。
- **风险**：为追求规范而过度阻碍调试工具或短期实验。
  - **缓解**：保留 `dev` / `debug` / 临时原型例外，但要求注明用途。

## 性能影响

- **CPU**：中性到正向；减少频繁的运行时控件构造通常更稳定。
- **内存**：中性；会增加一些场景资源文件，但有利于控件复用和缓存。
- **加载时间**：中性；预制场景实例化与动态拼树在本项目规模下差异可控。
- **网络**：无。

## 迁移计划

1. 将当前通过代码动态拼装大量固定布局的 Godot 页面视为原型债务，而不是正式模板。
2. 从新的正式功能开始，优先建立 `Screen.tscn + Screen.gd + Item.tscn` 的结构。
3. 对数量可变的 UI 列表，先抽出可复用 item scene，再将旧逻辑改为实例化该场景。
4. 对已存在的原型脚本，按功能切片逐步迁移，不进行一次性全量重写。
5. 在 `docs/technical-preferences.md` 中写入简明硬规则，并在相关 `specs/` 中重复当前功能的 UI 约束。

## 验证标准

- 新增正式 UI 功能时，默认能看到对应 `.tscn` 场景与配套脚本，而不是只有一个在代码里拼树的 `.gd`。
- 对于动态列表，代码主要实例化预制 item scene，而不是直接 `new` 多层控件树。
- UI 脚本不再同时拥有控件构造、玩法状态、公式和表现逻辑。
- 后续 AI 在实现 UI 功能时，会优先选择场景拼装 + 绑定的结构。
