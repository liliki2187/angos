# 事件检定结算

> **状态**：草案
> **作者**：Codex + 仓库综合整理
> **最后更新**：2026-05-08
> **对应支柱**：调查必须导向发刊；两条主轴必须持续拉扯

## 概览

这个系统负责把派遣风险结算出来。它将员工属性、节点难度和对手压力转换为结果层级，底层模型是拆分二项式检定。在 Angus 中，它是“规划”和“后果”之间的桥。

## 玩家幻想

玩家应感觉到准备工作是有价值的，风险是可读的，而危险机会依然诱人，因为系统会展示胜率，却不会让结果失去悬念。

## 详细设计

### 核心规则

1. 事件检定使用二项式成功投掷，而不是简单阈值判定。
2. 难度映射为单骰基础成功率。
3. 检定被拆成两个池：
   - 调查池：探索 + 洞察 + 神秘
   - 行动池：生存 + 理性
4. 对手值通过随机抵消有效骰子数来施压，同时保持底层概率模型一致。
5. 结果层级分为：
   - 大成功：两个池都达到阈值
   - 小成功：只有一个池达到阈值
   - 失败：两个池都未达到阈值
6. 结果层级会映射到线索质量和副作用，而不是直接终止游戏。

### 危险标识与黑骰任务

普通任务继续使用基础结果层级。当前已明确区分 **危险标识** 与 **黑骰任务**：

- **危险标识**：不是独立任务类型，而是普通任务、红色截稿或深度链某一环上的高难度标记。它表示需求更高、失败后果更重，可能带来 debuff，但机制仍应简单，不使用黑骰。
- **黑骰任务**：通常是深度调查任务链的最后一环；也可由个别事件、主线相关临时任务触发。黑骰任务使用特殊判定机制，具体黑骰能力由关卡设计定义。

黑骰任务结果层级为：

| 结果 | 含义 |
|------|------|
| 大成功 | 达成目标且避开主要反噬。 |
| 成功 | 达成目标，但可能留下轻微代价或风险标记。 |
| 失败 | 未完全达成目标，仍可能留下弱线索或后续压力。 |
| 大失败 | 触发较重 debuff，影响当局后续。 |

后续进入角色骰系统后，非黑骰任务暂只看是否达标；黑骰任务才使用上述四档。旧 HTML 原型中的“高危五档”表述应视为历史过渡方案。

角色骰首版中，任务需求里的多个属性表示“哪些骰面能为本任务提供有效点”，而不是要求每个属性都在单次掷骰中逐项硬达标。普通任务的实际达标口径为：已计入的相关骰面有效点达到任务目标值。目标值由需求总量折算，当前 HTML 原型为：

```
need_total = sum(node.need values)
need_target = max(need_total >= 4 ? 2 : 1, ceil(need_total / 3))
success = sum(selected relevant face values) >= need_target
```

这样保留“探索 / 生存 / 洞察等属性决定哪些角色适配”的规划意义，同时避免 3 名角色每人只掷 1 面时，多属性任务因为属性数量下限而被错误抬成几乎不可能成功。

难度验收不能只看需求条是否覆盖，也不能把 3 人满编当作所有任务的默认前提。首周 / 新手区建议区间：

- **普通 / 常驻任务**：最低难度层，应允许适配 1 到 2 人就有可玩的成功率。适配 1 人约 35% 到 55%，适配 2 人约 60% 到 80%，适配 3 人约 75% 到 90%；若需要 3 人满编才到 40% 左右，就不应标为普通任务。
- **线索追查 / 深度链第 1 环**：略高于普通任务，但仍应以 2 人为默认可玩单位。适配 2 人约 50% 到 70%，适配 3 人约 65% 到 85%。
- **深度链中段 / 红色截稿**：可以要求更明确的角色适配或支援资源。适配 2 人约 40% 到 60%，适配 3 人约 55% 到 75%；红色截稿因为会关闭，应避免低到“基本不可能”。
- **危险 / 高危任务**：允许 2 人冒险、3 人可搏、道具明显改善。适配 2 人约 30% 到 50%，适配 3 人约 45% 到 65%，使用对应道具或临时资源后约 60% 到 80%。
- **黑骰终局**：难点来自黑骰反噬和四档后果，不应只靠基础达标率压死。适配 3 人可低，适配 5 人应约 45% 到 65%，支援资源后应能进入约 60% 以上。

若具体任务低于对应范围，优先调整任务需求总量、目标公式、骰面密度、区域解锁或支援资源，而不是只改变风险标签。

### 状态与切换

| 状态 | 进入条件 | 退出条件 | 行为 |
|------|---------|---------|------|
| `preview` | 已选好合法节点与员工 | 按下执行 | 展示理论成功分布 |
| `roll_generation` | 按下执行 | 骰子与抵消生成完毕 | 生成投掷数组与被抵消索引 |
| `resolution` | 投掷已生成 | 结果已应用 | 统计有效命中并给出层级 |
| `post_result` | 结果已应用 | 玩家返回探索 | 输出线索与状态变化 |

### 与其他系统的交互

- **探索与节点派遣** 提供员工总值、节点难度、阈值和对手值。
- **线索与内容素材库存** 消费结果层级和线索元数据。
- **反馈 UI** 会读取投掷数组、被抵消索引和概率摘要用于表现。
- **宏观属性与现实偏移** 可能接收来自失败或高神秘成功的后续变化。

## 公式

### 难度到单骰成功率

```
p =
  0.60 if difficulty == "easy"
  0.50 if difficulty == "normal"
  0.3333333333 if difficulty == "hard"
```

| 变量 | 类型 | 范围 | 来源 | 说明 |
|------|------|------|------|------|
| `difficulty` | enum | easy/normal/hard | 节点数据 | 难度标签 |
| `p` | float | 0.0-1.0 | 查表 | 单骰成功率 |

### 对手分配与抵消

```
enemy_a = floor(enemy_attr * attr_a / max(1, attr_a + attr_b))
enemy_b = enemy_attr - enemy_a
n_a_eff = max(0, attr_a - enemy_a)
n_b_eff = max(0, attr_b - enemy_b)
```

| 变量 | 类型 | 范围 | 来源 | 说明 |
|------|------|------|------|------|
| `enemy_attr` | int | 0+ | 节点数据 | 对手压力 |
| `attr_a` | int | 0+ | 员工聚合值 | 调查池大小 |
| `attr_b` | int | 0+ | 员工聚合值 | 行动池大小 |
| `n_a_eff`、`n_b_eff` | int | 0+ | 计算值 | 抵消后的有效骰子数 |

### 拆分结果分布

```
P_A = P(X_A >= k_A), where X_A ~ Binomial(n_a_eff, p)
P_B = P(X_B >= k_B), where X_B ~ Binomial(n_b_eff, p)
P_major = P_A * P_B
P_minor = P_A * (1 - P_B) + (1 - P_A) * P_B
P_fail = (1 - P_A) * (1 - P_B)
```

| 变量 | 类型 | 范围 | 来源 | 说明 |
|------|------|------|------|------|
| `k_A`、`k_B` | int | 0+ | 节点数据 | 两个池各自的阈值 |
| `P_major` | float | 0.0-1.0 | 计算值 | 大成功概率 |
| `P_minor` | float | 0.0-1.0 | 计算值 | 小成功概率 |
| `P_fail` | float | 0.0-1.0 | 计算值 | 失败概率 |

## 边界情况

| 场景 | 预期行为 | 理由 |
|------|----------|------|
| 有效骰子变成 0，且阈值大于 0 | 该池自动失败 | 维持概率模型完整性 |
| 阈值为 0 | 该池自动成功 | 保持公式定义完备 |
| 对手压力严重偏向单池 | 总抵消量仍然守恒 | 维持模型一致性 |
| UI 选择数字模式而非骰子模式 | 底层随机逻辑一致，只是表现不同 | 防止两种模式数值不同步 |

## 依赖

| 系统 | 方向 | 依赖性质 |
|------|------|----------|
| 探索与节点派遣 | 本系统依赖它 | 消费检定输入数据 |
| 线索与内容素材库存 | 它依赖本系统 | 根据结果层级确定线索质量 |
| 反馈、日志与预测 UI | 它依赖本系统 | 展示概率与投掷结果 |
| 宏观属性与现实偏移 | 软依赖 | 某些结果可能修改长期状态 |

## 调参旋钮

| 参数 | 当前值 | 安全范围 | 提高后的效果 | 降低后的效果 |
|------|--------|----------|--------------|--------------|
| `p_easy` | 0.60 | 0.55-0.70 | 远征更容易成功 | 成功稳定性下降 |
| `p_normal` | 0.50 | 0.45-0.55 | 中档玩法更稳定 | 不确定性更尖锐 |
| `p_hard` | 0.3333 | 0.25-0.40 | 高难节点没那么惩罚 | 风险门槛更严 |
| 节点阈值 `k_A`、`k_B` | 按节点配置 | 低到中 | 失败率上升 | 派遣规划压力减轻 |
| 对手值 `enemy_attr` | 按节点配置 | 0-5+ | 抵消压力更强 | 成功曲线更平滑 |

## 视觉 / 音频需求

| 事件 | 视觉反馈 | 音频反馈 | 优先级 |
|------|----------|----------|--------|
| 概率预览 | 清晰展示大成功 / 小成功 / 失败百分比 | 可无 | 高 |
| 骰子结算 | 明显的投掷揭示与抵消过程 | 骰子或结算提示音 | 中 |
| 结果揭示 | 大而清晰的结果层级与后果文本 | 鲜明的成功 / 失败提示音 | 高 |

## UI 需求

| 信息 | 显示位置 | 更新频率 | 条件 |
|------|----------|----------|------|
| 池需求 | 任务面板 | 节点变化时 | 派遣规划阶段 |
| 队伍总值 | 任务面板 | 员工变化时 | 派遣规划阶段 |
| 大成功 / 小成功 / 失败分布 | 任务面板 | 每次合法预览 | 执行前 |
| 投掷细节与抵消 | 结果区 / 日志 | 执行时 | 仅骰子模式 |
| 危险标识风险说明 | 任务面板 / 结果区 | 带危险标识的任务配置与结算时 | `riskTier == high`，但无黑骰 |
| 黑骰结果说明 | 结果区 | 黑骰任务结算时 | 任务配置了黑骰机制 |

## 验收标准

- [ ] 数字预览与真实结算使用完全相同的难度映射。
- [ ] 对手抵消保留“总骰子量”语义，而不是只删掉已成功的点数。
- [ ] 大成功 / 小成功 / 失败层级与文档阈值一致。
- [ ] 结果输出包含足够数据，供 UI 回放或日志记录。
- [ ] 线索质量可直接由结果层级驱动，无需额外隐藏规则。
