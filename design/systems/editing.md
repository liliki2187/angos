# 周刊编辑系统设计

## 定位

编辑系统是每周循环的后半段，负责把探索阶段得到的线索转成报道，再通过组版与结算把这一周转换为销量、收入、订阅和长期属性变化。

它既是探索的消费端，也是下一周长期状态的生产端。

## 当前实现目标

正式成长版面可以保留“报社等级影响版面数量”的长线规划，但当前 Godot 原型的近期实现目标以固定 6 版位为准，先完成一个完整可结算的周循环。

固定 6 版位：

| id | 名称 | weight | 作用 |
|----|------|--------|------|
| `front-main` | 头版头条 | 1.0 | 最高曝光 |
| `front-side` | 头版次条 | 0.7 | 高曝光 |
| `feature-1` | 重点专题 A | 0.45 | 中高曝光 |
| `feature-2` | 重点专题 B | 0.4 | 中高曝光 |
| `inner-1` | 内页 A | 0.2 | 标准曝光 |
| `inner-2` | 内页 B | 0.2 | 标准曝光 |

## 故事数据结构

编辑部消费的最小单位是故事，而不是线索。

建议故事字段：

| 字段 | 说明 |
|------|------|
| `id` | 故事唯一标识 |
| `title` | 报道标题 |
| `tags` | 1-2 个题材标签 |
| `quality` | `Bronze / Silver / Gold` |
| `baseValue` | 基础价值 |
| `negatives` | 负面标签数组 |
| `source` | `exploration` 或 `filler` |

线索转故事的基础映射：

| 线索字段 | 故事字段 |
|---------|---------|
| `title` | `title` |
| `type` | `tags` 的主标签 |
| `tier` | `quality` 与 `baseValue` |

不足稿件数量由系统自动补足，避免因探索条数过低导致编辑部无法验证整条结算链路。

## 题材三轴

编辑与结算阶段不再只看“科学 / 玄学 / 流量”的概念分层，而要进入更可计算的三轴占比。

| 轴 | 包含标签 |
|----|----------|
| 公共事务 P | Politics / Military / Economy |
| 大众关注 M | Sport / Shopping |
| 轻内容 L | Gossip / Pets / Humor |

三轴用于：

- 判断是否偏科
- 判断是否均衡
- 推动编辑定位 `editorialProfile`
- 影响需求与收益乘数

## 编辑定位

`editorialProfile` 是跨周继承的长期编辑倾向。

- 负向偏信息型
- 正向偏热度型
- 居中为平衡型

它的作用不是单独决定胜负，而是调节：

- 连携收益
- 多样性收益
- 广告匹配表现

## 组版规则

### 基础交互

- 故事可拖拽到任意版位
- 版位最多放 1 篇报道
- 允许覆盖已有报道
- 支持单槽移除和全版面清空

### 实时预览

拖拽悬停时应实时预览：

- 利润变化
- 销量变化
- 连携高亮
- 偏科减益
- 替换提示

这部分是当前 Godot 里尚未正式落地、但最值得从网页原型吸收的交互层。

## 结算模型

### 中间量

至少需要统计：

- 已填版位数
- 空版位数
- `totalBaseValue`
- `uniqueTags`
- `comboRaw`
- `negPenaltyPoints`
- `linkedTags`
- 三轴占比

### 核心乘数

当前基线应保留以下乘数：

- `mQuality`
- `mCombo`
- `mDiversity`
- `mLayout`
- `mEmpty`
- `mPenalty`
- `mLink`
- `mBalance`
- `mBias`

### 结算结果

至少输出：

- 需求 `demand`
- 实际销量 `sold`
- 卖报收入
- 订阅收入
- 广告收入
- 总收入
- 纸张 / 工资 / 运营成本
- 净利润 `profit`
- 下一周订阅数

## 与探索系统的关系

编辑系统不是独立小游戏，它必须承接探索结果。

完整衔接关系：

1. 探索阶段产出线索。
2. 线索转为本周故事库的核心稿件。
3. 系统补足随机稿件。
4. 玩家完成组版并结算。
5. 结算回写到周刊长期状态。

如果这条链没打通，编辑系统的数值再完整也只是单独面板。

## 长线扩展

在完成固定 6 版位原型后，再考虑恢复更长期的成长设计：

- 报社等级提升版面数量
- 强制刊登 / 禁止刊登任务
- 势力与专题规则
- 万能缓冲位
- 连载与跨周专题加成

这些适合放在第二阶段，不应阻塞当前 Godot 的完整周循环实现。

## Godot 落地建议

- 先做固定 6 槽结算版，再谈等级扩展。
- 先做数值和流程正确，再做高级视觉排版。
- 编辑部必须从探索线索直接生成故事库，避免人工造数。
- 结算结果必须回写全局状态，形成真正的下一周输入。
