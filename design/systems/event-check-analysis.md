## 1. 系统定位与核心价值

### 1.1 在游戏架构中的位置

事件检定系统是 ANGUS 的**核心数值引擎**，贯穿游戏所有交互场景：

```
探索事件 → 触发选项 → 玩家决策 → [事件检定] → 结果分支 → 属性变化 → 影响后续事件
              ↑                                      ↓
         编辑系统事件                              势力声望变化
```

它是玩家体验"策略+随机性"的核心载体，决定了事件的"手感"。

### 1.2 核心设计目标

| 目标 | 描述 | 对应文档章节 |
|------|------|-------------|
| 策略决策感 | 玩家根据属性、难度评估风险后做选择 | 设计文档 §2-3 |
| 戏剧性呈现 | 二段式对抗增强叙事张力 | 设计文档 §4.2 |
| 信息透明 | 显示理论成功率，有据决策 | 设计文档 §5 |
| 配置驱动 | 策划通过 Excel 表配置，程序按表驱动 | 配置表模板 §2 |
| 可扩展性 | 支持分档、组合逻辑配置化 | 设计文档 §8 |

### 1.3 参考对标

本系统复刻自《苏丹的游戏》（Sultan's Game）的事件检定机制：
- 伯努利试验 + 二项分布的数学模型
- 难度→单骰成功率的映射关系
- "减骰子数"的对抗语义

**差异点**：ANGUS 在苏丹基础上新增了 split 分别检定、分档结果、双模式切换等扩展。

## 2. 数学模型详解

### 2.1 伯努利试验基础

每颗骰子是一次伯努利试验：成功概率 p，失败概率 1-p。

| 参数 | 符号 | 来源 |
|------|------|------|
| 骰子数 | n | 由属性值决定（single: n=attr; combined: n=attrA+attrB; split: 分别） |
| 单骰成功率 | p | 由难度映射 + Δp 修正 |
| 成功阈值 | k | 策划配置 |
| 有效骰子数 | n_eff | max(0, n - enemyAttr) |

### 2.2 二项分布

命中数 X ~ Binomial(n_eff, p)，即：

P(X = i) = C(n_eff, i) × p^i × (1-p)^(n_eff-i)

检定成功条件：X ≥ k

### 2.3 难度映射表

| 难度 | p_base | 含义 | 策划适用场景 |
|------|--------|------|-------------|
| easy | 0.60 | 60% 单骰成功率 | 新手引导、低风险事件 |
| normal | 0.50 | 50% 单骰成功率 | 标准事件、常规决策 |
| hard | 1/3 ≈ 0.3333 | 33.3% 单骰成功率 | 高风险高回报、Boss 级事件 |

**设计意图**：hard 难度下每颗骰子有 2/3 概率失败，当骰子数较少时会非常不稳定，制造紧张感。

### 2.4 Δp 修正机制

```
p_final = clamp(p_base + Δp, 0, 1)
```

| Δp 范围 | 效果 | 策划使用场景 |
|---------|------|-------------|
| +0.10 ~ +0.30 | 明显降低难度 | 天气有利、道具加成、临时 buff |
| +0.01 ~ +0.09 | 微调降低 | 轻微环境优势 |
| 0 | 无修正 | 默认 |
| -0.01 ~ -0.09 | 微调增加 | 轻微环境劣势 |
| -0.10 ~ -0.30 | 明显增加难度 | 天气恶劣、诅咒、debuff |

**建议**：Δp 严格控制在 [-0.3, +0.3]，超出此范围应改变难度档位而非大幅修正。

### 2.5 概率计算公式

**单/综合检定理论成功率**：
```
P(X ≥ k) = Σ C(n_eff, i) × p^i × (1-p)^(n_eff-i)，i 从 k 到 n_eff
```

**split 检定三档概率**（BANDS 组合逻辑）：
```
P_A = P(X_A ≥ kA)   // A 子检定成功率
P_B = P(X_B ≥ kB)   // B 子检定成功率
P_major  = P_A × P_B                          // 大成功：双通过
P_minor  = P_A(1-P_B) + (1-P_A)P_B           // 小成功：仅一项通过
P_fail   = (1-P_A)(1-P_B)                     // 失败：都不通过
```

## 3. 三种检定类型深度分析

### 3.1 单属性检定（single）

**数学模型**：
```
n = attrA
n_eff = max(0, n - enemyAttr)
X ~ Binomial(n_eff, p)
成功: X ≥ k
```

**适用场景分析**（结合事件脚本示例）：
| 场景 | 属性 | 阈值 | 对手 | 体验目标 |
|------|------|------|------|---------|
| 封面临时改题 | social | 4 | 固定 3 | 社交能力 vs 内部阻力 |
| 独自战斗 | combat | 5 | NPC attr | 战斗能力 vs 敌方力量 |
| 灵异调查 | imagination | 3 | 0 | 纯个人能力展示 |

**设计要点**：
- 最简单的检定形式，适合新手引导阶段
- 骰子数 = 属性值，玩家直觉理解"属性越高骰子越多"
- 对手对抗直接减骰子，叙事上理解为"对手在削弱你"

**数值平衡关注点**：
- 当 attr=5, enemy=3, k=3, p=0.5 时：P = P(Bin(2,0.5)≥3) = 0（必失败！）
- 这说明策划需要仔细校验 n_eff ≥ k 的边界情况

### 3.2 综合多属性检定（combined）

**数学模型**：
```
n = attrA + attrB
n_eff = max(0, n - enemyAttr)
X ~ Binomial(n_eff, p)
成功: X ≥ k
```

**合并方式**（combine_formula）：

| 方式 | 公式 | 适用场景 |
|------|------|---------|
| SUM | n = attrA + attrB | 简单直接，两个能力等权 |
| WEIGHT_SUM | n = w_A × attrA + w_B × attrB | 某个能力更重要 |

**适用场景分析**：
| 场景 | 属性 A | 属性 B | 体验目标 |
|------|--------|--------|---------|
| 拿下广告主 | wisdom | social | 数据+关系双管齐下 |
| 科学辟谣 | wisdom | rationality | 知识+理性论证 |
| 危机公关 | social | imagination | 口才+创意话术 |

**设计亮点**（结合事件脚本示例 2）：
- 支持 CheckBand 分档结果：同一次检定可以有 3 种结果（失败/小成功/大成功）
- 分档不影响检定逻辑本身，只在结果判定后映射到不同 tier
- `enemy_attr_source = npc_attr` 允许对手属性来自 NPC，而非固定值

**数值平衡关注点**：
- combined 的骰子数更多，阈值 k 也应该相应提高
- 建议策划按 k ≈ (attrA + attrB - enemyAttr) × p 的比例设定

### 3.3 分别多属性检定（split）

**数学模型**：
```
A: X_A ~ Binomial(nA_eff, p), 成功: X_A ≥ kA
B: X_B ~ Binomial(nB_eff, p), 成功: X_B ≥ kB
```

**对抗分摊**（PROPORTIONAL 策略）：
```
enemyA = floor(enemyAttr × attrA / (attrA + attrB))
enemyB = enemyAttr - enemyA
nA_eff = max(0, attrA - enemyA)
nB_eff = max(0, attrB - enemyB)
```

**组合逻辑**（split_combine_logic）：

| 逻辑 | 条件 | 结果 |
|------|------|------|
| ALL | A 通过 AND B 通过 | 成功；否则失败 |
| ANY | A 通过 OR B 通过 | 成功；否则失败 |
| BANDS | A⊕B / A∧B / ¬A∧¬B | 小成功/大成功/失败 |

**适用场景分析**：
| 场景 | 属性 A | 属性 B | 体验目标 |
|------|--------|--------|---------|
| 暗访偷拍+圆谎 | conceal | social | 两阶段行动 |
| 调查+报道 | wisdom | imagination | 素材收集+内容创作 |
| 逃跑+战斗 | conceal | combat | 潜行失败后转战斗 |

**设计亮点**（结合事件脚本示例 3）：
- **非二元结果**：小成功保住部分成果，不是简单的 succeed/fail
- **两段紧张感**：玩家要同时关注两组骰子
- **对抗分摊策略**：PROPORTIONAL 按属性占比分配，直觉合理
  - 也可扩展为 A_FIRST / B_FIRST / CUSTOM，让策划精细控制

**数值平衡关注点**：
- split 的两组检定独立，可能出现"A 很稳但 B 很悬"的情况
- 建议 kA / kB 的比例与 attrA / attrB 的比例大致一致
- 对抗分摊后，某一组 n_eff 可能为 0，需做边界处理

## 4. 对抗系统深度分析

### 4.1 对手属性来源

| 来源 | 枚举值 | 说明 | 示例 |
|------|--------|------|------|
| 无对抗 | none | enemyAttr = 0 | 纯个人能力检定 |
| 玩家输入 | input | 原型中用输入框 | 原型演示 |
| NPC 属性 | npc_attr | 运行时从 NPC 读取 | 市场总监的谈判力 |
| 固定值 | fixed | 使用 enemy_attr_fixed | 固定难度 3 |

**设计意图**：
- `npc_attr` 是正式游戏中最常用的模式——不同 NPC 有不同的"对抗力"
- `fixed` 适合特殊事件（如固定的环境难度）
- `input` 主要用于原型调试和 AI 辅助测试

### 4.2 二段式表现的技术实现

**核心约束**：随机作废必须保持概率等价性。

```
正确做法：
1. 生成 n_self 个 Bernoulli(p) 结果
2. 从中随机选 enemyAttr 个索引，标记为作废
3. 统计未作废结果的命中数

错误做法（会改变概率分布！）：
1. 生成 n_self 个结果
2. 优先作废"成功"的骰子 → ❌ 偏向失败
3. 将成功翻为失败 → ❌ 完全改变分布
```

**等价性证明**（直觉）：
- 在 i.i.d 伯努利序列中，随机抽取子集与重新生成等数量试验，分布相同
- 因此"投 n_self 颗再随机作废 enemyAttr 颗"等价于"直接投 (n_self - enemyAttr) 颗"

**动画时序**：
```
t=0ms        玩家掷骰（显示所有骰子结果）
t=100ms      第一颗作废骰子变灰（—）
t=200ms      第二颗作废骰子变灰
...
t=800ms      最后统计，显示最终结果
```
每颗作废间隔 80-110ms，制造"对手在逐一削弱你"的紧迫感。

### 4.3 split 对抗分摊详解

**PROPORTIONAL 策略**：
```
nTotal = attrA + attrB
enemyA = floor(enemyAttr × attrA / nTotal)
enemyB = enemyAttr - enemyA  // 保证总量一致
```

示例：attrA=8, attrB=4, enemyAttr=3
- enemyA = floor(3 × 8/12) = floor(2) = 2
- enemyB = 3 - 2 = 1
- nA_eff = 8 - 2 = 6
- nB_eff = 4 - 1 = 3

**边界情况**：如果 attrA 很小（如 attrA=1），floor 可能导致 enemyA=0，全部扣减集中在 B 上。这对于 B 组可能是毁灭性的——需要策划意识到这个风险。

**其他分摊策略对比**：

| 策略 | 公式 | 优点 | 缺点 |
|------|------|------|------|
| PROPORTIONAL | 按属性占比 | 公平直觉 | 弱属性可能被扣光 |
| A_FIRST | 先扣 A，剩余扣 B | 可控性高 | 不公平，偏袒 B |
| B_FIRST | 先扣 B，剩余扣 A | 可控性高 | 不公平，偏袒 A |
| CUSTOM | 策划指定 enemyA/enemyB | 完全可控 | 配置成本高 |

## 5. 双模式展示系统分析

### 5.1 模式对比

| 维度 | 纯数值模式（numeric） | 骰子模式（dice） |
|------|---------------------|------------------|
| 信息密度 | 高（直接显示概率和结果） | 低（需要观察动画过程） |
| 决策速度 | 快（看概率直接选） | 慢（需要等待动画） |
| 沉浸感 | 低（数字驱动） | 高（视觉+动画驱动） |
| 对手存在感 | 弱（只是后台数值） | 强（二段式动画） |
| 适用玩家 | 效率型、老玩家 | 体验型、新玩家 |

### 5.2 模式切换设计

- 默认模式由策划配置（`show_mode_default`）
- 玩家可随时一键切换
- 两种模式使用**完全相同的底层结算逻辑**，只是展示层不同
- 统一输出结构中包含 `roll` 数据，纯数值模式不展示但保留，便于调试

### 5.3 理论成功率展示

**必须展示**：总体成功率百分比（至少 1 位小数或整数）

**推荐展示**：
- 有效骰子数：n_eff = n_self - enemyAttr
- split 模式下分别展示 A/B 组的成功率
- split BANDS 模式下展示三档概率分布（大成功/小成功/失败）

**可选展示**：
- 风险标签（如"高风险"/"稳操胜券"/"五五开"）
- 与上次类似检定的对比

## 6. 配置表数据架构分析

### 6.1 三表结构

```
EventCheck（主表）──┬── CheckBand（分档表，可选）
                    │      ↓
                    │   TextKey（文案表，可选）
                    └── 通过 check_id 关联
```

### 6.2 EventCheck 字段完整清单

**通用字段**（11 个）：
event_id, option_id, check_id, check_type, difficulty_id, delta_p, show_mode_default, show_probability, has_opponent, enemy_attr_source, enemy_attr_fixed

**single 字段**（4 个）：
attr_a_id, attr_a_source, attr_a_fixed, k

**combined 额外字段**（5 个）：
attr_b_id, attr_b_source, attr_b_fixed, combine_formula, attr_a_weight, attr_b_weight

**split 额外字段**（5 个）：
attr_b_id, attr_b_source, attr_b_fixed, k_a, k_b, split_combine_logic, opponent_split_policy

**校验规则**：
- check_id 全表唯一
- event_id + option_id 组合唯一
- check_type 决定哪些字段必填
- 数值约束：attr ≥ 0, k ≥ 0, delta_p ∈ [-0.3, +0.3]

### 6.3 配置表扩展性

| 扩展方向 | 实现方式 | 影响 |
|---------|---------|------|
| 新增检定属性 | 在 attr_*_id 枚举中添加 | 属性系统需同步 |
| 新增难度档位 | 在 difficulty_id 枚举中添加，映射新 p_base | 需策划定义 p 值 |
| 新增合并方式 | 在 combine_formula 枚举中添加 | 需实现新的 n 计算公式 |
| 新增组合逻辑 | 在 split_combine_logic 枚举中添加 | 需实现新的结果映射 |
| 三属性检定 | 扩展 combined 或新增 triple 类型 | 工作量较大 |

## 7. UI/UX 交互规范分析

### 7.1 必须（Must Have）

|------|------|--------|
| 检定类型标识 | 当前是 single/combined/split | P0 |
| 属性值显示 | attrA、attrB 的当前值 | P0 |

### 7.2 推荐（Should Have）

| 元素 | 说明 | 优先级 |
|------|------|--------|

| 分档结果详情 | CheckBand 的命中数范围 | P1 |

### 7.3 骰子模式 UX 细节

**投骰阶段**：
- 骰子排列为一行（single/combined）或两行（split A/B）
- 每颗骰子显示 ✓（命中）或 ×（未命中）
- 可加轻微动画（骰子翻转/弹出）

**对抗阶段**：
- 被抵消的骰子逐个变为灰色 —（划线样式）
- 每颗间隔 80-110ms
- 可加音效（对抗/抵消声）

**结算阶段**：
- 最终命中数高亮显示
- 与阈值 k 对比（≥ k 则绿色成功提示）
- split 模式两组分别展示

## 8. 程序接口与数据结构分析

### 8.1 统一输出结构

```json
{
  "mode": "numeric|dice",
  "checkType": "single|combined|split",
  "p": 0.5,
  "probabilityShown": 0.623,
  "opponent": {
    "enemyAttr": 3,
    "enemyA": 1,
    "enemyB": 2
  },
  "roll": {
    "playerDice": {
      "A": [true, false, true],
      "B": [false, true],
      "combined": [true, false, true, false, true]
    },
    "negatedIndices": {
      "combined": [0, 3, 4],
      "A": [1],
      "B": [0, 2]
    }
  },
  "result": {
    "pass": true,
    "tier": "fail|minor_success|major_success",
    "successCount": 4,
    "successCountA": 2,
    "successCountB": 2,
    "k": 3,
    "kA": 2,
    "kB": 2
  }
}
```

**设计优点**：
- 统一结构：三种检定类型返回相同格式，UI 处理逻辑统一
- 完整数据：`roll` 字段包含每颗骰子的结果和被抵消的索引，支持动画回放
- `probabilityShown`：UI 可直接显示，无需重复计算

### 8.2 核心函数拆分建议

```
EventCheckEngine
├── calculate_probability(config) → float           // 计算理论成功率
├── execute_check(config) → CheckResult            // 执行检定（含随机）
├── resolve_opponent(config) → OpponentResult       // 解算对抗扣减
├── resolve_split_opponent(config) → SplitOpponent  // split 对抗分摊
├── map_result_to_tier(result, bands) → string      // 映射分档结果
└── generate_dice_display(result) → DiceDisplay    // 生成骰子展示数据
```

## 9. 边界条件与异常处理

### 9.1 数值边界

| 条件 | 行为 | 处理 |
|------|------|------|
| n_eff = 0 | 必失败（X=0，除非 k=0） | UI 提示"对手完全压制" |
| k > n_eff | 必失败 | 提前判定，不掷骰 |
| k = 0 | 必成功 | 提前判定，不掷骰 |
| p = 0 | 全部未命中 | Δp 极端情况 |
| p = 1 | 全部命中 | Δp 极端情况 |
| attrA = 0 且 single | n=0，必失败 | 提前判定 |
| split 中某组 n_eff = 0 | 该组必失败 | 其他组仍正常检定 |

### 9.2 配置校验

| 校验项 | 规则 | 错误提示 |
|--------|------|---------|
| check_id 唯一性 | 全表不重复 | "检定 ID 重复" |
| 必填字段非空 | 根据 check_type 校验 | "缺少必填字段" |
| 枚举值合法 | 在预定义范围内 | "无效的检定类型" |
| k ≤ n 的最大值 | k 不能超过属性原始值 | "阈值超过骰子数" |
| CheckBand 无重叠 | 区间不重叠 | "分档区间重叠" |
| enemyA + enemyB = enemyAttr | 总量一致 | 已由公式保证 |

### 9.3 运行时异常

- **NPC 属性缺失**：`enemy_attr_source = npc_attr` 但 NPC 无对应属性 → 使用默认值 0 + 日志警告
- **资源加载失败**：骰子纹理/音效缺失 → 降级为纯数值模式
- **浮点精度**：二项分布累加时注意浮点误差 → 使用 Decimal 或整数运算

## 10. 与现有系统的关联分析

### 10.1 与属性系统（attributes.md）的关联

**直接关联**：事件检定使用的属性值来自属性系统中定义的**个人属性**。

| 属性系统定义 | 事件检定用途 | 对应检定场景 |
|-------------|-------------|-------------|
| 探索 | 可作为检定属性 | 探索事件效率检定 |
| 生存 | 可作为检定属性 | 危险环境存活检定 |
| 洞察 | 可作为检定属性 | 发现隐藏线索检定 |
| 想象 | 可作为检定属性 | 内容创作质量检定 |
| 理性 | 可作为检定属性 | 科学类任务检定 |
| 笔力 | 可作为检定属性 | 报道质量检定 |
| 胆识 | 可作为检定属性 | 高风险任务检定 |
| 诡思（灵视） | 可作为检定属性 | 超自然事件交互 |
| 人脉（话术） | 可作为检定属性（等价 social） | 社交类任务 |
| 造势 | 可作为检定属性 | 营销传播检定 |

**需要同步的事项**：
- 属性系统中需明确每个属性的 ID（如 wisdom/social/conceal 等），与配置表的 `attr_a_id` 枚举一致
- 属性值范围影响检定体验——属性值 1-30 对应 1-30 颗骰子
- 属性成长系统会影响检定难度曲线设计

**需要添加到 attributes.md 的内容**：在个人属性表中增加"检定属性 ID"列，明确每个属性在检定系统中的标识符。

### 10.2 与势力系统（factions.md）的关联

**间接关联**：事件检定中的对手属性可能来自势力 NPC。

| 势力 | 对抗检定场景 | 对手属性来源 |
|------|-------------|-------------|
| 政府 | 监管约谈事件 | 政府官员影响力（npc_attr） |
| 科学会 | 学术辩论事件 | 科学家逻辑力（npc_attr） |
| 密教会 | 灵异调查事件 | 仪式难度（fixed） |
| 财团 | 商务谈判事件 | 市场总监谈判力（npc_attr） |

**势力任务与检定的关系**：
- 势力发布的强制/禁止任务本身不直接涉及检定
- 但完成任务的**过程事件**可能触发检定（如说服 NPC、突破封锁等）
- 检定结果（大成功/小成功/失败）可能影响势力声望变化幅度

**需要添加到 factions.md 的内容**：在势力任务描述中增加"可能触发的事件检定类型"字段，明确哪些任务涉及检定。

### 10.3 与编辑系统（editing.md）的关联

**业务关联**：事件检定结果直接影响周刊编辑的输入。

| 编辑系统环节 | 事件检定影响 |
|-------------|-------------|
| 版面内容 | 探索事件的检定结果决定可获得哪些卡牌（线索卡/材料卡） |
| 卡牌质量 | 检定结果（大成功）可能获得更高质量的素材 |
| 体系连携 | 事件检定结果可能偏向某种体系，影响连携策略 |
| 周刊属性 | 检定成功/失败可能影响公信、诡名等周刊属性 |
| 开天窗 | 检定失败导致素材不足，可能被迫开天窗 |

**设计建议**：
- 探索事件 → 检定 → 根据结果 tier 发放不同质量的卡牌
- 大成功：获得稀有卡牌 + 额外素材
- 小成功：获得普通卡牌
- 失败：可能获得低质量卡牌或无卡牌

**需要添加到 editing.md 的内容**：在探索产出表中增加"检定结果对产出质量的影响"字段。

### 10.4 与探索系统（exploration.md）的关联

**直接关联**：探索系统是事件检定的主要触发源。

| 探索系统环节 | 事件检定触发点 |
|-------------|---------------|
| 探索点事件 | 进入探索点后的事件选项触发检定 |
| 角色派遣 | 派遣角色卡执行任务时的能力检定 |
| 道具使用 | 某些道具可能影响检定参数（如 Δp 修正） |
| 探索点解锁 | 特定属性阈值可能需要检定而非简单比较 |

**探索产出与检定的关系**：
- 当前探索产出表定义了四种产出类型（线索卡/材料卡/道具卡/探索节点）
- 事件检定系统提供了更细粒度的产出控制：不同 tier 对应不同产出
- 建议：将 CheckBand 表中的 `reward_id` 与探索产出系统对接

**需要添加到 exploration.md 的内容**：
- 探索点事件配置中增加检定相关字段
- 探索产出规则中增加"检定结果对产出类型和数量的影响"

## 11. 开发风险评估

### 11.1 技术风险

| 风险 | 级别 | 描述 | 缓解措施 |
|------|------|------|---------|
| 概率一致性 | 🔴 高 | 对抗表现破坏概率等价性 | 严格使用"随机作废"算法，单元测试覆盖 |
| 浮点精度 | 🟡 中 | 二项分布累加的浮点误差 | 使用高精度运算或查表 |
| split 复杂度 | 🟡 中 | 两组检定 + 分摊 + 三档结果 | 充分的测试用例 |
| 动画时序 | 🟢 低 | 骰子动画与结算逻辑的同步 | 使用状态机管理动画流程 |
| 配置表解析 | 🟢 低 | Excel 格式读取和校验 | 使用成熟的表格解析库 |

### 11.2 设计风险

| 风险 | 级别 | 描述 | 缓解措施 |
|------|------|------|---------|
| 数值平衡 | 🟡 中 | 不同检定类型的难度体验不一致 | 大量数值模拟验证 |
| 体验单调 | 🟢 低 | 长期重复相同检定模式 | 通过事件多样性缓解 |
| 信息过载 | 🟢 低 | split 模式下信息展示过于复杂 | 渐进式展示 + 默认简化模式 |

### 12.2 技术选型建议

| 组件 | 建议 | 理由 |
|------|------|------|
| 概率计算 | GDScript 内置 | 精度足够，避免引入额外依赖 |
| 配置表格式 | JSON 或 CSV | 比 Excel 更易版本管理和程序解析 |
| 动画系统 | Godot AnimationPlayer + Tween | 原生支持，性能好 |
| 状态管理 | 信号驱动 | Godot 推荐模式，检定完成发信号通知 UI |

### 12.3 验证方法

1. **概率验证**：运行 10000 次模拟，统计实际成功率与理论值误差 < 1%
2. **等价性验证**：比较"直接投 n_eff 颗"与"投 n 颗再作废"的成功率，偏差 < 0.1%
3. **边界验证**：n_eff=0、k=0、p=0、p=1 等极端情况正确处理
4. **配置校验验证**：伪造非法配置表，确认所有校验规则生效
