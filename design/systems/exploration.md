# 世界探索系统设计

## 定位

探索系统是周循环的前半段，负责把“全球选点 -> 派遣职员 -> 事件检定 -> 线索产出”串起来，并为后续编辑部组版提供可消费的线索输入。

当前长期规则以：

- `design/systems/full-chain-loop.md`
- `design/systems/event-check-design.md`
- `design/systems/event-check-analysis.md`

为准。早期仅用于网页实验的 d6 说明不再作为主设计基线。

## 周期结构

每周固定 7 天。

1. 回合起始：刷新随机简报、势力提示、临时探索节点。
2. 全球地图：查看可进入区域，按标签筛选潜在线索方向。
3. 区域探索：选择节点、配置 1-3 名职员、执行检定。
4. 线索入库：把本周探索结果整理成线索条目。
5. 日数耗尽或主动结束：进入编辑部阶段。
6. 发刊结算结束后：进入下一周，保留长期状态，清空周内临时状态。

## 长期状态

探索阶段至少维护以下跨周状态：

- 周序号 `week`
- 剩余天数 `day`
- 周刊五维：公信、诡名、声望、守序、狂性
- 本周线索池
- 已解锁区域 / 节点
- 与势力、剧情和特殊事件相关的阶段性标记

约束：

- 公信与诡名形成叙事倾向对冲。
- 守序与狂性形成风险对冲。
- 狂性原则上不自动回落，只允许被特定系统临时压制。

## 地图层级

### 全球层

全球层负责展示区域入口与筛选信息。

- 初始仅开放少量基础区域。
- 其余区域由声望、属性、剧情条件或关键线索解锁。
- 标签筛选只影响高亮和推荐，不直接修改区域存在性。

建议的区域字段：

| 字段 | 说明 |
|------|------|
| `id` | 区域唯一标识 |
| `name` | 区域显示名 |
| `unlocked` | 是否已解锁 |
| `hint` | 解锁条件提示 |
| `pulse` | 是否高亮提示玩家关注 |
| `nodes` | 区域内节点列表 |

### 区域层

区域层展示可执行的探索节点。

节点至少区分三类：

- `permanent`：常驻节点
- `temp`：临时节点，可附带时限
- `hidden`：隐藏节点，由条件驱动显示

建议的节点字段：

| 字段 | 说明 |
|------|------|
| `id` | 节点唯一标识 |
| `name` | 节点名 |
| `days` | 消耗天数 |
| `need` | 属性需求 |
| `tags` | 科学纪实 / 神秘玄学 / 世俗流量倾向 |
| `difficulty` | 检定难度 |
| `enemyAttr` | 对抗值 |
| `kind` | permanent / temp / hidden |
| `deadlineDay` | 临时节点截止日，可选 |
| `unlock` | 隐藏节点解锁条件，可选 |
| `chain` | 链式前置状态，可选 |

## 派遣与检定

### 派遣规则

- 每次探索派遣 1-3 名职员。
- 节点需求由所选职员对应属性累加满足。
- 日数不足时禁止执行节点。

### 与事件检定系统的关系

探索阶段默认使用当前已经成型的二项分布检定框架，而不是单独再维护一套简化规则。

推荐的探索默认形态：

- 调查池：探索 + 洞察 + 诡思
- 现场池：生存 + 理性
- 结果档位：大成功 / 小成功 / 失败 / 大失败

映射建议：

| 结果档位 | 探索产出 |
|---------|---------|
| 大成功 | 高质量线索，可能附带额外属性收益或隐藏节点推进 |
| 小成功 | 标准线索，允许进入编辑部 |
| 失败 | 低质量碎片线索或有限产出 |
| 大失败 | 伴随守序下降、狂性上升或现场风险后果 |

如需更强的现场戏剧性，可在基础检定外追加：

- 特殊骰标签
- 对手干预
- 玩家反制
- 节点特有的风险状态

这些扩展不应改写基础概率语义，只应挂在检定前后。

## 线索产出

探索的标准输出不是直接生成“报道”，而是先生成线索。

建议线索字段：

| 字段 | 说明 |
|------|------|
| `title` | 线索标题 |
| `type` | `sci` / `occult` / `pop` |
| `tier` | 1-3 档，决定后续转稿质量 |
| `source_region` | 来源区域，可选 |
| `source_node` | 来源节点，可选 |
| `risk_flags` | 特殊风险或污染标记，可选 |

线索在进入编辑部时再映射为故事卡：

- `title` -> 报道标题
- `type` -> 报道题材标签
- `tier` -> 质量档与 `baseValue`

## 与编辑系统的接口

探索系统要明确对编辑部输出什么，而不是只定义“探索很有趣”。

本项目的稳定接口是：

1. 本周探索结束时，输出线索列表。
2. 编辑部根据线索生成本周可用故事库。
3. 故事库不足部分再由系统补齐随机稿件。
4. 结算后仅保留长期状态，不保留周内临时线索池。

这部分总览见 `design/systems/full-chain-loop.md`。

## Godot 落地建议

- 探索层应做成状态驱动，而不是散落在多个临时场景中。
- 区域、节点、职员、事件检定配置应逐步表驱动。
- 探索结果必须可直接喂给编辑部，而不是额外手工转换。
- 当前 Godot 实现优先级应放在“能跑完整一周”而不是先扩大量级。
